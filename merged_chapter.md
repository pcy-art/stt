# Part 1. ASR의 이해와 AI 환경 구축 (기초/인프라 편)

## 제1장. 프로젝트 개요 및 기반 지식

본 장에서는 본 업무 매뉴얼의 작성 배경과 지향점을 명확히 정의하고, 전체 프로젝트의 아키텍처와 데이터 흐름을 조망합니다. 특히 **내부망(폐쇄망)**이라는 특수한 환경적 제약 속에서 성공적으로 모델을 개발하고 운영하기 위해 반드시 갖추어야 할 실무 지식을 다룹니다.

---

### 1.1 문서 목적

#### 1.1.1 프로젝트 최종 목표
본 문서의 궁극적인 목적은 자동 음성 인식(ASR, Automatic Speech Recognition) 시스템의 핵심 원리를 부서원이 깊이 있게 이해하고, 이를 실무에 즉각 적용하도록 돕는 데 있습니다. 

특히 잡음이 심하거나 신호 감쇄로 인해 품질이 저하된 **특수 환경의 음성 데이터**에서도 높은 전사(Transcription) 정확도를 확보하는 것이 핵심입니다. 이를 위해 Whisper와 같은 대형 모델에 **LoRA(Low-Rank Adaptation)** 기반 파인튜닝 기법을 적용하여 모델 성능을 극한으로 끌어올리는 체계적인 과정을 습득합니다. 최종적으로는 데이터 품질 관리부터 모델 최적화, 내부망 배포에 이르는 전 과정을 외부 지원 없이 부서 자체적으로 수행할 수 있는 **'독립적 실무 역량 내재화'**를 목표로 합니다.

#### 1.1.2 문서의 대상 독자
* **AI 모델 엔지니어:** Whisper 모델의 아키텍처를 이해하고, LoRA 파인튜닝 코드 작성 및 GPU 자원 최적화, 추론 성능 개선을 담당하는 인력
* **데이터 관리자 (Data Steward / QA):** 원시 데이터 수집 및 라벨링 가이드라인 수립, 학습 데이터셋 구축 및 데이터 정합성 검증 파이프라인을 관리하는 담당자
* **인프라 및 운영 담당자:** 내부망 환경 내 컨테이너(Docker) 기반 서빙 환경을 구축하고, 모델 배포 및 리소스 관리(OOM 대응 등)를 담당하는 인력

#### 1.1.3 문서의 핵심 교육 목표
본 문서를 통해 독자가 달성해야 할 구체적인 목표는 다음과 같습니다.
1. **오프라인 인프라 자립화:** 인터넷이 차단된 환경에서도 패키지(Wheel) 및 Docker 이미지를 활용해 독립적인 개발/운영 환경을 구축할 수 있다.
2. **도메인 특화 데이터 정제:** 특수 환경 음성에 맞춘 전사 가이드라인을 이해하고, MFA(Montreal Forced Aligner)를 활용해 고품질의 학습 데이터를 생성할 수 있다.
3. **고효율 모델 학습 기법 습득:** LoRA 기법을 활용해 한정된 자원으로도 도메인에 최적화된 파인튜닝 파이프라인을 설계하고 운영할 수 있다.
4. **안정적인 온프레미스 서빙:** FastAPI 등을 활용해 구축된 모델을 API 형태로 안정적으로 배포하고, 실제 서비스 환경에 적용할 수 있다.

---

### 1.2 전체 시스템 아키텍처 및 데이터 흐름

전체 ASR 시스템은 크게 [데이터 준비] → [모델 학습] → [서비스 배포]의 3단계 파이프라인으로 구성됩니다. 각 단계는 서로 유기적으로 연결되어 있으며 데이터의 흐름은 일방향과 피드백 루프를 동시에 가집니다.

#### 1.2.1 음성 데이터 수집 및 파싱 파이프라인
실제 운영 시스템(VHF 통신망, CCTV, 녹취 서버 등)에서 발생한 원시 음성(Raw Audio) 데이터를 주기적으로 수집합니다. 
수집된 데이터는 포맷이 다양하므로 (예: 8kHz 통신 음성, 48kHz 고음질 음성 등) FFmpeg 등의 파싱 모듈을 거쳐 **16kHz, 16-bit, Mono 채널의 통일된 WAV 포맷**으로 변환 및 적재됩니다.

#### 1.2.2 데이터 정제 및 정렬 아키텍처
적재된 음성 데이터는 잡음 제거(Denoising)를 거친 후, VAD(Voice Activity Detection)를 통해 의미 있는 발화 구간만 추출됩니다. 이후 기본 모델을 활용해 자동 생성된 초벌 전사 결과를 인간 작업자(Human-in-the-loop)가 검토 및 교정한 후, 해당 텍스트와 오디오 구간을 강제 정렬(Forced Alignment)하여 밀리초(ms) 단위의 타임스탬프가 매핑된 고품질 JSONL 형태의 학습 데이터셋으로 생성합니다.

#### 1.2.3 모델 학습 및 추론 서버 구성도
*   **학습부 (Training Node):** GPU 자원(NVIDIA H100)을 활용하여 HuggingFace Accelerate 기반 학습을 수행합니다. 베이스 모델(Whisper)은 Freeze하고 LoRA 어댑터 가중치만 업데이트하여 체크포인트를 생성합니다.
*   **추론부 (Inference Node):** 학습이 완료된 LoRA 가중치는 베이스 모델에 병합(Merge)된 후, Faster Whisper 또는 NVIDIA Triton Inference Server를 통해 API 형태로 서빙됩니다. 사용자의 음성 파일 요청이 인입되면 즉시 전사된 텍스트와 신뢰도(Confidence Score)를 반환합니다.

### 1.3 내부망(폐쇄망) 환경 제약사항 요약

본 프로젝트는 외부 인터넷 망과 완벽히 차단된 내부망(On-Premise 폐쇄망)에서 구축되어야 하므로 일반적인 클라우드 기반 AI 개발과는 다른 철저한 사전 준비가 필요합니다.

#### 1.3.1 네트워크 차단 환경 및 보안 규정
*   **인터넷 차단:** `pip install`, `apt-get`, `docker pull` 등 외부 리소스를 다운로드하는 모든 명령어가 동작하지 않습니다. 
*   **오프라인 환경 동기화 및 Docker 반입 전략:** 네트워크가 완벽히 차단되어 있으므로, 내부망 서버와 동일한 OS 및 GPU 드라이버 환경을 인터넷이 연결된 외부망(로컬 PC 또는 클라우드)에 먼저 구축해야 합니다. 이 외부 환경에서 필요한 모든 패키지(PyTorch, CUDA, Python 라이브러리 등)와 의존성을 설치하여 완벽히 구동됨을 테스트한 후, 해당 환경을 통째로 **Docker 이미지(.tar 파일)**로 추출(Export)하여 내부망으로 반입합니다. 이러한 방식을 사용해야만 내부망 진입 후 빈번하게 발생하는 치명적인 패키지 의존성 충돌(Dependency Hell) 문제를 원천 차단하고 시스템을 원활하게 구동할 수 있습니다.
*   **보안 규정:** 외부에서 반입되는 모든 파일(Docker 이미지, 모델 가중치, 소스코드)은 단독망 PC를 통한 악성코드 검사와 보안 부서의 승인을 거쳐야 합니다.

#### 1.3.2 서버 하드웨어 사양 (CPU, GPU, RAM)
*   내부망에 할당된 하드웨어 자원은 고정되어 있으며 클라우드처럼 동적 확장이 불가능합니다.
*   특히 Whisper Large 모델의 파인튜닝 시 VRAM(비디오 메모리) 부족 현상(OOM)이 빈번하게 발생할 수 있으므로, 할당된 GPU(80GB VRAM을 가진 H100)의 스펙을 정확히 인지하고 그에 맞는 Batch Size와 Mixed Precision(fp16) 전략을 설정해야 합니다.

#### 1.3.3 소프트웨어 라이선스 및 오픈소스 사용 정책
폐쇄망에 반입할 외부 오픈소스(Python 라이브러리, Whisper 모델, FastAPI 등)는 기업 내 오픈소스 라이선스 정책(MIT, Apache 2.0 등)을 위반하지 않는지 사전에 검토되어야 합니다. GPL 라이선스가 포함된 패키지의 경우 상용 서비스 탑재 시 소스코드 공개 의무가 발생할 수 있으므로 사용을 지양합니다.

### 1.4 실시간(Streaming) vs 비실시간(Batch) ASR 처리 방식

ASR 시스템은 서비스의 성격에 따라 입력 음성을 처리하는 방식이 크게 두 가지로 나뉩니다. 본 프로젝트의 아키텍처 선택 기준을 확립하기 위해 두 방식을 비교합니다.

#### 1.4.1 실시간(Streaming) 처리의 특징과 지연 시간 요구사항
*   **특징:** 사용자가 말을 하는 동시에 1~2초 단위의 음성 청크(Chunk)를 서버로 전송하여 실시간으로 텍스트를 스트리밍하여 화면에 뿌려주는 방식입니다. (예: 유튜브 라이브 자막, 실시간 통역 앱)
*   **한계:** 전체 문맥을 파악하기 전에 즉각적으로 단어를 예측해야 하므로 **문맥 인지 능력이 떨어지고 상대적으로 정확도가 낮습니다.** 또한, 짧은 지연 시간(Low Latency)을 보장하기 위해 서버 자원이 항시 대기 상태여야 하므로 유지 비용이 큽니다.

#### 1.4.2 비실시간(Batch) 처리의 장점 (정확도 및 처리량)
*   **특징:** 통화 녹음 파일, 회의 녹음 파일 등 이미 완결된 긴 오디오 파일을 한 번에(또는 큰 단위로 분할하여) 서버에 전송하고 처리 결과를 기다리는 방식입니다.
*   **장점:** 모델이 문장의 시작과 끝의 **전체 문맥(Context)을 모두 파악한 상태에서 전사하므로 정확도(WER)가 월등히 높습니다.** 또한 서버의 GPU 자원을 최대한 효율적으로 활용하여 여러 파일을 동시에 병렬 처리(Batching)할 수 있어 대규모 데이터 처리에 적합합니다.

#### 1.4.3 본 프로젝트의 방식 선정 이유
본 프로젝트의 핵심 과제는 **"고잡음/심층 감쇄 환경에서 품질이 저하된 오디오의 정확한 텍스트 변환"**입니다. 
잡음 환경에서는 단어 하나하나의 음향적 특징이 모호해지기 때문에 앞뒤 문맥(Context)을 통한 언어 모델의 추론 능력이 결정적인 역할을 합니다. 따라서 지연 시간보다는 **극한의 정확도가 요구되며, Whisper 모델의 구조(Bidirectional Attention) 또한 전체 문장을 한 번에 처리할 때 가장 우수한 성능을 발휘합니다.** 
결론적으로 본 시스템은 대규모 파일 기반의 **비실시간(Batch) 오프라인 추론 방식을 핵심 아키텍처로 채택**하여 구축을 진행합니다.

# 제2장. ASR (Automatic Speech Recognition) 기초 및 핵심 아키텍처

본 장에서는 음성 신호가 디지털 텍스트로 변환되는 물리적·수학적 과정을 이해하고, 과거부터 현재까지 ASR 기술이 어떻게 진화해 왔는지 분석합니다. 특히 최신 **End-to-End(E2E)** 모델의 구조를 파악함으로써, 우리가 사용할 Whisper 모델의 동작 원리를 깊이 있게 이해합니다.

---

### 2.1 ASR의 동작 원리와 파이프라인

#### 2.1.1 음성 신호의 디지털화 (Feature Extraction)
컴퓨터는 연속적인 아날로그 음성 파형(Waveform)을 그대로 이해할 수 없습니다. 따라서 전처리 과정을 통해 특징 벡터를 추출해야 합니다.
*   **Sampling & Quantization:** 아날로그 파형을 초당 수만 번(예: 16kHz) 나누어 디지털 수치로 변환합니다.
*   **STFT (Short-Time Fourier Transform):** 시간축의 데이터를 주파수축의 데이터로 변환하여 '어떤 주파수 성분이 강한가'를 파악합니다.
*   **Mel-Spectrogram:** 인간의 청각 구조(저주파에 민감하고 고주파에 둔감함)를 반영하여 주파수 간격을 재조정합니다. **Whisper를 포함한 대부분의 최신 모델은 이 Mel-Spectrogram을 입력값으로 사용합니다.**

#### 2.1.2 전통적인 ASR의 3요소 (Modular ASR)
E2E 모델 이전에는 다음과 같은 세 가지 모듈을 독립적으로 학습시켜 결합했습니다.
1.  **음향 모델 (Acoustic Model):** "이 소리 데이터는 'ㄱ' 혹은 'a'라는 음소일 확률이 높다"를 판별합니다.
2.  **발음 사전 (Lexicon/Pronunciation Model):** " 'ㅎ-ㅏ-ㄴ-ㄱ-ㅡ-ㄹ'이라는 음소 조합은 '한글'이라는 단어다"라는 매핑 테이블입니다.
3.  **언어 모델 (Language Model):** " '철수가' 다음에 올 단어는 '밥을'일 확률이 높다"와 같이 문장의 자연스러움을 결정합니다.

---

### 2.2 ASR 아키텍처의 세대별 발전

#### 2.2.1 1세대: 통계적 모델 (HMM-GMM)
*   **작동 방식:** 음성 신호를 상태(State) 간의 전이 확률로 계산하는 **HMM(Hidden Markov Model)**과 각 상태의 확률 분포를 추정하는 **GMM(Gaussian Mixture Model)**을 결합했습니다.
*   **한계:** 데이터가 많아져도 성능 향상에 한계가 명확하며, 음소 단위로 아주 정교하게 라벨링된 데이터가 필요했습니다. 특히 잡음 환경에서 확률 계산이 급격히 무너지는 취약성이 있습니다.

#### 2.2.2 2세대: 신경망 기반 연결주의 모델 (RNN/CTC)
*   **작동 방식:** 시계열 데이터 처리에 특화된 **RNN(Recurrent Neural Network)**이나 **LSTM**을 사용합니다. **CTC(Connectionist Temporal Classification)** 손실 함수를 도입하여, 입력 음성 길이와 출력 텍스트 길이가 달라도 학습이 가능해졌습니다.
*   **장점:** 이전 시점의 정보를 기억하므로 문맥 파악 능력이 향상되었습니다.
*   **한계:** 입력 데이터가 길어질수록 연산 속도가 저하되며, 병렬 처리가 불가능하여 대규모 학습에 막대한 시간이 소요됩니다.

#### 2.2.3 3세대: 트랜스포머 기반 End-to-End (E2E) 모델
현재 가장 진보된 방식으로, 입력(Audio)부터 출력(Text)까지를 하나의 거대한 신경망(Transformer)으로 통합했습니다.
*   **Self-Attention 메커니즘:** 문장의 모든 부분과 음성의 모든 부분을 동시에 비교하여, 현재 시점에서 어떤 음성 특징에 집중해야 할지(Attention) 스스로 결정합니다.
*   **Whisper의 특징:** OpenAI에서 개발한 Whisper는 68만 시간이라는 압도적인 양의 다국어/다양한 환경 데이터를 학습했습니다. 덕분에 잡음이 섞인 환경에서도 문맥을 통해 단어를 '추론'하는 능력이 매우 뛰어납니다.

---

### 2.3 트랜스포머 기반 ASR 구조의 특징

#### 2.3.1 인코더-디코더 (Encoder-Decoder) 아키텍처
트랜스포머의 인코더-디코더 구조는 음성이라는 '소리 신호'를 텍스트라는 '언어 정보'로 번역하는 핵심 엔진입니다.

*   **인코더 (Encoder) - "소리의 특징 추출기":**
    입력된 Mel-Spectrogram을 받아 각 시간대별 소리의 특징을 분석합니다. 단순히 소리의 크기나 주파수만 보는 것이 아니라, 앞뒤 맥락을 살펴 이 소리가 어떤 음운적 특성을 가졌는지 **고차원적인 특징 벡터(Hidden Representation)**로 압축합니다. Whisper의 경우, 매우 깊은 층의 인코더를 사용하여 노이즈가 섞인 음성에서도 유의미한 특징을 매우 잘 잡아냅니다.
*   **디코더 (Decoder) - "문장 생성기":**
    인코더가 넘겨준 특징 정보와, **'직전 시점까지 자신이 생성한 단어'**들을 결합하여 다음 단어를 예측합니다. 이를 **자기 회귀(Auto-regressive)** 방식이라고 합니다. 
    *   **Cross-Attention:** 디코더는 단어를 생성할 때마다 인코더의 전체 음성 정보 중 어떤 부분(시간대)을 참고해야 할지 매번 다시 계산합니다. 예를 들어 "학교에"라는 단어를 뱉을 때, 인코더가 분석한 소리 데이터 중 '학교'에 해당하는 시간대에 집중하는 식입니다.

#### 2.3.2 Self-Attention 메커니즘
기존 RNN(LSTM) 방식은 음성 데이터를 순차적으로 읽어야 했기에 앞부분의 정보를 뒷부분에서 잊어버리는 문제가 있었으나, 트랜스포머는 Self-Attention을 통해 이를 해결했습니다.

*   **글로벌 맥락 파악:** Self-Attention은 입력된 음성 프레임 전체를 동시에 비교합니다. 따라서 1분 길이의 긴 문장이라도 문장 초반의 정보와 후반의 정보를 즉각적으로 연결하여 문맥을 파악합니다.
*   **병렬 연산의 이점:** 모든 데이터를 한꺼번에 처리할 수 있어, **H100과 같은 고성능 GPU 자원**을 최대로 활용할 수 있습니다. 이는 학습 시간을 획기적으로 단축시키는 요인이 됩니다.
*   **음성에서의 역할:** 소리가 울리거나(Reverb) 잡음이 섞여 특정 구간의 발음이 뭉개졌을 때, Attention은 주변의 깨끗한 음성 구간이나 문맥 정보를 참조하여 뭉개진 구간의 의미를 성공적으로 복원해냅니다.

#### 2.3.3 CTC (Connectionist Temporal Classification) 기법의 이해
음성 인식 모델을 설계할 때 가장 큰 고민은 **"10초짜리 음성(수천 프레임)과 그 결과물인 10글자(토큰)를 어떻게 정렬(Alignment)할 것인가"**입니다.

*   **정렬 문제의 해결:** CTC는 모델이 매 프레임마다 '글자' 혹은 '공백(Blank, $\epsilon$)'을 예측하게 합니다. 
    *   **Collapse Rule:** 만약 모델이 "ㅎ-ㅎ-ㅏ-ㅏ-ㅏ-ㄴ-ㄴ" 처럼 중복해서 예측했다면, 이를 하나로 합치고($a-a \rightarrow a$) 공백 토큰을 제거하여 최종적으로 "한"이라는 단어를 도출합니다.
*   **Whisper와의 차이점:** 
    *   **CTC 방식 (Encoder-only):** 주로 실시간성이 중요한 가벼운 모델(Wav2vec 2.0 등)에서 사용하며, 디코더 없이 인코더만으로 빠르게 결과를 뽑아낼 때 유리합니다.
    *   **Attention 방식 (Whisper 등):** 인코더와 디코더가 상호작용하며 훨씬 정교하게 문맥을 파악합니다. CTC보다 연산량은 많지만, **심한 잡음 환경이나 복잡한 도메인**에서는 문맥 추론 능력이 뛰어난 Attention 방식이 압도적인 성능을 보입니다.
*   **학습 포인트:** 우리 프로젝트는 Whisper를 사용하므로 CTC를 직접 구현하진 않지만, 데이터 관리자가 **오디오와 텍스트의 강제 정렬(MFA)**을 수행할 때 이러한 '시간축 정렬'의 개념을 이해하고 있어야 데이터의 품질을 정확히 검수할 수 있습니다.

---

### 2.4 잡음/감쇄 환경에서의 ASR 한계점과 도메인 튜닝의 필요성

범용 ASR 모델(Whisper 등)은 방대한 데이터를 학습하여 일반적인 상황에서는 뛰어난 성능을 보입니다. 하지만 우리가 다루는 **특수한  환경**은 모델이 학습한 '표준적인' 데이터 분포를 크게 벗어나 있습니다. 이로 인해 발생하는 주요 문제점은 다음과 같습니다.

#### 2.4.1 신호 대 잡음비($SNR$) 저하와 환각(Hallucination)
*   **기술적 배경:** $SNR(Signal-to-Noise Ratio)$이 낮아지면 음향 특징 추출기(Encoder)가 유효한 음성 특징과 배경 소음을 구분하지 못합니다.
*   **환각 현상(Hallucination):** Whisper와 같은 자기 회귀(Auto-regressive) 모델은 소리가 들리지 않거나 잡음만 들릴 때, **"무언가 말을 했을 것"**이라고 가정하고 문맥상 그럴듯한 문장을 스스로 창조해냅니다. 이는 정보의 정확성이 생명인 특수 임무 환경에서 치명적인 오류(Critical Error)가 됩니다.
*   **반복 루프(Repeat Loop):** 잡음 구간에서 특정 단어나 문장을 무한히 반복 출력하는 현상 역시 범용 모델이 잡음 환경을 정상적인 데이터로 인식하지 못해 발생하는 대표적인 오작동입니다.

#### 2.4.2 도메인 특화 어휘와 OOV(Out-of-Vocabulary) 문제
*   **문제 원인:** 범용 모델의 어휘 사전은 일상적인 대화에 최적화되어 있습니다. "브라보(Bravo)", "탱고(Tango)"와 같은 포네틱 코드나 "현 시각 기상", "좌표" 등 도메인 특화 용어는 모델에게 생소한 데이터입니다.
*   **대치 오류(Substitution Error):** 모델은 자신이 모르는 단어가 들리면 강제로 자신이 아는 가장 유사한 발음의 단어로 치환합니다.
    *   *예시:* "좌표 보고" $\rightarrow$ "조표 보고" / "송신 바람" $\rightarrow$ "성신 바람"
*   **결과:** 이러한 사소한 단어의 왜곡은 전체 문장의 의미를 완전히 변질시키며, 사후 교정 작업에 과도한 인적 자원을 소모하게 만듭니다.

#### 2.4.3 발화 중첩 및 간투어(Filler Words)에 의한 디코딩 붕괴
*   **발화 중첩:** 다자간 무선 통신에서 화자가 겹칠 경우, 모델의 어텐션(Attention) 가중치가 분산되어 핵심 화자의 음성을 놓치거나 두 화자의 말을 기괴하게 섞어서 출력합니다.
*   **간투어 처리:** 긴박한 상황에서 빈번한 "어...", "음..." 등의 간투어는 문장의 문맥 흐름을 방해합니다. 범용 모델은 이러한 간투어 뒤에 올 '진짜 의미 있는 단어'를 예측하는 데 어려움을 겪으며, 심한 경우 이후 문장 전체를 생략(Deletion Error)해 버리기도 합니다.

---

### 💡 결론: 왜 LoRA 기반 파인튜닝인가?

이러한 문제들은 단순한 오디오 전처리(Noise Reduction)만으로는 완벽히 해결되지 않습니다. **모델 자체가 우리 도메인의 소리 특징과 언어 습관을 배워야 합니다.**

1.  **데이터의 힘:** 우리가 보유한 특수 환경 데이터셋은 모델에게 "이런 잡음이 섞인 소리가 사실은 이런 글자다"라는 것을 가르치는 유일한 정답지가 됩니다.
2.  **LoRA(Low-Rank Adaptation)의 효율성:** 수십억 개의 파라미터를 가진 Whisper 모델 전체를 재학습하는 것은 막대한 컴퓨팅 자원(VRAM)이 필요하며 모델의 기존 지식을 파괴(Catastrophic Forgetting)할 위험이 있습니다. 
3.  **최적의 타협점:** LoRA 기법을 사용하면 모델의 핵심 지식은 보존하면서, **우리 도메인에 필요한 '미세한 조정 가중치'만을 학습**하여 H100과 같은 한정된 자원 내에서 가장 빠르고 정확하게 도메인 특화 모델을 구축할 수 있습니다.

# 제3장. AI 개발 환경 및 핵심 기술 용어

본 장에서는 음성 인식 모델(Whisper)을 학습하고 추론하는 데 근간이 되는 딥러닝 소프트웨어 환경과, 이를 물리적으로 가속하는 하드웨어 핵심 기술에 대해 살펴봅니다. 또한, 내부망(폐쇄망) 환경에서 성공적인 시스템 구축을 위해 반드시 알아야 할 라이브러리 의존성 관리 및 도커 반입 전략을 다룹니다.

---

### 3.1 딥러닝 프레임워크와 하드웨어 가속 구조

수천만 개에서 수십억 개의 파라미터(가중치)를 가진 딥러닝 모델을 원활하게 구동하기 위해서는 텐서 연산을 지원하는 프레임워크와 이를 뒷받침하는 강력한 GPU 가속 기술이 필수적입니다.

#### 3.1.1 PyTorch 기반 텐서 연산
*   **개념:** PyTorch는 Python 기반의 오픈소스 머신러닝 프레임워크로, 복잡한 신경망(Neural Network) 모델을 쉽고 직관적으로 설계하고 학습할 수 있게 해주는 가장 대중적인 도구입니다.
*   **핵심 역할:** 음성 데이터(오디오 파형)나 텍스트 데이터는 컴퓨터가 계산할 수 있도록 모두 숫자로 이루어진 다차원 행렬, 즉 **'텐서(Tensor)'**로 변환되어 모델에 입력됩니다. PyTorch는 이 거대한 텐서들의 곱셈과 덧셈 연산을 효율적으로 수행하며, 특히 학습 과정에서 모델의 오차(Loss)를 줄이기 위해 가중치를 업데이트하는 **'자동 미분(Autograd)'** 기능을 핵심적으로 제공합니다. Whisper의 OpenAI 공식 구현체 및 HuggingFace의 생태계 모두 PyTorch를 기반으로 작동합니다.

#### 3.1.2 NVIDIA GPU 병렬 연산 (CUDA)
*   **개념:** CUDA(Compute Unified Device Architecture)는 NVIDIA가 개발한 병렬 컴퓨팅 플랫폼이자 프로그래밍 모델입니다. 운영체제가 GPU 하드웨어와 직접 소통할 수 있도록 해주는 소프트웨어 계층(API)입니다.
*   **핵심 역할:** CPU가 복잡한 소수의 연산을 '직렬(순차적)'로 빠르게 처리하는 데 특화되어 있다면, GPU는 단순한 연산을 수천 개의 코어를 통해 **'동시(병렬)'**에 처리하는 데 특화되어 있습니다. ASR 모델의 학습은 수십억 번의 단순 행렬 곱셈으로 이루어지므로, CUDA를 통해 이 연산들을 GPU의 수많은 코어에 분산(Dispatch)시킴으로써 학습과 추론 속도를 CPU 대비 수십~수백 배 이상 폭발적으로 가속할 수 있습니다.

#### 3.1.3 딥러닝 최적화 라이브러리 (cuDNN)
*   **개념:** cuDNN(CUDA Deep Neural Network library)은 딥러닝 모델에서 빈번하게 사용되는 필수 알고리즘들을 GPU 환경에 맞춰 극도로 최적화해 놓은 NVIDIA의 전용 라이브러리입니다.
*   **핵심 역할:** CUDA가 GPU 하드웨어를 움직이는 '엔진'이라면, cuDNN은 딥러닝 연산의 효율을 극대화하는 '고급 윤활유'와 같습니다. 트랜스포머 아키텍처에 쓰이는 행렬 연산, 어텐션(Attention), 정규화(Normalization), 활성화 함수(Activation) 등의 기본 연산들이 가장 짧은 시간 내에 처리되도록 하위 수준(Low-level)에서 어셈블리어 단위로 최적화되어 있습니다. 따라서 PyTorch 소스코드가 완전히 동일하더라도 시스템 내 **cuDNN의 유무나 버전에 따라 실제 학습 속도는 매우 큰 차이**를 보입니다.

---

### 3.2 GPU 및 라이브러리 의존성 관리

AI 모델 구축 프로젝트에서 실무자들이 가장 빈번하게 실패를 겪고 많은 시간을 낭비하는 구간이 바로 하드웨어 드라이버와 소프트웨어 라이브러리 간의 "버전 불일치(Version Mismatch)" 문제입니다. 외부망 접속이 차단된 폐쇄망 환경에서는 이 문제가 더욱 치명적입니다.

#### 3.2.1 NVIDIA Driver, CUDA Toolkit, PyTorch 간 호환성 표
딥러닝 모델이 GPU를 정상적으로 인식하고 가속하기 위해서는 아래 3가지 계층의 버전이 완벽하게 맞물려야 합니다.
1.  **NVIDIA Display Driver:** 물리적 GPU 하드웨어를 OS(Linux/Windows)에 인식시키는 가장 밑단 드라이버 (예: 버전 535.xx)
2.  **CUDA Toolkit & cuDNN:** 드라이버 위에서 병렬 연산 C++ API를 제공하는 핵심 툴킷 및 가속 라이브러리 (안정화 조합 예: CUDA 12.1 + cuDNN 8.9)
3.  **PyTorch:** CUDA API를 호출하여 실제 파이썬 환경에서 모델 연산을 수행하는 딥러닝 라이브러리 (예: 버전 2.3.0+cu121)

*   **설치 원칙:** 무턱대고 최신 버전을 설치하면 안 됩니다. 내가 설치하려는 **PyTorch 버전이 지원하는 CUDA Toolkit 버전**을 가장 먼저 확인하고, 해당 CUDA 버전을 지원하는 **최소 NVIDIA Driver 버전**을 역순으로 확인하여 호환성 매트릭스에 맞게 설치해야 합니다.

#### 3.2.2 버전 불일치로 인한 OOM 및 런타임 에러 원인
의존성이 꼬일 경우 다음과 같은 심각한 오류가 발생합니다.
*   **CUDA 버전에 맞지 않는 PyTorch 설치:** GPU를 아예 인식하지 못하여 `AssertionError: Torch not compiled with CUDA enabled` 에러가 발생하며, 모든 연산이 강제로 CPU로 할당되어 학습이 사실상 불가능해집니다.
*   **드라이버와 CUDA Toolkit 불일치:** `CUDA error: system has unsupported display driver / cuda driver combination` 에러가 발생하며 프로세스가 즉시 종료됩니다.
*   **cuDNN 버전 불일치 또는 누락:** 코드는 정상적으로 실행되는 것처럼 보이나, 내부 최적화가 깨져 메모리 할당(Memory Allocation) 효율이 급감합니다. 이로 인해 알 수 없는 시점에 **OOM(Out of Memory)** 에러를 뱉고 학습이 중단되거나, 연산 속도가 비정상적으로 느려지는 현상이 발생합니다.

### 3.3 내부망 오프라인 개발 환경 구축

외부 인터넷 연결이 불가능한 폐쇄망 환경에서는 `pip install`과 같은 기본적인 패키지 설치 명령어조차 사용할 수 없습니다. 따라서 외부망(인터넷 접속 가능 PC)에서 미리 환경을 구성하고 이를 반입하는 철저한 오프라인 구축 전략이 요구됩니다.

#### 3.3.1 패키지 종속성 해결을 위한 Wheel/Tarball 미러링
인터넷이 연결된 환경에서 프로젝트에 필요한 모든 Python 라이브러리의 의존성(Dependencies)을 스캔하여 `.whl`(Wheel) 파일이나 `.tar.gz`(Tarball) 소스 파일로 한 번에 다운로드 받아야 합니다.
*   **오프라인 설치 전략:** `pip download -r requirements.txt` 명령어를 사용하여 관련된 수십 개의 종속 패키지를 로컬 디렉터리에 다운로드한 후, 내부망으로 반입하여 `pip install --no-index --find-links=./디렉터리패스 -r requirements.txt` 명령어로 네트워크 통신 없이 순수 로컬 파일만으로 설치를 진행합니다.

#### 3.3.2 Docker 컨테이너 기반 격리 환경(Image Freeze)
복잡한 의존성을 가진 AI 모델을 OS에 직접 설치하면 다른 프로젝트와 충돌할 위험이 높습니다. 이를 방지하기 위해 컨테이너 기술인 Docker를 적극 활용합니다.
*   **공식 사전 빌드(Pre-built) 이미지 사용의 중요성:** 깡통 리눅스(Ubuntu 등) 베이스 이미지에 CUDA, cuDNN, PyTorch 등을 직접 `apt-get`으로 일일이 설치하는 것은 의존성 충돌을 유발할 확률이 극도로 높으며 매우 비효율적입니다. 반드시 NVIDIA나 PyTorch에서 공식적으로 제공하는 사전 빌드 이미지(예: `pytorch/pytorch:2.3.0-cuda12.1-cudnn8-devel`)를 베이스 이미지로 사용해야 합니다. 공식 이미지는 하위 레벨의 C++ 컴파일러와 GPU 가속 라이브러리가 완벽하게 튜닝 및 검증되어 있으므로, 우리는 그 위에 프로젝트에 필요한 Python 패키지만 추가로 설치하면 됩니다.
*   **이미지 동결(Freeze) 및 추출:** 외부망에서 베이스 이미지 위에 필요한 패키지를 모두 설치하고 완벽하게 구동되는지 테스트합니다. 환경 구성이 끝나면 `docker save -o asr_env.tar asr_image:latest` 명령어를 통해 이미지를 단일 압축 파일(.tar)로 추출(Freeze)합니다.
*   **내부망 반입:** 추출된 tar 파일을 내부망 서버에 복사한 후 `docker load -i asr_env.tar` 명령어를 실행하면, 외부망과 100% 동일한 격리된(Isolated) 환경을 몇 분 만에 완벽하게 복제할 수 있습니다.

#### 3.3.3 HuggingFace 모델 캐시 및 사전 다운로드 전략
Whisper 모델 가중치(Weights)와 토크나이저(Tokenizer) 파일은 HuggingFace Hub에서 다운로드됩니다. 내부망에서는 이 API 통신이 차단되므로 다음과 같은 조치가 필요합니다.
*   **모델 사전 다운로드:** 외부 환경에서 `huggingface-cli download openai/whisper-large-v3` 등의 명령어를 사용해 모델의 가중치 파일(`.bin` 또는 `.safetensors`), `config.json`, `vocab.json` 등 전체 레포지토리 파일을 특정 폴더에 내려받습니다.
*   **로컬 캐시 경로 설정:** 내부망에 반입한 후, 코드 내에서 `model = WhisperForConditionalGeneration.from_pretrained("openai/whisper-large-v3")` 처럼 원격 저장소를 호출하던 방식을 `from_pretrained("/내부망/절대경로/whisper-large-v3")`와 같이 **로컬 폴더 경로**를 바라보도록 수정해야 모델 로드가 정상적으로 수행됩니다.

### 3.4 H100(Hopper) 하드웨어 최적화 가이드

보유 중인 **NVIDIA H100(Hopper 아키텍처)**은 기존 Ampere(A100) 아키텍처 대비 비약적인 성능 향상이 있으나, 이를 위해 요구되는 최소 소프트웨어 사양이 존재합니다.

*   **아키텍처 호환성:** H100의 성능을 100% 활용하기 위해서는 **NVIDIA Driver 525.60.13** 이상, **CUDA 12.0** 이상의 환경 구성을 강력히 권장합니다. (CUDA 11.8도 구동은 가능하나 최신 텐서 코어 가속 기능이 제한될 수 있음)
*   **GPU 가속 커넥터:** 호스트 OS의 GPU 자원을 Docker 내부로 전달하기 위해 `nvidia-container-toolkit` 설치가 필수입니다. 설치 후 `daemon.json` 설정에 `nvidia` 런타임을 등록해야 컨테이너 내부에서 `nvidia-smi` 명령어를 통한 GPU 인식이 가능해집니다.

### 3.5 효율적인 자산 관리 및 운영 전략

내부망 환경에서 AI 시스템의 유연한 운영을 위해 소프트웨어(이미지)와 데이터(모델 가중치)를 분리하여 관리하는 전략을 채택합니다.

#### 3.5.1 실행 환경과 데이터의 분리 (Decoupling)
*   **Docker 이미지 (코드/라이브러리):** PyTorch, CUDA, Whisper 실행 소스코드 등 변하지 않는 '환경'만 포함합니다. 이를 통해 이미지 크기를 최소화하여 배포 속도를 높입니다.
*   **외부 볼륨 마운트 (가중치 파일):** Whisper Large v3 모델 파일과 같이 용량이 크고 변경 가능성이 있는 데이터는 Docker 이미지 내부가 아닌, **서버의 물리적 스토리지**에 별도로 보관합니다.
*   **운영 이점:** 모델 버전 업데이트(예: 파인튜닝된 신규 가중치 적용) 시, 무거운 Docker 이미지를 다시 빌드하거나 반입할 필요 없이 서버 내의 특정 경로에 있는 파일만 교체하면 즉시 적용이 가능합니다.

#### 3.5.2 GPU 리소스 분할 및 격리
*   **컨테이너별 GPU 할당:** H100 4장을 보유한 환경에서는 각 컨테이너 실행 시 `--gpus '"device=0"'` 처럼 특정 GPU ID를 지정하여 할당합니다.
*   **효과:** 학습 컨테이너(Trainer)와 서비스 컨테이너(API Server)가 서로의 GPU 메모리를 침범하지 않도록 격리하여 시스템의 안정성을 확보합니다.

# Part 2. Whisper 아키텍처와 도메인 데이터 (데이터/전처리 편)

## 제4장. 핵심 ASR 모델 (Whisper) 심층 분석

본 장에서는 현재 프로젝트에서 베이스라인으로 채택한 OpenAI의 Whisper 모델을 심층적으로 분석합니다. 먼저, 모델 크기별 특성과 파라미터를 파악하여 실제 서비스 환경에 가장 적합한 모델을 선정하는 기준을 확립하고, 이후 타임스탬프 예측 원리 등 Whisper의 핵심 내부 동작 구조를 이해합니다. 이를 통해 후속 파인튜닝 작업에서 발생할 수 있는 다양한 문제를 사전에 예측하고 대응하는 토대를 마련합니다.

---

### 4.1 주요 최신 ASR 모델 비교

현재의 딥러닝 기반 ASR 모델들은 수천만 개에서 수십억 개의 파라미터(가중치)를 가진 대형 모델 중심으로 고도화되어 있습니다. 어떤 모델을 현재 프로젝트에서 베이스라인으로 채택하고 Whisper를 특별히 선택했는지 그 근거를 이해하기 위해서는, 동일 범주의 주요 모델들과의 비교 분석이 필수적입니다.

#### 4.1.1 Wav2vec 2.0 (자기지도 학습, Self-Supervised Learning)

*   **개념:** Meta(Facebook)에서 개발한 모델로, 레이블이 없는 수천~수만 시간의 비지도 원시 오디오 데이터만으로 음성의 핵심 표현(Representation)을 학습하는 자기지도 학습(SSL) 기반의 접근 방식을 채택합니다.
*   **장점:** 레이블(전사문) 작업에 드는 비용과 시간을 획기적으로 절감하면서도 준수한 성능을 달성할 수 있어, 전사 데이터가 부족한 저자원(Low-resource) 언어 환경에서 매우 유리합니다.
*   **단점:** 발화 겹침(Overlapping) 처리 및 도메인 전환 시의 강건성이 상대적으로 낮으며, 복잡한 음향 환경에서 성능 개선 폭이 제한적입니다. 학습 데이터가 충분한 상황에서는 지도 학습 기반 모델 대비 성능이 낮게 나타나는 경향이 있습니다.

#### 4.1.2 Conformer 구조 및 특징

*   **개념:** Google에서 개발한 ASR 아키텍처로, 트랜스포머의 전체 문맥 이해 능력(Global Context)과 CNN(합성곱 신경망)의 지역적 음향 패턴 인식 능력(Local Context)을 하나의 인코더 블록 내에서 결합한 혼합형 구조입니다.
*   **장점:** 인코더 단독 구조임에도 불구하고 자동 정렬(CTC)을 적용하여 전사 속도가 매우 빠르며, 짧은 발화 인식 분야(실시간 ASR 적용)에서 뛰어난 성능을 발휘합니다.
*   **단점:** 다국어 및 멀티태스크 처리에 대한 공식 확장이 Whisper에 비해 제한적입니다. 디코더 없이 CTC만으로 시퀀스를 생성하므로, 오류 교정 역할의 언어 모델링 능력이 구조적으로 약합니다.

#### 4.1.3 Whisper의 대규모 약지도 학습(Weakly Supervised) 및 Zero-shot 성능

*   **개념:** OpenAI가 인터넷에서 수집한 68만 시간이 넘는 방대한 양의 '자연 발생적인 음성'과 해당 오디오의 자막 등 '느슨한 레이블(Weak Label)'을 대조하는 약지도 학습(Weakly Supervised Learning) 방식으로 학습된 대규모 다국어·다목적 ASR 모델입니다. 데이터 수집 과정에서 완벽한 정렬을 강제하지 않아도 되므로 대규모 학습 데이터 확보가 가능했습니다.
*   **채택 근거:** 방대한 학습 데이터와 멀티태스크 구조 덕분에, 별도의 파인튜닝 없이(Zero-shot) 다양한 언어와 환경에서도 범용적으로 높은 수준의 전사 성능을 보입니다. 특히, 별도 학습 없이 새로운 언어를 텍스트로 변환하거나 심지어 영어로 번역(Translate)까지 수행하는 강력한 제로샷(Zero-shot) 다국어 처리 능력은 타 모델과 비교하여 가장 큰 차별점입니다. 이러한 이유로 **본 프로젝트에서는 Whisper large-v2 또는 large-v3 모델을 베이스라인으로 선정**하여 파인튜닝을 수행합니다.

---

### 4.2 Whisper 모델 크기별 특성과 성능 비교

Whisper는 파라미터 수에 따라 Tiny부터 Large까지 5단계의 크기로 제공됩니다. 각 단계에 해당하는 GPU 메모리(VRAM) 요구량, 추론 처리 속도(RTF), 그리고 주요 언어별 인식 오류율(WER/CER)을 비교하여 운영 환경에 맞는 모델을 선택해야 합니다.

#### 4.2.1 파라미터 수에 따른 크기별 성능 비교표

파라미터 수가 많아질수록 모델의 언어 이해 및 음향 모델링 능력이 정교해지며, 결과적으로 인식 오류율(WER: 낮을수록 우수)이 개선됩니다.

*   **Tiny (39M) / Base (74M) / Small (244M):** 컴퓨팅 자원이 극히 제한된 엣지(Edge) 환경에서의 실시간 처리에는 적합하나, 한국어처럼 형태론적으로 복잡한 언어에서는 WER이 20~40% 이상으로 실용적 수준에 미치지 못합니다. 특히 저자원 언어나 잡음 환경에서는 없는 말을 만들어내는 환각(Hallucination) 현상이 빈번하게 발생합니다.
*   **Medium (769M):** Small 모델 대비 인식 오류율이 눈에 띄게 개선됩니다. 다만, 한국어 등 비영어권 언어에서 VRAM 소비량과 배치 처리 속도 간의 균형이 깨지기 쉬워, 제한된 학습 자원 환경이나 빠른 처리가 요구되는 실시간 서비스에서는 운용상 단점을 드러냅니다.
*   **Large (1550M):** 15억 개(1.55B)의 파라미터를 가진 모델로, 현재 공개된 Whisper 계열 중 가장 높은 다국어 인식 성능으로 최고 수준의 WER을 달성합니다. 군사/안전 도메인의 높은 정확도 요구 사항을 고려하면 **실무상 Large(v2 또는 v3) 모델을 베이스라인으로 선정**하여 파인튜닝을 수행하는 것이 표준입니다.

### 4.2.2 언어별 성능 지표 비교 (English vs Multilingual)

Whisper는 99개의 언어를 지원하지만, 전체 학습 데이터의 언어별 구성 비율과 품질 차이로 인해 인식 성능에 편차가 존재합니다. 한국어(Korean) 기반 ASR 시스템의 목표 성능을 합리적으로 설정하기 위해, OpenAI의 공식 릴리즈 노트 및 벤치마크 자료를 바탕으로 성능 편차와 현실적인 파인튜닝 기대치를 분석합니다.

**[표] Whisper 모델 버전별 주요 언어 오류율 비교 (OpenAI 공식 벤치마크 기준)**

| 언어 | 측정 지표 | large-v2 | large-v3 | 파인튜닝 후 기대치 |
| :--- | :--- | :--- | :--- | :--- |
| 영어 (EN) | WER | 3 ~ 5% | 2.5 ~ 3% | 1~2% 수준 달성 가능 |
| 한국어 (KO) | CER | 6 ~ 10% | 5 ~ 8% | 3~5% 수준 목표 |
| 중국어 (ZH) | CER | 5 ~ 9% | 4 ~ 7% | 3~5% 수준 목표 |
| 일본어 (JA) | CER | 6 ~ 10% | 5 ~ 8% | 3~5% 수준 목표 |
| 러시아어 (RU) | WER | 6 ~ 10% | 5 ~ 8% | 3~5% 수준 목표 |
*(※ 위 수치는 Common Voice, FLEURS 등 표준 오픈 데이터셋 기준이며, 오차율은 낮을수록 우수함)*

**핵심 해석 및 실무 적용 지침**

*   **측정 지표 기준 (WER vs CER):** 영어, 러시아어 등 띄어쓰기로 단어가 명확히 구분되는 언어는 **WER(단어 오류율, Word Error Rate)**을 기본 지표로 사용합니다. 반면 한국어, 중국어, 일본어는 문자 단위의 정확도를 측정하는 **CER(문자 오류율, Character Error Rate)**을 실무 표준으로 삼습니다. 특히 OpenAI 공식 자료에서도 "한국어는 띄어쓰기를 생략하는 경우가 많아 오픈 데이터셋 정답지의 띄어쓰기 불일치가 심하다"고 지적하고 있으며, 이 때문에 한국어 모델의 실무 성능 평가는 CER에 의존하는 것이 정확합니다.
*   **버전 간 성능 격차와 아키텍처 변화:** OpenAI의 발표에 따르면, large-v3 모델은 large-v2로 수집한 400만 시간의 의사 라벨링(Pseudo-labeled) 데이터와 100만 시간의 약지도(Weakly labeled) 데이터를 혼합하여 추가 학습했습니다. 또한 입력 멜 주파수 빈(Mel frequency bins)을 80개에서 128개로 확장하여 고주파 대역의 해상도를 높였습니다. 그 결과 v2 대비 전반적인 다국어 인식 오류율이 **10~20% 상대적으로 감소**하였습니다.
*   **파인튜닝의 필요성 (실무 상황 고려):** 위 표의 수치(5~8%)는 팟캐스트나 정제된 음성 등 표준 발화 환경에서의 벤치마크 결과입니다. 실제 잡음이 심하게 섞인 **특수 환경** 음질에서는 그 이상의 치명적인 인식 오류가 발생하게 됩니다. 더불어 **특수 환경의 장비 특성 및 도메인 고유의 전문 용어(은어, 약어 등)**가 다수 존재하므로, 범용 모델을 그대로 서비스에 적용할 수 없습니다. 따라서 해당 도메인 환경에서는 **LoRA 파인튜닝을 통한 맞춤형 도메인 적응(Domain Adaptation)이 필수적**입니다.

#### 4.2.3 요구 VRAM 및 추론 속도(RTF) 분석

*   **VRAM 요구량:** Large 모델 기준, fp16(반정밀도) 정밀도로 추론(Inference) 시 최소 약 6~8GB의 VRAM이 필요합니다. 다만, **학습(LoRA 파인튜닝)** 시에는 역전파를 위한 그래디언트와 옵티마이저 상태가 GPU 메모리에 추가로 적재되므로, 배치 사이즈(Batch Size)에 따라 최소 24GB(RTX 3090/4090) ~ 80GB(A100/H100) 수준의 VRAM이 필요합니다.
*   **RTF (Real-Time Factor):** 실시간 처리 배수를 나타내는 지표입니다. RTF가 0.1이라면, 10초 분량의 음성을 단 1초 만에 처리할 수 있다는 의미입니다. 배치 처리 및 양자화(Quantization)를 적용하면 Large 모델임에도 RTF를 0.05 이하로 유지하는 것이 가능합니다.

---

### 4.3 내부 동작 구조와 타임스탬프 예측 원리

Whisper 모델이 오디오 신호를 입력받아 전사문과 타임스탬프를 출력하는 상세한 내부 처리 흐름을 이해하고, 데이터셋 구축 및 모델 파인튜닝 과정에서 이를 활용하는 방법을 파악합니다.

#### 4.3.1 로그 멜 스펙트로그램(Mel-Spectrogram) 변환 과정

*   **변환 이유:** 원시 오디오 파형(Waveform) 상태의 1차원 시계열 신호를 트랜스포머 모델이 처리하는 방식으로 바로 입력할 수 없습니다. 모델이 이해하기 유리한 '시간-주파수 2D 이미지' 형태로 변환하는 전처리 과정을 거칩니다.
*   **처리 절차:** 디지털화된 오디오 신호는 짧은 구간씩 쪼개어(Short-Time Fourier Transform, STFT) 시간 흐름에 따라 변화하는 주파수 에너지 분포를 나타내는 2D 행렬(80개의 멜 주파수 채널, v3의 경우 128채널)로 변환되고, 이 행렬은 2개의 1D 합성곱(Conv1D) 레이어를 거쳐 트랜스포머 인코더에 입력됩니다.
*   **중요사항:** Whisper는 처음부터 16kHz 샘플링 레이트의 오디오로 고정 학습되었습니다. 따라서 입력되는 오디오가 8kHz의 전화 음성이거나 48kHz의 고음질 레코딩이더라도, 전처리 과정에서 아무런 예외 없이 **16kHz로 리샘플링(Resampling)**하는 표준화 절차가 반드시 적용되어야 합니다.

#### 4.3.2 다국어(Multilingual) 처리 및 Task 토큰 (Translate/Transcribe)

Whisper는 추론을 시작하기 직전에 전용 제어 토큰을 모델에 주입(Prompting)하여 \"이번 오디오는 한국어이며, 전사(Transcribe)를 수행하라\"라는 태스크 조건을 명시적으로 지정하는 멀티태스크 구조를 갖습니다.

*   **토큰 구성 예시:** `<|startoftranscript|><|ko|><|transcribe|><|notimestamps|>`
*   **실무 의미:** 이 스페셜 토큰 구성(Prompting) 방식을 통해 모델 가중치를 전혀 수정하지 않고도 특정 입력 언어를 지정된 언어의 텍스트로 전사(Transcribe)하거나, 반대로 영어로 번역(Translate)하는 추론 방식을 자유롭게 전환할 수 있습니다.

#### 4.3.3 시간 토큰(Time-aligned tokens) 예측 원리와 타임스탬프 생성

Whisper는 별도의 전용 정렬(Alignment) 모델 없이도 전사문의 **"발화 시작 시간과 종료 시간"**을 시간 토큰과 텍스트 토큰을 함께 예측하는 방식으로 출력할 수 있습니다.

*   **구조:** 모델의 어휘(Vocabulary) 사전에는 0.00초부터 30.00초까지 0.02초 간격으로 세분화된 총 1,500개의 시간 토큰(`<|0.00|>` ~ `<|30.00|>`)이 일반 텍스트 토큰과 함께 등록되어 있습니다.
*   **출력 예시:** 모델이 오디오를 처리한 뒤 `<|0.00|>지금 <|2.50|><|2.50|>현장 상황 보고합니다<|3.80|>` 와 같이 텍스트 토큰 사이사이에 시간 토큰을 삽입하여 각 발화 구간의 타임스탬프 정보를 직접 생성합니다.

---

### 4.4 입력 길이 제한(30초)의 구조적 한계와 해결책

#### 4.4.1 3000 프레임 고정 위치 인코딩(Fixed Positional Encoding)

Whisper의 트랜스포머 구조는 오디오의 각 프레임이 전체 시퀀스에서 어느 위치에 있는지 모델에게 알려주기 위해 위치 인코딩(Positional Embedding)을 사용합니다. 이 위치 인코딩의 최대 길이가 오디오 30초에 해당하는 **1,500개(또는 3,000개) 프레임으로 고정**되어 있습니다. 따라서 한 번의 추론에서 처리할 수 있는 음성의 최대 길이는 구조적으로 **정확히 30초**로 제한됩니다.

#### 4.4.2 Sliding Window 기법을 활용한 긴 음성 처리

30초보다 긴 음성을 처리하기 위해서는 소프트웨어적 분할 전략이 필요합니다.

*   **VAD 기반 청크 분할(Chunking):** 30초가 넘는 음성은 VAD(Voice Activity Detection)를 활용해 발화 종료 지점(무음 구간)을 기준으로 자연스러운 경계에서 잘라, 30초 이내의 청크(Chunk) 단위로 모델에 순차 전달합니다.
*   **슬라이딩 윈도우(Sliding Window):** 모델 레벨에서 30초 이하로 창(Window)을 고정하고, 문맥 단절을 최소화하기 위해 이전 청크와 일정 부분 겹치도록(Overlap) 창을 이동시키며 처리합니다.
*   **실무 적용:** `Faster-Whisper`, `insanely-fast-whisper` 등 주요 추론 라이브러리는 이 슬라이딩 윈도우 로직을 내부적으로 자동 처리하는 래퍼(Wrapper) 형태로 구현되어 있어, 실무에서는 해당 라이브러리를 표준으로 사용합니다.

---

### 4.5 주요 Whisper 추론 구현체(Implementation) 비교

Whisper API를 직접 사용하더라도, 어떤 구현 라이브러리를 통해 추론(Inference)을 수행하느냐에 따라 속도와 VRAM 효율이 크게 달라집니다. 현재 ASR 시스템 개발 환경에서 검토한 3가지 주요 추론 구현체를 정리합니다.

#### 4.5.1 OpenAI Whisper (Original)

*   **개요:** OpenAI에서 공식적으로 배포하는 순수 Python/PyTorch 기반의 구현체입니다.
*   **장점:** 가장 표준적인 코드로 동작하여 디버깅과 구조 파악이 용이하며, 공식 지원 여부 확인이 명확합니다.
*   **단점 및 용도:** 추론 속도가 상대적으로 느리고 VRAM 효율이 낮습니다. 프로토타입 테스트 및 아이디어 검증에는 유용하나, 단독으로 대용량 배치 처리(Batch Processing)에 적용하기에는 성능적 한계가 있습니다.

#### 4.5.2 Faster-Whisper (CTranslate2 기반) ★ 실무 표준

*   **개요:** C++ 기반의 고성능 트랜스포머 추론 엔진인 `CTranslate2`를 백엔드로 활용하여 Whisper에 최적화된 추론 라이브러리입니다.
*   **장점:** 기존 구현체 대비 **추론 속도가 최대 4배 이상 향상**되며, `int8` 또는 `fp16` 양자화(Quantization)를 적용하여 VRAM 사용량을 대폭 줄이면서도 인식 정확도를 거의 유지합니다. (Large 모델을 8GB VRAM 환경에서도 원활하게 구동 가능)
*   **실무 적용:** 성능과 안정성의 균형이 가장 우수하여 현재 시점에서 **업계 사실상 표준(De Facto Standard)**입니다. 본 매뉴얼에서도 중점적으로 다루는 주요 추론 라이브러리입니다.

#### 4.5.3 insanely-fast-whisper (Flash Attention 2 기반)

*   **개요:** HuggingFace `transformers` 라이브러리 위에 **Flash Attention 2** 알고리즘을 접목하여 처리 속도를 극한까지 끌어올린 구현체입니다.
*   **장점:** GPU의 메모리 대역폭 병목을 해소하기 위해 어텐션 연산을 메모리 효율적으로 재구성하여, 배치 처리 시 `Faster-Whisper`보다도 더 높은 처리량(Throughput)을 달성할 수 있습니다.
*   **실무 적용:** H100(80GB HBM3) GPU 환경에서, 방대한 양의 음성 데이터를 일괄(Batch) 전사해야 할 때 매우 유리합니다. H100은 초당 3.35 TB/s에 달하는 HBM3 메모리 대역폭과 TensorCore를 활용하여, A100 대비 2~3배 이상의 추론 속도 향상을 기대할 수 있어 대규모 데이터 전처리 파이프라인에 적극 활용됩니다.

# 제5장. 음성 데이터 파이프라인 및 전처리 (Pre-processing)

성공적인 ASR 파인튜닝의 8할은 양질의 데이터 구축에 있습니다. 본 장에서는 수집된 원시 음성 데이터를 Whisper 모델이 학습할 수 있는 최적의 상태로 가공하는 일련의 파이프라인을 구축합니다.

---

### 5.1 원시 데이터(Raw Data) 수집 및 포맷 통일

다양한 소스(무전기 녹음, CCTV, 스마트폰 등)에서 수집된 오디오는 포맷, 샘플링 레이트, 채널 수가 제각각입니다. 이를 일관된 형태로 통일하는 작업이 파이프라인의 첫 단추입니다.

#### 5.1.1 샘플링 레이트(Sampling Rate)와 Bit-depth 이해 (16kHz, 16-bit 필수)
*   **샘플링 레이트(Hz):** 1초의 소리를 몇 번 쪼개서 디지털로 기록할 것인가를 의미합니다. (예: CD 음질은 44.1kHz). 
*   **통일 규격:** Whisper 인코더는 16kHz로 고정 학습되어 있습니다. 만약 8kHz의 전화망 음성을 그대로 넣으면 모델이 제대로 특징을 잡지 못합니다. 따라서 모든 데이터는 **16kHz 샘플링 레이트, 16-bit Bit-depth, Mono(단일 채널)** 형태의 비압축 WAV 파일로 일괄 리샘플링 및 변환되어야 합니다.

#### 5.1.2 FFmpeg를 활용한 대규모 오디오 일괄 변환 스크립트 작성
오디오/비디오 처리의 표준 도구인 `ffmpeg`를 Python의 `subprocess` 모듈로 호출하여 수십만 개의 파일을 병렬 변환하는 스크립트가 필요합니다.
*   **변환 명령어 예시:** 
    `ffmpeg -i input.mp3 -ar 16000 -ac 1 -c:a pcm_s16le output.wav`
    (의미: 샘플링 레이트 16000, 오디오 채널 1개(Mono), 16-bit PCM 코덱 사용)

### 5.2 실전 도메인 잡음 특성 분석 및 복원(Denoising) 기법

단순한 일상 대화와 달리 군사/안전 도메인의 무선 통신 데이터는 치명적인 환경 소음과 기계적 왜곡을 동반합니다. 도메인별 잡음과 감쇄 특성을 정확히 이해해야 이를 복원하는 최적의 전처리 필터를 설계할 수 있습니다.

#### 5.2.1 육상/해상/항공 도메인별 환경 노이즈 특성
*   **육상 (Land):** 차량 주행 소음과 디젤 엔진에서 발생하는 저주파 진동(rumble)이 배경에 지속적으로 깔리며, 도심이나 야외 환경에서 발생하는 문 닫힘 소리, 물체 충돌음, 공사 소음 등과 같은 충격성 소음(impulse noise)이 음성을 순간적으로 가리는 현상이 빈번하게 발생합니다.
*   **해상 (Sea):** 함정 내 거대한 엔진/스크류 소음, 파도 치는 소리, 함교의 매서운 바닷바람 소리(Wind buffeting)가 넓은 주파수 대역에 걸쳐 백색 잡음(White Noise) 형태로 지속적으로 유입됩니다.
*   **항공 (Air):** 헬기나 전투기의 로터(Rotor) 회전음, 제트 엔진의 고주파 소음(Whine), 조종석 내부 산소마스크의 쇳소리 나는 호흡음이 목소리에 섞여 들어오며, 음성이 매우 탁하고 기계음처럼 들리는 특성이 있습니다.

#### 5.2.2 무전 교신 코덱 및 신호 왜곡/감쇄 특성
*   **협대역(Narrowband) 주파수 절단:** 일반적인 아날로그 VHF/UHF 무전이나 통신망은 대역폭을 아끼기 위해 사람 목소리의 핵심 대역인 300Hz ~ 3400Hz 사이만 전송하고 위아래 주파수는 싹둑 잘라냅니다. 이로 인해 음성이 "오래된 전화기 너머로 듣는 듯한" 먹먹한 소리로 변질됩니다.
*   **저비트레이트 보코더(Vocoder) 왜곡:** 보안을 위한 비화기(암호화) 장비나 위성 통신에서는 데이터 크기를 극단적으로 줄이는 CVSD, MELP 등의 코덱을 사용합니다. 원본 파형을 버리고 특징점만 추출해 전송하므로, 복원 시 '로봇 쇳소리(Artifacts)'와 같은 심각한 디지털 왜곡이 발생합니다.
*   **신호 감쇄(Attenuation) 및 페이딩(Fading):** 산악 지형, 해상의 기상 악화, 또는 고속 비행으로 인해 전파가 약해지면(감쇄), 소리가 지지직거리며 끊기거나 화자의 볼륨이 들쭉날쭉해지는 페이딩 현상이 일어납니다.

#### 5.2.2-A 왜곡 유형별 ASR 정확도 영향도 분석

각 왜곡 유형이 Whisper 계열 모델의 음성 인식 오류율(CER)에 미치는 메커니즘과 영향 규모를 분석합니다. 이 분석은 전처리 파이프라인의 **우선순위 결정**과 **학습 데이터 선별 기준** 설정에 직접 활용됩니다.

**[표] 왜곡 유형별 ASR 정확도 영향도 요약**

| 왜곡 유형 | 주요 발생 환경 | 영향 메커니즘 | CER 상승 추정 | 전처리 우선순위 |
| :--- | :--- | :--- | :---: | :---: |
| 협대역 주파수 절단 (300~3,400Hz) | VHF/UHF 아날로그 무전 | 고주파 음소(ㅅ, ㅈ, ㅊ 등 마찰음) 유실 → 음소 구분 불가 | +15~30% | ★★★★☆ |
| 저비트레이트 보코더 왜곡 (MELP/CVSD) | 비화기, 위성 통신 | 포먼트(Formant) 왜곡으로 모음 경계 붕괴, 환각 유발 | +30~60% | ★★★★★ |
| 임펄스 노이즈 (총성/폭음) | 육상 전투 환경 | 짧고 강한 과부하 신호로 주변 수백 ms 구간의 특징 벡터 오염 | +10~25% | ★★★☆☆ |
| 백색 잡음 지속 유입 (엔진/로터음) | 해상·항공 환경 | SNR 저하로 전반적 인식률 감소, 무음 구간 환각 증가 | +20~40% | ★★★★☆ |
| 신호 감쇄/페이딩 | 산악·원거리 교신 | 볼륨 급변으로 특징 추출 불안정, 발화 구간 누락 | +25~50% | ★★★★★ |
| 협채널 위상 간섭 (스테레오 믹스다운) | 다채널 녹음 | 위상 상쇄(Phase Cancellation)로 특정 주파수 대역 소실 | +5~15% | ★★☆☆☆ |

> **추정값 주의:** 위 CER 상승 수치는 Whisper Large-v3 베이스 모델(파인튜닝 전) 기준의 참고 범위입니다. 실제 수치는 잡음 강도(SNR), 언어, 화자 수에 따라 크게 달라집니다.

**왜곡 유형별 상세 영향 메커니즘:**

*   **협대역 주파수 절단의 영향:** 한국어에서 ㅅ(s), ㅈ(ts), ㅊ(tsh) 등의 마찰음·파찰음은 3,000Hz 이상의 고주파 성분으로 구분됩니다. 300~3,400Hz로 제한된 협대역 환경에서는 이 고주파 성분이 완전히 유실되어, 음향 모델이 마찰음을 인식하지 못하고 유사 음소로 치환하는 대체 오류(Substitution Error)가 집중적으로 발생합니다. 예를 들어 "송신"이 "종신", "위치"가 "위지"로 잘못 전사되는 현상이 빈번합니다.

*   **보코더 왜곡의 영향:** MELP 등의 저비트레이트 코덱은 음성을 물리적 파형이 아닌 음원(Source)과 필터(Filter)로 분리하여 전송하는 선형 예측 방식을 사용합니다. 복원 과정에서 포먼트 주파수가 미세하게 이동하거나 왜곡되면, Whisper의 멜 스펙트로그램 인코더가 해당 모음의 특징 벡터를 다른 모음으로 인식하게 됩니다. 이는 모음 오류가 연쇄적으로 이어지는 **디코딩 붕괴(Cascading Error)** 현상을 유발하여 CER 상승 폭이 가장 큽니다. 전처리 우선순위 1위로 분류되는 이유입니다.

*   **신호 감쇄/페이딩의 영향:** Whisper 인코더는 입력 오디오를 내부적으로 -1.0~1.0 범위로 정규화하지만, 짧은 구간(수백 ms) 내에서 볼륨이 급격히 변동하는 페이딩 환경에서는 이 정규화가 추종하지 못합니다. 볼륨이 갑자기 낮아진 구간에서는 멜 스펙트로그램의 에너지 표현이 무음에 가까워지고, VAD가 해당 구간을 무음으로 오판하여 발화 자체를 누락(Deletion Error)시키는 가장 치명적인 결과를 낳습니다.

**전처리 우선순위 결정 원칙:**

위 영향도 분석을 바탕으로, 제한된 전처리 리소스 내에서 최대의 인식률 향상 효과를 얻기 위한 우선순위는 다음과 같습니다.

1.  **최우선 (즉각 적용):** 보코더 왜곡 대응 (생성적 복원 모델) + 신호 감쇄 대응 (AGC/볼륨 정규화)
2.  **고우선 (필수 적용):** 협대역 주파수 절단 대응 (Band-pass Filter 및 고주파 대역 복원)
3.  **중간 (상황별 적용):** 백색 잡음 억제 (Spectral Gating) + 임펄스 노이즈 제거
4.  **저우선 (필요 시 적용):** 위상 간섭 대응 (스테레오 → 모노 변환은 필수이나 위상 처리는 선택)

#### 5.2.3 파형 복원 및 음질 개선 파이프라인
위와 같이 심하게 훼손된 신호를 AI가 인식하기 좋게 복원하기 위해 다음과 같은 전처리 기술들을 순차적으로 적용합니다.
*   **대역 통과 필터링 (Band-pass Filtering):** 300Hz 미만의 저주파 엔진 진동음과 4000Hz 초과의 고주파 기계 소음을 강제로 잘라내어(Cut-off), 모델이 사람 목소리에만 집중할 수 있도록 돕습니다.

| 도메인 | 권장 통과 대역 |
|---|---|
| 해상 VHF | 300Hz ~ 3,000Hz |
| 항공 VHF | 300Hz ~ 2,500Hz |
| 육상 전술망 | 300Hz ~ 3,400Hz |

*   **스펙트럴 게이팅 (Spectral Gating):** 해상/항공에서 지속적으로 깔리는 백색 잡음의 주파수 프로필을 분석한 뒤, 말소리가 있는 구간에서 해당 노이즈 주파수의 크기만큼만 빼주는(Subtraction) 고전적 DSP 기법입니다. (Python `noisereduce` 등 활용)
*   **볼륨 정규화 및 자동 이득 제어 (Normalization & AGC):** Whisper 모델은 내부적으로 오디오를 자체 정규화하므로 어느 정도의 볼륨 차이는 스스로 극복합니다. 하지만 1) 소리가 너무 작아 밑바닥에 깔린 환경 잡음(Noise Floor)과 구분이 안 될 때, 2) 전파 감쇄(Fading)로 인해 짧은 구간 내에서 소리가 커졌다 작아지기를 급격히 반복할 때, 3) 소리가 너무 커서 파형이 잘려나가는(Clipping) 현상이 발생할 때는 전사 정확도가 뚝 떨어집니다. 따라서 ASR 입력 전단에서 볼륨을 일정한 표준 레벨(LUFS 알고리즘)로 평탄하게 맞추어 주는(AGC) 작업은 인식률을 일관되게 유지하는 데 매우 중요합니다.
*   **복원 시 주의점:** 딥러닝 기반의 강력한 음성 복원(Speech Enhancement) 모델을 사용하여 노이즈를 100% 제거하려는 시도는 오히려 화자 고유의 발음 특성(Formant)까지 깎아먹어 ASR 성능을 치명적으로 떨어뜨릴 수 있습니다. 노이즈를 '완벽히 없애는 것'이 아니라 '문맥을 유추할 수 있을 정도로 덜어내는 것'이 핵심 원칙입니다.

#### 5.2.4 심각한 노이즈 환경의 차세대 음성 복원 전략(연구/실험적 접근)
기존의 고전적 DSP 방식이나 단순 딥러닝 마스킹(Masking) 방식으로 심각한 소음(Severe Noise)을 무리하게 깎아내려다 보면, 목소리 성분까지 함께 깎여나가 소리가 깡통처럼 들리는 '아티팩트(Artifact)' 현상이 필연적으로 발생합니다. 이를 극복하기 위해 최근에는 소음을 지우는 것을 넘어 **파형을 새로 그려내는 최첨단 생성적 복원 프로세스**가 도입되고 있습니다.

**1. 2단계 하이브리드 파이프라인 (Cleaning + Resynthesis)**
가장 안정적인 실무 접근법으로, 1차 필터링 후 2차 생성을 수행합니다.
*   **Step 1. 소음 억제 (CNN 기반):** MP-SENet이나 DEMUCS 같은 모델을 사용하여 스펙트로그램 상에서 배경 소음을 1차적으로 걷어냅니다. 목소리가 다소 왜곡되더라도 '신호 대 잡음비(SNR)'를 우선적으로 확보합니다.
*   **Step 2. 생성적 복원 (Generative):** 1차로 닦인 음성을 VoiceNoNG, Voicebox 등 생성 모델에 넣습니다. 모델은 남아있는 목소리의 파편을 분석해 "이 사람은 원래 이런 목소리였을 것"이라고 유추하며 빠진 주파수 성분을 다시 그려냅니다.

**2. 디퓨전 및 플로우 매칭 (Diffusion & Flow Matching)**
*   **작동 원리:** 깨끗한 음성에 노이즈를 섞는 과정을 역으로 학습한 모델입니다. 아주 심각한 노이즈(화이트 노이즈, 기계음 등) 상태를 '완전한 소음'으로 간주하고, 점진적으로 음성 신호만 추출해냅니다.
*   **장점:** 갑자기 터지는 폭음이나 말소리가 겹치는 등의 비정상적인 소음 복구에 매우 강합니다. (대표 모델: DiffSpeech, AudioLDM)

**3. 신경망 코덱 기반 시맨틱 재합성 (Neural Codec + Semantic Resynthesis) ★ 차세대 핵심 전략**

최근 음성 AI 분야에서 가장 주목받는 접근 방식으로, 잡음이 심한 오디오를 물리적 파형(Waveform) 단계에서 닦아내는 것이 아니라 **의미를 가진 코드(Token)로 해석한 뒤 깨끗한 목소리로 다시 출력**하여 ASR 모델에 입력하는 '생성형 음성 복원(Generative Speech Enhancement)' 또는 '시맨틱 재합성(Semantic Resynthesis)' 방식입니다. 기존의 통계적 노이즈 제거 방식보다 훨씬 강력한 성능을 보여주며, 기술적으로 충분히 실현 가능합니다.

**[심화] ASR 모델과 Neural Codec 복원 모델이 소리를 다루는 방식의 차이**

두 모델은 같은 오디오를 입력받아도 '목표'가 근본적으로 다릅니다. 이 차이를 이해하는 것이 생성적 복원이 왜 강력한지를 설명하는 핵심입니다.

| 구분 | ASR (Whisper 등) | Neural Codec (EnCodec 등) |
| :--- | :--- | :--- |
| **목표** | 소리 → 정확한 **텍스트** | 소리 → **인간 목소리 패턴**으로 압축 |
| **판단 방식** | '사과'인지 '파파'인지 확률이 비등하면 확신을 잃고 전사 포기 또는 오인식 | '사과'의 뜻은 몰라도 "잡음 속에서 성대가 낼 수 있는 가장 확률 높은 주파수 형태"를 탐색 |
| **실패 시 결과** | 오인식·환각·누락 (결과값이 틀리면 끝) | 부정확한 복원이지만 ASR이 읽기 쉬운 형태로 근사화 |
| **비유** | "이 글자가 뭐야?"라는 시험 | "가려진 부분에 어울리는 그림을 그려봐" (Inpainting) |

*   **잠재 공간(Latent Space)의 역할:** 소리가 오염되어도 리듬, 에너지 강약, 화자 음색 같은 '뼈대'는 미세하게 남아있습니다. 신경망은 이 희미한 흔적을 보고 "이건 80% 확률로 이런 형태의 음성 토큰일 것"이라고 예측합니다.
*   **언어 모델(LM)의 개입:** Qwen3-TTS 같은 최신 모델은 단순 소리 복원에 그치지 않습니다. 앞뒤 음성 토큰의 흐름을 보고 "이런 발음 뒤에 저런 발음이 오는 것이 자연스럽다"는 언어적 통계를 활용합니다. 소리가 아예 안 들리는 부분도 모델이 보유한 지식을 총동원하여 '깨끗한 목소리로 상상해서 채워 넣는(Infilling)' 것이 가능합니다.

---

**① 핵심 원리: 잠재 공간(Latent Space)을 통한 3단계 정보 추출**

일반적인 음성 인식은 잡음이 섞인 스펙트로그램을 그대로 해석하려다 오류가 발생합니다. Neural Codec(EnCodec, DAC 등)을 활용한 방식은 다음 3단계를 거칩니다.

*   **Step 1. 이산 토큰화(Discrete Tokenization):** 오디오를 유한한 숫자의 집합(Codebook)으로 변환합니다. 이때 신경망은 잡음 속에서도 "이 소리는 'ㄱ' 발음이며, 특정 억양을 가졌다"는 핵심 특징(Feature)을 포착하여 토큰으로 압축합니다. 물리적 잡음 성분은 이 과정에서 자연스럽게 걸러집니다.
*   **Step 2. 언어 모델링(Language Modeling):** Qwen-Audio나 최신 TTS 모델들처럼 LLM 아키텍처를 사용하여, 오염된 토큰 시퀀스에서 문맥상 가장 적절한 '깨끗한 토큰'이 무엇인지 예측(Infilling/Denoising)합니다. 앞뒤 문맥을 통해 손실된 정보를 추론하므로, 물리적으로 복원 불가능한 구간도 의미 단위에서 재구성이 가능합니다.
*   **Step 3. 재합성(Resynthesis):** 예측된 깨끗한 토큰을 Neural Codec의 디코더에 통과시켜 잡음이 완벽하게 제거된 음성 파형을 생성합니다. (대표 모델: Qwen3-TTS, EnCodec, DAC)

**② ASR 모델 적용 시의 장점**

*   **인식률(WER/CER)의 획기적 개선:** ASR 모델(Whisper 등)이 가장 힘들어하는 '비정형 잡음(갑작스러운 소음, 심한 에코, 보코더 왜곡)' 상황에서 모델이 이해하기 쉬운 깨끗한 파형을 미리 만들어 주기 때문에 인식 정확도가 급상승합니다. 5.2.2-A에서 분석한 보코더 왜곡(+30~60% CER 상승)이나 신호 감쇄(+25~50% CER 상승) 상황에서의 효과가 특히 큽니다.
*   **강력한 복원력:** 신호 감쇄가 심해 물리적 증폭으로는 한계가 있는 데이터도 모델이 '문맥'을 통해 단어를 추론하여 복원하므로, 기존 방식으로는 판독 불가능한 파일도 전사가 가능해집니다.

**③ 실무 적용 시 반드시 고려해야 할 리스크**

이 방식은 강력하지만 '생성형(Generative)' 모델이라는 점에서 다음 리스크를 반드시 관리해야 합니다.

| 리스크 | 내용 | 대응 방안 |
| :--- | :--- | :--- |
| **환각(Hallucination)** | 음성이 심하게 오염된 경우 모델이 실제와 다른 단어로 '그럴듯하게' 복원할 위험이 있으며, 전사 데이터의 신뢰도 문제로 이어질 수 있음 | 복원 전·후 음성을 함께 Whisper에 입력하여 신뢰도 점수를 비교하고, 신뢰도가 낮은 구간은 HITL 검수로 연계 (→ 6장 참조) |
| **연산 자원 소모** | 단순 노이즈 캔슬링 대비 고사양 GPU(H100 등) 자원을 대량 소모 | H100 온프레미스 환경에서 배치 추론(Batch Inference)으로 처리량 최적화. 실시간 처리 구간과 일괄 처리 구간을 분리 운용 |
| **화자 식별 정보 변형** | 재합성 과정에서 원래 화자의 고유한 음색이 미세하게 변할 수 있음 | **전사(Transcription) 목적에서는 큰 문제 없음.** 단, 화자 인증(Speaker Verification) 목적으로도 오디오를 활용해야 하는 경우 원본 파일을 별도 보관 |

**④ 왜 전사가 안 되던 파일이 복원 후에는 가능해지는가?**

ASR 모델(Whisper)은 입력된 파형에 잡음이 많으면 그 잡음을 연산에 그대로 포함합니다. 반면, 복원 모델을 거치면 다음 세 가지 변화가 발생하여 ASR이 처리 가능한 상태로 바뀝니다.

*   **노이즈 강제 제거:** 복원 모델은 '인간의 목소리 토큰'만 생성하도록 학습되었으므로, 기계 소음·백색 잡음은 아예 모델의 '표현 가능한 단어장(Codebook)'에 존재하지 않습니다. 따라서 결과 파형에서 잡음이 물리적으로 삭제됩니다.
*   **신호 강화:** 전파 감쇄로 약해진 구간은 모델이 보유한 '표준적인 목소리 분포 데이터'를 바탕으로 재합성(Resynthesis)되어 선명해집니다. 물리적 증폭과 달리 잡음도 함께 키워지지 않습니다.
*   **ASR 친화적 입력 생성:** 이렇게 만들어진 '가짜지만 깨끗한' 음성은 ASR 모델 입장에서 훨씬 결정론적(Deterministic)인 데이터입니다. Whisper가 모호한 확률 분포 대신 뚜렷한 음소 패턴을 인식할 수 있어, 전사 불가 구간이 전사 가능 구간으로 전환됩니다.

**※ 실무자를 위한 '심각한 노이즈' 복원 워크플로우 (H100 인프라 기준)**
온프레미스 대규모 GPU 환경에서 이를 직접 구현할 때의 추천 순서는 다음과 같습니다.
1.  **데이터 전처리:** 오디오를 16kHz 또는 24kHz로 리샘플링하고 DC Offset을 제거합니다.
2.  **1차 Denoising:** `DEMUCS` (Pre-trained) 모델을 가동하여 큰 소음 대역을 걷어냅니다.
3.  **2차 Restoration:** 1차 결과물을 `Voicebox`나 `Qwen3-TTS`의 인퍼런스 모드에 넣어 목소리의 고유 질감과 손실된 대역을 복원합니다.
4.  **Post-Processing:** `Vocos` 같은 고성능 보코더(Vocoder)를 사용하여 최종 출력물의 해상도(Super-resolution)를 끌어올립니다.

> ⚠️ **복원 한계 구간 처리 원칙**: 위 워크플로우를 모두 거쳐도 STT 인식 가능 수준에 미치지 못하는 구간은 **학습 데이터에서 제외**하는 것이 원칙이다. 품질 하한선 이하의 데이터를 억지로 포함하면 모델이 복구 불가능한 잡음 패턴을 '정상 입력'으로 학습하여 전반적인 인식률을 오히려 낮춘다.
>
> 복원 한계 구간을 줄이려면 전처리 파이프라인 자체를 개선하는 것이 올바른 방향이다. 다음 순서로 접근한다.
> 1. **SNR 측정 선행**: 인식 실패 구간의 SNR을 실측하여 복원 불가 기준값을 정한다.
> 2. **복원 모델 교체 또는 추가**: 파이프라인의 1차·2차 복원 모델이 해당 잡음 유형(예: 로터음, 보코더 왜곡)에 특화되어 있는지 검토하고 교체한다.
> 3. **데이터 증강으로 내성 확보**: 복원 한계를 인정하고, 해당 잡음 유형을 학습 데이터에 합성(Data Augmentation)하여 모델 자체가 열악한 입력에 강해지도록 훈련한다. (→ 6.4절 참조)

### 5.3 발화 구간 추출 (VAD: Voice Activity Detection)

음성 파일 내에는 실제로 말을 하지 않는 긴 무음(Silence) 구간이 존재합니다. 이 무음 구간을 그대로 학습이나 추론에 넣으면 심각한 문제가 발생합니다.

#### 5.3.1 VAD의 필요성 (환각 방지 및 초고속 전사)
*   **연산량 감소와 고속 전사 (Speed-up):** 무전 통신이나 관제망 녹음 파일의 경우, 하루 24시간 중 실제로 사람이 말하는 '유효 발화 구간'은 10~20% 내외에 불과하며 나머지 80% 이상은 단순한 대기 상태의 무음이나 기계적 잡음입니다. 전체 파일을 그대로 ASR 엔진에 넣으면 무의미한 잡음 구간까지 무거운 트랜스포머 연산을 수행하므로 막대한 GPU 자원과 시간이 낭비됩니다. VAD를 통해 **'목소리가 존재하는 구간'만 정확히 발췌(Crop)하여 모델에 던지면, 처리해야 할 오디오 데이터 양이 획기적으로 줄어들어 동일한 하드웨어로도 수배 이상 빠른 초고속 전사 처리가 가능**해집니다.
*   **환각(Hallucination) 및 오류 방지:** Whisper 모델은 무음이나 지속적인 잡음이 길게 이어지면 "어떤 말이 있는데 내가 놓친 것일까?"라고 착각하여, 이전에 했던 말을 무한히 반복하거나(Repetition) 전혀 존재하지 않는 엉뚱한 문장을 지어내는 고질적인 버그가 있습니다. VAD를 거쳐 순수한 발화 구간만 입력하면 이러한 치명적 오류를 사전에 완벽히 예방할 수 있습니다.
*   **자연스러운 청크 분할 (Chunking):** Whisper는 한 번에 최대 30초까지만 인식할 수 있습니다. 1시간짜리 녹음을 단순히 기계적으로 30초씩 자르면 단어 중간이 잘려나가 인식률이 떨어집니다. 따라서 VAD가 찾아낸 '말이 끝나는 지점(무음 구간)'을 기준으로 오디오를 자연스럽게 잘라(예: 5초~15초 단위) 모델에 전달하는 파이프라인 구성이 필수적입니다.

#### 5.3.2 최신 VAD 모델 활용 및 임계값(Threshold) 튜닝
과거에는 음량(Volume) 기반으로 무음을 잘라냈으나, 잡음 환경에서는 노이즈의 음량이 커서 구분이 불가합니다. 
*   **Silero VAD:** 최근 가장 널리 쓰이는 가벼운 딥러닝 기반 VAD 모델로, 소리의 크기가 아니라 음향적 특징을 분석해 사람의 '목소리'인지 '잡음'인지를 기가 막히게 구분해 냅니다.
*   **임계값(Threshold) 설정:** "이 소리가 목소리일 확률이 0.5 이상이면 잘라라"와 같이 타겟 도메인 환경의 노이즈 수준에 맞추어 Probability Threshold 값을 조정하는 튜닝 작업이 데이터 정제 파이프라인의 핵심입니다.

#### 5.3.3 극심한 잡음 및 감쇄 환경에서의 VAD 한계점과 대응
뛰어난 딥러닝 기반의 VAD 모델이라 할지라도, 특수 도메인의 극악한 환경에서는 다음과 같은 뚜렷한 한계를 노출합니다.
*   **노이즈를 음성으로 오인 (False Positive):** 전투기 로터음이나 무전기의 지속적인 쇳소리가 사람의 성대 진동(Voiced sound) 주파수 패턴과 우연히 겹칠 경우, VAD가 이를 사람의 목소리로 착각하여 무의미한 잡음 구간을 계속 발췌해버립니다. 이 경우 앞서 언급한 '고속 전사'라는 VAD 도입 목적이 퇴색됩니다.
*   **감쇄된 음성 누락 (False Negative):** 반대로 전파 감쇄가 심각하여 목소리가 주변 환경 소음 바닥(Noise Floor) 아래로 파묻힌 경우, 실제로는 화자가 교신 중임에도 VAD가 이를 단순 잡음으로 치부하여 음성 구간 자체를 통째로 날려버립니다. 이는 모델이 문맥을 인식할 기회조차 박탈당하는 가장 치명적인 데이터 유실(Data Loss)입니다.
*   **실무 대응 방안:** 이러한 VAD의 맹점을 극복하기 위해 실무에서는 1) 앞서 5.2절의 필터링(대역 통과, AGC 등)을 최우선적으로 거쳐 신호를 1차 정제한 뒤에 VAD를 통과시키는 **순차적 파이프라인**을 구축하고, 2) VAD의 확률 임계값(Threshold)을 의도적으로 매우 낮게(예: 0.1~0.2) 설정하여, 불필요한 노이즈 구간을 조금 더 떠안더라도 진짜 음성 구간을 놓치는 일(False Negative)을 철저히 막아내는 보수적인 세팅을 채택합니다.

### 5.4 데이터 정합성 검증 및 품질 필터링 (Data Validation)

ASR 시스템의 성능, 특히 LoRA와 같은 미세 조정(Fine-tuning)의 성패는 학습 데이터의 양보다 **'정답지(Ground Truth)의 무결성'**에 의해 결정됩니다. 아무리 방대한 데이터라도 음성과 텍스트가 어긋나 있다면 모델은 잘못된 매핑을 학습하게 되어 오히려 성능이 퇴화합니다. 따라서 전처리 파이프라인의 마지막 단계에서는 다음과 같은 엄격한 통계적·구조적 검증 로직을 반드시 통과해야 합니다.

#### 5.4.1 음성-텍스트 대응 밀도 검증 (CPS 분석)

한국어 발화의 물리적 특성을 바탕으로 음성 데이터의 길이와 전사된 텍스트의 양이 상식적인 범위 내에 있는지 검증합니다. 이를 위해 **CPS(Characters Per Second, 초당 글자 수)** 지표를 활용합니다.

*   **CPS 분석의 원리:** 일반적인 한국어 대화의 발화 속도는 초당 약 5~15자 내외입니다. 만약 10초짜리 음성에 텍스트가 2글자뿐이거나(0.2 CPS), 반대로 100글자가 적혀 있다면(10.0 CPS 이상이나 문맥에 따라 상이), 이는 VAD의 오작동으로 무음 구간이 과도하게 포함되었거나 전사 과정에서 텍스트가 누락/중복되었음을 의미합니다.
*   **도메인 특수성 고려:** 우리가 타겟으로 하는 잡음/감쇄 환경의 통신 데이터는 긴박한 상황에서의 빠른 발화나, 반대로 전파 상태 악화로 인한 매우 느리고 끊어지는 발화가 공존합니다. 따라서 일반적인 기준보다 하한선을 낮게 설정(약 3.0 CPS)하여 유효한 데이터를 최대한 보존하는 전략을 취합니다.

**[표] CPS 기반 데이터 필터링 가이드라인**

| 구분 | CPS 범위 | 기술적 판단 및 원인 | 조치 사항 |
| :--- | :--- | :--- | :--- |
| **저밀도 구간** | 3.0 미만 | 전사문 누락 또는 VAD가 무음/잡음을 발화로 오인 | 학습 제외 (Noise-only) |
| **정상 구간** | 3.0 ~ 18.0 | 음성과 텍스트의 정합성이 통계적으로 일치함 | **최종 학습 데이터 포함** |
| **고밀도 구간** | 18.0 초과 | 오디오 데이터 손상 또는 텍스트 중복 기록 오류 | 학습 제외 (Mismatch) |

#### 5.4.2 다단계 비정상 청크(Chunk) 자동 필터링 파이프라인

MFA(강제 정렬) 실패나 VAD의 분할 오류로 인해 발생하는 불량 샘플을 제거하기 위해, 아래의 검증 단계를 순차적으로 적용하는 'QA 파이프라인'을 가동합니다.

1.  **물리적 가용성 검증:** JSONL 메타데이터에 기록된 오디오 파일 경로가 실제 내부망 서버 내에 존재하는지, 파일 헤더가 손상되지 않았는지 가장 먼저 확인합니다.
2.  **최소 유의미성 검증:** 전사된 텍스트가 공백을 제외하고 2자 미만인 데이터(예: "네", "아")는 모델이 도메인 특화 어휘를 학습하는 데 기여도가 낮으므로 과감히 제외합니다.
3.  **데이터 규격 적합성 검증:** 6.3절에서 언급한 것처럼 Whisper 모델은 구조적으로 30초 이상의 입력을 처리할 수 없습니다. 따라서 패딩(Padding) 여유를 고려하여 **29.5초를 초과하는 청크**는 학습에서 제외하거나 재분할 프로세스로 환류시킵니다.
4.  **통계적 이상치 제거:** 앞선 5.4.1절의 CPS 기준을 적용하여 음성-텍스트 간의 밀도가 비정상적인 샘플을 최종적으로 걸러냅니다.

#### 5.4.3 검증 결과 리포팅 및 파이프라인 피드백

필터링 과정에서 탈락한 데이터는 단순히 삭제하는 것이 아니라, **'제외 사유별 통계'**를 JSONL 형태의 로그로 남겨야 합니다. 

*   **임계치 관리:** 만약 특정 데이터셋에서 제외율이 20%를 상회한다면, 이는 원천 데이터의 품질 문제보다는 **VAD의 임계값(Threshold)이 너무 낮거나 MFA의 음향 모델이 도메인에 맞지 않아 정렬에 실패**하고 있다는 강력한 신호입니다. 
*   **선순환 구조:** 이 리포트를 바탕으로 앞단의 전처리 설정을 재조정(Iterative Tuning)함으로써, 최종적으로는 고품질의 도메인 특화 데이터셋을 완성하게 됩니다. 필터링을 통과한 정제된 데이터는 이어지는 6장의 규격에 맞춰 최종 학습용 자산으로 변환됩니다.

# 제6장. 학습을 위한 오디오-텍스트 데이터셋 생성 (JSONL)

음성을 적절한 길이(Chunk)로 잘라내고 노이즈를 필터링했다면, 이제 모델에게 정답지(Label)를 쥐어줄 차례입니다. 본 장에서는 통일된 전사 가이드라인을 세우고, 작업자의 피로도를 최소화하는 Human-in-the-Loop(HITL) 파이프라인을 거쳐, 최종 학습 포맷인 JSONL 파일로 만드는 과정을 설명합니다. **ASR 파인튜닝에서 "쓰레기를 넣으면 쓰레기가 나온다(Garbage in, Garbage out)"는 법칙은 데이터 라벨링 품질에 가장 민감하게 반응합니다.** 전사의 일관성과 정확도가 곧 최종 모델의 성능을 결정합니다.

---

### 6.1 도메인 특화 라벨링 가이드라인 수립

여러 명의 라벨러가 참여하더라도 일관성 있는 정답 데이터가 구축되도록 프로젝트 시작 전 명확하고 세부적인 기준을 문서화해야 합니다. 이 가이드라인 문서는 라벨러 교육 자료이자 품질 검수의 기준이 됩니다.

#### 6.1.1 전사 방식 결정: 발음 전사 vs 의미 전사
가장 먼저 결정해야 할 핵심 정책은 **들리는 대로 적을 것인가, 의미에 맞게 고칠 것인가**입니다.

*   **발음 전사 (Phonetic Transcription):** 화자가 실제로 발음한 소리를 그대로 옮깁니다. "갔어요"를 "가써요"로, "됩니다"를 "뎁니다"로 적는 방식입니다. **ASR 엔진의 일반적 목적과 Whisper 아키텍처의 특성상, 발음 전사가 원칙입니다.** 모델이 음향 신호와 텍스트를 정확히 매핑해야 하는데, 발음된 소리와 다른 표준어가 정답으로 있으면 혼란이 생깁니다.
*   **의미 전사 (Semantic Transcription):** 알아듣기 힘든 사투리나 비표준어를 표준어로 교정하여 적습니다. 정보 추출이나 요약 등 후단 NLP 작업이 목적이라면 이 방식이 더 유리합니다.
*   **결론:** 본 프로젝트에서는 **발음 전사**를 원칙으로 하되, 표기법은 한글 맞춤법에 준하여 혼동을 최소화합니다. 프로젝트 목표에 따라 팀 내에서 합의하고 가이드라인에 명시해야 합니다.

#### 6.1.2 숫자, 영문, 도메인 특수 용어의 한글 표기 원칙 (ITN 역변환 고려)
*   **숫자 표기:** "12시"로 들리더라도 아라비아 숫자(12)를 직접 쓰지 않고 발음 그대로 **"열두 시"** 또는 **"십이 시"**와 같이 한글로 풀어 씁니다. 아라비아 숫자나 영문이 섞인 문자열은 Whisper 토크나이저(Tokenizer)가 처리하는 방식이 불안정하여 학습 효율을 떨어뜨립니다. 숫자를 다시 아라비아 숫자로 변환하는 작업(예: "열두" → 12)은 10장에서 다루는 **후처리(ITN: Inverse Text Normalization)** 단계에서 수행합니다.
*   **포네틱 코드 및 영문 약어:** "Alpha", "Bravo", "Oscar Mike" 등의 NATO 포네틱 코드 및 무전 용어는 한글 발음 그대로 **"알파", "브라보", "오스카 마이크"**로 전사합니다.
*   **도메인 특수 용어 화이트리스트(Whitelist):** "OK", "Copy", 무기 체계명 등 프로젝트 고유의 도메인 용어를 미리 목록화하고 해당 용어의 표기 규칙을 명시합니다. 예: "카피" (수신 확인 의미로 사용 시), "아웃" (교신 종료 시). 이 목록은 10장의 맞춤법 교정 후처리에서도 예외 처리 사전으로 활용됩니다.

#### 6.1.3 특수 상황 전사 기준 (불명료 구간, 간투어)
*   **불명료(Unintelligible) 구간:** 노이즈가 극심하여 사람도 내용을 판단하기 불가능한 구간은 임의로 추정하여 전사하지 않고 `[불명]`과 같은 태그로 처리합니다. 잘못된 추측 전사는 오히려 모델을 망칩니다.
*   **간투어 및 비언어적 소음:** "어...", "그...", "음" 등의 간투어(Filler)는 프로젝트 목적에 따라 생략하거나 `[간투]` 태그로 통일합니다. 기침, 숨소리, 웃음 등 비언어 소음은 전사하지 않는 것을 원칙으로 하여 텍스트를 깔끔하게 유지합니다.
*   **묵음 후 재시작 패턴:** 무전 교신에서 흔히 나타나는 "오버", "로저" 등 관용적 교신 종료 멘트는 별도 태그 처리 없이 그대로 전사합니다.

---

### 6.2 초벌 전사 및 인간 작업자(HITL) 교정 파이프라인

수천 시간의 음성을 사람이 처음부터 끝까지 듣고 타이핑하는 것은 막대한 시간과 비용이 요구됩니다. **Human-in-the-Loop (HITL)** 방식은 AI가 먼저 초안을 만들고, 사람은 수정에만 집중하게 하여 생산성을 극적으로 끌어올립니다.

#### 6.2.1 기본 Whisper 모델을 활용한 자동 초벌 전사 (Auto-Transcription)
파인튜닝 전의 기본 Whisper Large 모델로도 초벌 전사를 생성할 수 있습니다. 도메인 특화 용어를 틀리게 전사하더라도 텍스트의 뼈대가 잡혀있으면 **작업자의 타이핑 시간이 3~5배 단축**됩니다.

```bash
# Faster-Whisper를 활용한 배치 초벌 전사 예시
# --output_format srt: 타임스탬프 포함 자막 형식으로 출력
# --language ko: 한국어 지정으로 속도 및 정확도 향상
whisper --model large-v3 --language ko \
        --output_format srt --output_dir ./transcripts/ \
        ./audio_chunks/*.wav
```

*   초벌 전사 결과물은 오디오와 함께 라벨링 도구에 연동됩니다.
*   모델이 **도메인 특화 용어(무전 은어, 무기체계명)에서 실수가 잦을 것**이라는 점을 사전에 작업자들에게 교육합니다.

#### 6.2.2 라벨링 툴(Label Studio 등) 연동 및 검수 워크플로우
*   **도구 선정:** 오픈소스 **Label Studio** 또는 **Audacity** 기반의 커스텀 파이프라인을 구성합니다. Label Studio는 오디오 플레이어, 텍스트 편집기, 승인(Accept)/거절(Reject) 기능을 웹 UI로 제공하여 팀 협업에 최적화되어 있습니다.
*   **검수 워크플로우:** 작업자는 오디오를 들으면서 Whisper가 생성한 초벌 텍스트를 처음부터 다시 타이핑하는 것이 아니라, **틀린 부분(도메인 용어 오류, 발음 차이)만 집중적으로 수정하고 확인(Review & Edit)**합니다.
*   **2단계 검수 (Reviewer):** 생산성이 중요한 1차 라벨러(수정)와 별도로 품질을 최종 확인하는 2차 검수자(Reviewer)를 두는 2-Stage 방식을 채택하면 데이터 품질을 크게 높일 수 있습니다.
*   **라벨러 간 일치도(IAA: Inter-Annotator Agreement):** 동일한 오디오를 여러 명에게 독립적으로 주어 전사 결과의 CER을 비교합니다. IAA가 일정 기준(예: CER 5% 이하) 이상이면 가이드라인을 재정비해야 합니다.

#### 6.2.3 능동 학습(Active Learning)을 통한 데이터 수집 우선순위화
데이터가 모여가는 중간 단계에서는 모든 오디오를 라벨링하는 것보다 **모델이 가장 헷갈려하는 어려운 샘플을 우선적으로 라벨링**하는 것이 효율적입니다.

*   **불확실도 샘플링 (Uncertainty Sampling):** 중간 점검용으로 학습한 모델이 특정 오디오에 대해 여러 전사 후보 중 확률값이 낮거나 헷갈리는(예: 신뢰도 0.4~0.6) 샘플들을 우선적으로 선별합니다.
*   **효과:** 모델의 약점을 집중 공략하는 데이터가 늘어나므로, 같은 라벨링 비용으로 더 빠르고 큰 성능 향상을 기대할 수 있습니다.

---

### 6.3 강제 정렬(Forced Alignment) 및 최종 JSONL 추출 (★핵심)

작업자가 검수한 "완벽한 전사 텍스트"와 10분짜리 "전체 오디오"가 있어도, 이 텍스트가 정확히 오디오의 몇 초부터 몇 초 사이에 발성된 것인지 알지 못하면 학습에 사용할 수 없습니다. **강제 정렬(Forced Alignment)**은 이 텍스트와 오디오를 시간 축 위에 정확히 매핑하는 핵심 기술입니다.

#### 6.3.1 발성 오디오와 텍스트의 타임스탬프 매핑 원리
*   **강제 정렬의 정의:** 음성 신호의 음향적 특징(음소 시퀀스)과 이미 알고 있는 정답 전사 텍스트를 **DTW(Dynamic Time Warping)** 또는 **CTC(Connectionist Temporal Classification)** 기반 알고리즘으로 강제로 매핑하여, 각 단어/음소가 오디오 상에서 시작하는 시간(Start Time)과 끝나는 시간(End Time)을 밀리초(ms) 단위로 추출하는 기술입니다.
*   **필요성:** 전체 녹음 파일을 문장 단위로 정확히 분절(Crop)해야만, Whisper 모델의 30초 입력 길이에 최적화된 학습 샘플을 생성할 수 있습니다.

#### 6.3.2 Whisper 어텐션 기반 기본 정렬 (`whisper-timestamped`)
과거부터 사용되어 온 기본적인 타임스탬프 추출 방식으로, Whisper 모델이 텍스트를 생성할 때 내부적으로 사용하는 **교차 어텐션(Cross-Attention) 가중치**를 분석하여 단어의 시간을 역산합니다.

*   **장점 (폐쇄망 관점):** 별도의 정렬용 음향 모델(Acoustic Model)이나 음소 사전을 추가로 반입할 필요가 없습니다. 텍스트 전사에 사용한 Whisper 모델 하나만으로 작동하므로 인프라 반입 및 파이프라인 구성이 매우 단순합니다.
*   **한계점 (Fuzzy Alignment):** 어텐션 가중치 자체가 시간 축에 대해 칼같이 떨어지는 선형적 구조가 아니기 때문에, 단어의 시작과 종료 시간이 다소 뭉개지거나 부정확하게 추출될 수 있습니다. 밀리초 단위의 정밀한 분절이 필요한 고품질 학습 데이터셋 구축에는 다소 부족함이 있습니다.

#### 6.3.3 WhisperX 기반 고정밀 강제 정렬
기본 정렬 방식의 '시간 뭉개짐' 문제를 극복하기 위해 텍스트 전사와 시간 정렬을 분리한 2단계(Two-stage) 파이프라인 구조입니다. 현재 ASR 데이터 전처리 분야의 실무 표준으로 널리 사용됩니다.

*   **동작 원리:**
    1.  **초벌 전사:** 내장된 Faster-Whisper 백엔드를 사용하여 원본 오디오에서 텍스트만 초고속으로 추출합니다.
    2.  **음향 모델 기반 강제 정렬:** 추출된 텍스트와 원본 오디오를 Wav2vec 2.0과 같은 '별도의 정렬 전용 음향 모델'에 통과시켜 CTC 기반으로 완벽하게 시간을 매핑합니다.
*   **핵심 장점:** 전통적인 MFA 방식에 버금가는 **밀리초 단위의 극도로 정밀한 타임스탬프**를 제공합니다. 또한 VAD(Voice Activity Detection) 필터가 기본 탑재되어 있어, 발성이 없는 무음 구간의 환각(Hallucination) 텍스트를 강제 정렬 과정에서 효과적으로 제거합니다.
*   **폐쇄망 환경 고려사항:** 이 방식을 내부망에서 오프라인으로 구동하기 위해서는 텍스트 전사용 Whisper 가중치 외에도, **정렬(Alignment)에 사용되는 한국어 특화 음향 모델 가중치(예: wav2vec2-large-xlsr-53-korean)**를 반드시 외부에서 추가로 다운로드하여 반입해야 합니다.

#### 6.3.4 데이터 분절 및 최종 JSONL 데이터셋 구축
고정밀 강제 정렬이 완료되어 각 발화 구간의 시작/종료 시각이 확정되면, 이를 바탕으로 실제 오디오 파일을 자르고 AI 모델 학습 규격인 JSONL 형태로 가공합니다.

*   **오디오 물리적 분절 (Cropping):** 정렬된 타임스탬프를 기준으로 원본 오디오 파일에서 해당 구간의 소리 데이터만 추출하여 30초 이내의 짧은 단위 파일(Chunk)로 개별 저장합니다. 이때 샘플링 레이트는 Whisper의 규격인 16kHz로 일괄 유지합니다.
*   **메타데이터 통합 (JSONL 작성):** 분절된 각 오디오 청크마다 파일이 저장된 로컬 경로(`audio_path`), 검수 및 정렬이 완료된 정답 텍스트(`sentence`), 그리고 해당 구간의 오디오 길이(`duration`)를 하나의 딕셔너리로 묶어 JSONL 파일에 한 줄씩 기록합니다.

**데이터셋 분할 (Split) 전략:**
최종 생성된 JSONL 데이터는 모델이 학습한 내용과 처음 보는 데이터를 구분할 수 있도록 다음과 같이 3가지 세트로 분할하여 사용합니다.

| 분할 | 비율 (권장) | 용도 |
| :--- | :---: | :--- |
| **Train (학습)** | 80% | 모델의 파라미터(가중치)를 직접 업데이트하는 데 사용 |
| **Validation (검증)** | 10% | 학습 중 에폭(Epoch)마다 과적합(Overfitting) 여부 감시 |
| **Test (평가)** | 10% | 학습이 완전히 종료된 후, 최종 모델의 성능을 객관적으로 측정 |

**최종 JSONL 구조 예시:**
```json
{"audio": {"path": "./output/chunk_00001.wav"}, "sentence": "통신 상태 양호합니다.", "duration": 3.2}
{"audio": {"path": "./output/chunk_00002.wav"}, "sentence": "현재 좌표는 알파 브라보입니다.", "duration": 4.5}
{"audio": {"path": "./output/chunk_00003.wav"}, "sentence": "[불명] 이상", "duration": 2.1}
```

*   **실무 중요 지침:** Train 세트와 Test 세트에 동일한 작업자의 음성(화자)이나, 동일한 날짜에 녹음된 완전히 똑같은 배경 소음이 섞여 들어가면 평가 결과가 과장될 수 있습니다. 이를 방지하기 위해 데이터를 무작위로 섞지 말고, 가급적 **녹음 세션(날짜) 또는 화자 단위로 그룹화하여 분할**하는 것이 신뢰도 높은 평가 방법입니다.

오디오 파일을 물리적으로 자르지 않고, 원본 파일의 경로와 타임스탬프 정보만 JSONL에 기록하여 학습 시점에 필요한 구간만 읽어 들이는 방식을 **'온더플라이(On-the-fly) 데이터 로딩'** 또는 **'메타데이터 기반 스트리밍'** 방식이라고 합니다.

이 방식은 수백 기가의 오디오 데이터를 중복 저장하지 않아도 되므로 저장 공간 효율성이 극도로 높습니다. 해당 방식의 구조와 파이프라인 구성 요소를 정리해 드립니다.

---

### 6.4. [심화] 메타데이터 기반 온더플라이 데이터셋 구성

물리적 분절 없이 원본 파일과 타임스탬프 정보만으로 학습 파이프라인을 구축하면, 데이터 관리의 복잡성을 획기적으로 줄일 수 있습니다.

#### 6.4.1 메타데이터 기반 JSONL 구조
이 방식에서는 `audio` 필드에 파일 경로를 넣고, 별도의 `start_time`과 `end_time` 필드를 추가하여 구간을 명시합니다.

**JSONL 예시:**
```json
{"audio": "/path/to/original_10min.wav", "start": 0.0, "end": 5.42, "sentence": "첫 번째 문장입니다."}
{"audio": "/path/to/original_10min.wav", "start": 5.42, "end": 12.1, "sentence": "두 번째 발화 내용입니다."}
{"audio": "/path/to/another_file.wav", "start": 0.0, "end": 8.5, "sentence": "다른 파일의 시작 부분입니다."}
```

#### 6.4.2 데이터 로딩 메커니즘 (Internal Logic)
학습 파이프라인(DataLoader)이 가동될 때, 시스템은 다음과 같은 단계를 거쳐 데이터를 모델에 전달합니다.

1.  **메타데이터 스캔:** JSONL을 한 줄씩 읽어 파일 경로와 시작/종료 시각을 파악합니다.
2.  **부분 로딩 (Partial Loading):** `librosa.load`나 `torchaudio.load`의 `frame_offset`과 `num_frames` 옵션을 사용하여, 전체 파일 중 **해당하는 구간의 바이너리 데이터만 메모리에 적재**합니다.
3.  **샘플링 및 변환:** 로드된 짧은 파형(Waveform)을 16kHz로 리샘플링하고 Whisper 인풋인 로그 멜 스펙트로그램으로 변환하여 모델에 입력합니다.

#### 6.4.3 이 방식의 장점과 고려사항

| 구분 | 내용 |
| :--- | :--- |
| **저장 공간 절약** | 400시간 분량의 원본을 유지하면서 수만 개의 조각 파일을 따로 생성할 필요가 없어 디스크 I/O 및 용량이 대폭 절감됩니다. |
| **유연한 데이터 정정** | 전사 내용이나 타임스탬프에 오류가 발견되었을 때, 오디오 파일을 다시 자를 필요 없이 **JSONL의 숫자만 수정**하면 즉시 반영됩니다. |
| **학습 부하 (Overhead)** | 학습 중 매번 대형 파일의 특정 시점을 찾아 읽어야 하므로, CPU와 디스크 읽기 속도에 약간의 부하가 발생합니다. (단, H100 환경의 고성능 서버라면 무시할 수 있는 수준입니다.) |
| **랜덤 액세스 성능** | 데이터가 압축되지 않은 `.wav` 형식일 때 가장 빠르게 작동합니다. 압축된 형식은 특정 지점을 찾기 위해 디코딩 과정이 추가될 수 있습니다. |

#### 6.4.4 실무 적용 팁
HuggingFace의 `datasets` 라이브러리를 사용할 경우, `IterableDataset` 기능을 활용하면 수백 기가의 원본 파일을 한꺼번에 메모리에 올리지 않고 필요한 순간에만 네트워크나 디스크에서 가져오는 **스트리밍 학습**이 가능합니다. 이는 내부망 환경에서 대규모 데이터를 효율적으로 처리하는 가장 진보된 방식입니다.

### 어떤 방식을 선택해야 할까?
*   **물리적 분절 방식 (6.3):** 데이터셋의 크기가 작고(예: 50시간 이하), 직관적인 데이터 관리를 원할 때 추천
*   **온더플라이 방식 (6.4):** 대규모 데이터를 다루며, 인프라 자원(H100)이 충분하여 학습 효율을 극대화하고 싶을 때 추천

# Part 3. 모델 튜닝 및 학습 파이프라인 구축 (학습/튜닝 편)

## 제7장. LoRA (Low-Rank Adaptation) 파인튜닝 기법

완성된 데이터셋을 가지고 15억 개의 파라미터를 가진 Whisper Large 모델의 모든 가중치를 학습하는 것은 막대한 GPU 자원과 시간이 소모됩니다. 본 장에서는 이 문제를 해결하기 위해 채택한 파라미터 효율적 미세 조정(PEFT) 방법론인 **LoRA(Low-Rank Adaptation)**의 수학적 원리, 하이퍼파라미터 설정 전략, 그리고 실전에서의 활용법까지 상세히 설명합니다.

---

### 7.1 PEFT (Parameter-Efficient Fine-Tuning) 소개

#### 7.1.1 Full Fine-Tuning의 문제점 (자원 및 재앙적 망각)
과거에는 모델을 특정 도메인에 맞추기 위해 전체 네트워크의 가중치를 새로 업데이트하는 **Full Fine-Tuning** 방식을 사용했습니다. 그러나 거대 모델(LLM/LSM) 시대가 도래하면서 다음의 치명적인 문제점들이 부각됩니다.

*   **막대한 VRAM 요구량:** Whisper Large 모델(1.5B 파라미터)을 Full Fine-Tuning하려면, 모델 파라미터 자체 외에도 역전파를 위한 **그래디언트(Gradient)**, 그리고 Adam 옵티마이저의 **1차 모멘트(moment1)**, **2차 모멘트(moment2)** 상태값까지 VRAM에 올라가야 합니다. bf16 정밀도 기준으로 파라미터 1개당 약 12~16바이트가 소요되어 전체적으로 최소 **80GB VRAM 이상**이 필요합니다. H100(80GB HBM3)을 단독으로 사용해도 배치 사이즈를 줄여야 겨우 학습이 가능하며, 대규모 배치(Batch=64 이상)를 위해서는 **NVLink 연결 H100 2장 이상의 멀티 GPU**가 권장됩니다.
*   **재앙적 망각(Catastrophic Forgetting):** 군사 도메인 데이터(예: 무전 교신)만으로 전체 모델 가중치를 새로 쓰면, 모델이 기존에 68만 시간 학습으로 쌓아온 범용적 언어 지식(일상어, 영어, 문맥 추론 능력)을 모두 망각합니다. 이 현상은 새 지식을 덮어쓰는 과정에서 기존 지식을 담당하던 신경망 연결 가중치가 파괴되는 신경과학적 현상으로, 파인튜닝 후 범용적 인식 성능이 치명적으로 퇴보합니다.
*   **학습 시간:** H100(80GB HBM3) GPU 단독으로도 Whisper Large의 Full Fine-Tuning은 수백 시간 이상 소요될 수 있습니다. (A100 대비 약 2~3배 빠르나, 여전히 방대한 비용)

#### 7.1.2 PEFT의 종류 비교 (Adapter, Prefix Tuning, LoRA)
위의 문제를 해결하기 위해 **기존 모델의 가중치는 동결(Freeze)하고, 아주 적은 수의 새로운 파라미터만 추가하거나 수정하는 PEFT** 방법론이 등장했습니다.

| 기법 | 작동 방식 | 추론 시 오버헤드 | 특징 |
| :--- | :--- | :---: | :--- |
| **Adapter Tuning** | 각 Transformer 레이어 사이에 작은 병목(Bottleneck) 모듈 삽입 | 있음 | 구조 변경이 필요, 순차 처리로 약간의 지연 발생 |
| **Prefix Tuning** | 입력 시퀀스 앞에 학습 가능한 가상 토큰(Soft Prompt) 추가 | 거의 없음 | 자연어 생성(NLG)에 특히 강함 |
| **LoRA** | 기존 가중치 행렬 옆에 저랭크 행렬 쌍을 병렬로 추가 | **없음** | 추론 시 기존 가중치에 병합(Merge) 가능, **현재 업계 표준** |

LoRA는 성능 손실이 없고, 추론 시 구조 변경 없이 기존 가중치에 어댑터를 병합(Merge)할 수 있어 **현재 실무 파인튜닝의 사실상 표준(De facto Standard)**으로 자리 잡았습니다.

---

### 7.2 LoRA의 수학적 원리와 구조

수식이 익숙하지 않은 분들도 이해할 수 있도록, 직관적인 비유를 먼저 제시하고 기술 원리를 부연합니다.

#### 7.2.1 핵심 아이디어: "책에 빽빽하게 필기하지 말고, 얇은 메모지에 핵심만 적어라"

Whisper Large 모델을 상상해 보겠습니다. 이 모델은 수년간 68만 시간의 음성을 공부한 뒤, **1,550,000,000개의 지식 조각(파라미터)**이 빽빽하게 적힌 두꺼운 교과서와 같습니다.

**Full Fine-Tuning의 문제:**
군사 무전 전문 용어를 추가로 가르치려면, 이 두꺼운 교과서의 모든 페이지를 처음부터 끝까지 다시 새로 써야 합니다. 시간도 엄청나게 걸리고, 기존에 잘 알던 내용(일상 대화, 영어 인식)이 지워질 위험도 있습니다.

**LoRA의 해결책:**
교과서(기존 모델)는 그대로 두고, 얇은 **포스트잇(LoRA 어댑터)** 하나를 교과서 옆에 붙입니다. 이 포스트잇에는 "이 소리가 들리면 '알파 브라보'로 읽어라" 같은 **추가 규칙만** 간단히 적어 둡니다. 시험 볼 때 모델은 교과서와 포스트잇을 동시에 참고하여 답을 냅니다.

> 📌 **한 줄 요약:** 기존 지식(교과서)을 건드리지 않고, 추가 지식(포스트잇)만 따로 훈련한다.

#### 7.2.2 "왜 포스트잇이 그렇게 작을 수 있는가?" — 저랭크 분해의 비밀

포스트잇(LoRA 어댑터)이 작은 이유는 **저랭크 행렬 분해(Low-Rank Decomposition)**라는 수학적 트릭 덕분입니다.

**비유로 이해하기:**
서울에서 부산으로 가는 길을 새로 만들어야 한다고 해봅시다.
- **Full Fine-Tuning 방식:** 서울에서 부산까지 이어지는 직통 고속도로 전체를 새로 건설합니다. (엄청난 비용과 시간)
- **LoRA 방식:** 서울 ↔ 대구 구간(짧은 구간 A)과 대구 ↔ 부산 구간(짧은 구간 B), 두 개의 짧은 우회 도로만 새로 놓습니다. (비용과 시간이 훨씬 적게 듦) 결과적으로 두 구간을 이으면 서울~부산을 연결하는 효과를 냅니다.

**수식으로 표현하면 이렇습니다:**
```
추가 학습할 변화량(ΔW) = 작은 행렬 B × 작은 행렬 A

  ┌──────────────┐        ┌───────────────────────────────┐
  │  행렬 A      │        │ 역할: "어떤 정보에 집중할까?" │
  │  (10000 × 8) │   ×    │ 랭크(r) = 8인 좁은 병목       │
  └──────────────┘        └───────────────────────────────┘
         ×
  ┌──────────────┐        ┌───────────────────────────────┐
  │  행렬 B      │        │ 역할: "집중한 정보를 어떻게   │
  │  (8 × 10000) │        │        펼쳐서 출력할까?"      │
  └──────────────┘        └───────────────────────────────┘
```

**파라미터 수 비교 (한눈에):**

| 방식 | 학습해야 할 조각 수 | 비유 |
| :--- | :---: | :--- |
| Full Fine-Tuning (10000×10000) | **1억 개** | 백과사전 전체 다시 쓰기 |
| LoRA (r=8) | **16만 개** | 포스트잇 한 장 (625배 절약!) |

*   **B를 처음에 '0'으로 설정하는 이유:** 학습을 시작할 때 포스트잇이 아무것도 적혀 있지 않은 빈 종이 상태(`ΔW = 0`)여야, 기존 교과서의 내용을 그대로 유지한 채 안정적으로 새 규칙을 추가 학습할 수 있습니다.

#### 7.2.3 "포스트잇의 영향력 조절" — 스케일링 팩터(α)

포스트잇(LoRA 어댑터)의 목소리 크기를 조절하는 볼륨 다이얼이 바로 **알파(α)**입니다.

```
최종 출력 = [기존 교과서 답] + (α/r) × [포스트잇 추가 답]
```

*   **α가 크면:** 포스트잇 내용을 더 강하게 반영합니다. 도메인 특화 용어 인식이 강해지지만, 너무 크면 오히려 불안정해질 수 있습니다.
*   **α가 작으면:** 포스트잇을 참고는 하되 기존 교과서를 더 많이 따릅니다.
*   **실무 권장값:** 통상 `α = r의 2배` (예: r=16이면 α=32)로 설정합니다.

#### 7.2.4 "포스트잇을 어디에 붙이나?" — 학습 대상 레이어 선택

Whisper 모델의 내부는 여러 방에 나뉜 공장과 같습니다. 각 방은 서로 다른 작업을 합니다. 포스트잇(LoRA 어댑터)은 이 중에서 **"이 소리가 어떤 맥락에서 어떤 의미인지 파악하는 방(Self-Attention 레이어)"** 에 주로 붙입니다.

*   `q_proj`, `k_proj`: "지금 이 소리에서 어떤 단서를 찾아야 하는가?"를 결정하는 방
*   `v_proj`, `out_proj`: "찾아낸 단서를 어떻게 텍스트로 바꿀까?"를 결정하는 방

이 4개의 핵심 방에 포스트잇을 붙이면 모델의 **전체 15억 개 파라미터 중 단 0.5% 미만만 업데이트**하면서도, 전체를 새로 쓰는 것에 맞먹는 도메인 적응 효과를 얻을 수 있습니다.

### 7.3 Whisper-LoRA 결합 시 핵심 하이퍼파라미터 설정

#### 7.3.1 랭크(r)와 알파(α) 설정 전략
실무에서의 권장 설정 값과 그 근거입니다.

| 시나리오 | 권장 r | 권장 α | 학습 가능 파라미터 수(Large 기준) |
| :--- | :---: | :---: | :--- |
| **빠른 검증 / 데이터 부족** | 4 ~ 8 | 8 ~ 16 | 약 1~3M 개 (전체의 0.07~0.2%) |
| **균형 잡힌 실무 표준** | **16 ~ 32** | **32 ~ 64** | 약 6~12M 개 (전체의 0.4~0.8%) |
| **고성능 / 데이터 충분** | 64 ~ 128 | 64 ~ 128 | 약 25~50M 개 (전체의 1.5~3.5%) |

*   **랭크가 높다고 항상 좋은 것은 아닙니다.** 데이터가 부족한 상태에서 r을 지나치게 높이면 어댑터가 훈련 데이터에 과적합(Overfitting)되어 오히려 성능이 나빠질 수 있습니다. 데이터 양에 비례하여 r을 점진적으로 높이는 전략이 안전합니다.

#### 7.3.2 LoRA 적용 대상 모듈(target_modules) 선택
Whisper LoRA 설정에서 `target_modules`는 어댑터를 삽입할 레이어 이름의 목록입니다.

```python
# HuggingFace PEFT 라이브러리 설정 예시
from peft import LoraConfig, get_peft_model

lora_config = LoraConfig(
    r=16,                       # 랭크 (r): 표현력과 파라미터 수의 트레이드오프
    lora_alpha=32,              # 스케일링 상수 (α): 통상 r의 2배 설정
    lora_dropout=0.05,          # 과적합 방지를 위한 드롭아웃
    bias="none",                # 바이어스 항은 학습하지 않음
    target_modules=[            # 어댑터를 삽입할 레이어 지정
        "q_proj",               # Self-Attention: Query 투영 행렬
        "k_proj",               # Self-Attention: Key 투영 행렬
        "v_proj",               # Self-Attention: Value 투영 행렬
        "out_proj",             # Self-Attention: 출력 투영 행렬
        # 디코더 Cross-Attention에도 적용 시 추가:
        # "encoder_attn.q_proj", "encoder_attn.v_proj"
    ],
    task_type="SEQ2SEQ_LM",     # Whisper는 인코더-디코더(Seq2Seq) 구조
)

model = get_peft_model(whisper_model, lora_config)
model.print_trainable_parameters()
# 출력 예: trainable params: 7,864,320 || all params: 1,551,702,016 || trainable%: 0.507
```

#### 7.3.3 VRAM 소모량 비교

| 학습 방식 | Whisper Large | 필요 VRAM (bf16) | H100 기준 학습 속도 |
| :--- | :---: | :---: | :--- |
| **Full Fine-Tuning** | 1.5B 파라미터 | ~80 GB+ (H100 × 1~2장, 소배치) | 매우 느림, 실용 불가 |
| **LoRA (r=16), batch=64** | 1.5B + 8M | **~40~50 GB (H100 × 1장)** | **약 1~3시간/10시간 데이터** |
| **LoRA (r=32), batch=128** | 1.5B + 16M | ~60~70 GB (H100 × 1장) | 약 2~5시간/10시간 데이터 |

### 7.4 Whisper와 LoRA의 결합 실무 전략

#### 7.4.1 특정 도메인 지식 주입 및 범용성 유지 효과
LoRA 방식을 사용하면 Full Fine-Tuning의 가장 큰 부작용인 **재앙적 망각**을 원천적으로 차단합니다.

*   본래 모델(Base Model)이 가진 68만 시간의 범용 음향/언어 지식은 동결된 `W₀` 안에 안전하게 보존됩니다.
*   작은 LoRA 어댑터(`ΔW = B×A`)는 도메인 특화 '필터' 역할만 담당합니다. 해상/항공/군사 환경 특유의 노이즈 패턴과 전문 OOV(Out-of-Vocabulary) 용어를 정확히 캐치하는 추가 지식이 베이스 모델 위에 얹히는 구조입니다.
*   결과적으로, **"일반 대화는 Whisper 수준으로 잘 알아듣되, 군사 무전 교신 전문 용어도 정확히 인식하는"** 최이상적인 ASR 시스템이 완성됩니다.

#### 7.4.2 추론 시 LoRA 가중치 병합(Merge) 전략
학습이 완료된 후, LoRA 어댑터(`B×A`)를 베이스 가중치(`W₀`)에 **수학적으로 병합(Merge)**하면 추론(Inference) 시의 추가 연산 오버헤드를 없앨 수 있습니다.

```python
# 학습 완료 후 LoRA 어댑터를 베이스 모델에 병합
from peft import PeftModel

# 1. 베이스 모델 + LoRA 어댑터를 함께 로드
model = PeftModel.from_pretrained(base_whisper_model, "./lora_adapter_dir/")

# 2. 어댑터를 베이스 가중치에 수학적으로 합산 (W = W₀ + α/r * BA)
merged_model = model.merge_and_unload()

# 3. 병합된 순수 Whisper 모델처럼 저장 -> 추론 시 별도 PEFT 라이브러리 불필요
merged_model.save_pretrained("./merged_whisper_ko_military/")
```

*   병합 후에는 일반 Whisper 모델처럼 배포 및 CTranslate2/ctranslate2 변환이 가능해집니다.
*   반면, 병합 전 어댑터 상태로 유지하면 어댑터를 실시간으로 교체하는 **멀티 도메인 스위칭** 전략이 가능합니다.

#### 7.4.3 다중 LoRA 어댑터 스위칭을 통한 멀티 도메인 지원
학습된 LoRA 가중치는 고작 **수십 MB~수백 MB** 단위의 가벼운 파일로 저장됩니다. 이 특성을 이용해 단일 베이스 모델로 여러 도메인을 동시에 지원하는 유연한 서비스 아키텍처를 구성할 수 있습니다.

```
[공유 베이스 모델: Whisper Large-v3]  ← GPU 메모리에 1회 로드
         │
         ├─── [어댑터 A] 해양_통신환경_LoRA.bin  (50MB)
         ├─── [어댑터 B] 지상_이동환경_LoRA.bin  (50MB)
         └─── [어댑터 C] 항공_통신환경_LoRA.bin  (50MB)
```

*   **동작 방식:** 전사 요청이 들어올 때 해당 채널의 도메인에 맞는 어댑터 파일을 실시간으로 선택하여 베이스 모델에 적용합니다.
*   **확장성:** 신규 도메인 추가 시 베이스 모델 재배포 없이 어댑터 파일 하나만 추가하면 됩니다.
*   **서비스 도입 시 고려사항:** 서비스 요청량이 폭발적으로 많아 여러 어댑터를 동시에 병렬 처리해야 한다면, `punica` 또는 `LoRAX` 같은 다중 LoRA 배치 추론(Multi-LoRA Serving) 라이브러리 도입을 검토할 수 있습니다.

### 7.5 QLoRA: 초저VRAM 환경을 위한 양자화 결합 기법

24GB VRAM 이하의 소규모 GPU 환경에서도 Whisper Large 파인튜닝이 가능하도록, LoRA에 **양자화(Quantization)**를 결합한 **QLoRA(Quantized LoRA)** 기법이 있습니다.

#### 7.5.1 QLoRA의 핵심 원리 (NF4 양자화 + LoRA)
*   **베이스 모델 4-bit 양자화 적재:** 학습 시 베이스 모델(`W₀`)을 fp16이 아닌 **4-bit NF4(NormalFloat4)** 형식으로 압축하여 VRAM에 올립니다. 이로써 베이스 모델이 차지하는 메모리가 fp16 대비 약 **4분의 1**로 줄어듭니다.
*   **LoRA 어댑터는 fp16으로 학습:** `W₀`는 4-bit으로 얼려두고, LoRA 어댑터 행렬(`A`, `B`)는 기존과 동일하게 fp16 정밀도로 업데이트합니다. 역전파 시에는 4-bit 베이스 모델을 임시로 fp16으로 역양자화(Dequantization)합니다.

#### 7.5.2 QLoRA vs LoRA 비교

| 항목 | LoRA (bf16 베이스) | QLoRA (4bit 베이스) |
| :--- | :---: | :---: |
| **VRAM 요구량** (batch=64, H100 기준) | ~40~50 GB | **~20~30 GB** |
| **학습 속도** | 기준 (H100 TensorCore 풀 활용) | 10~25% 느림 (역양자화 오버헤드) |
| **최종 성능** | 기준 | 거의 동일 (미미한 차이) |
| **권장 GPU** | **H100 (80GB HBM3)** | H100에서도 배치 2배 증가 가능 |

*   H100(80GB) 환경에서는 일반 LoRA를 bf16으로 학습해도 VRAM이 충분하므로 **QLoRA보다 일반 LoRA 사용을 권장**합니다.
*   QLoRA는 VRAM이 부족한 저사양 환경을 위한 기법이며, H100 환경에서는 연산 오버헤드 없이 높은 배치 사이즈로 속도를 극대화하는 것이 유리합니다.

### 7.6 LoRA 성능 향상을 위한 학습 데이터 규모 가이드

LoRA는 적은 데이터로도 효과를 발휘하는 것이 장점이지만, **"어느 정도의 데이터가 있어야 유의미한 성능 향상이 나타나는가?"**는 실무에서 매우 중요한 설계 기준입니다. 아래에서는 Whisper ASR 파인튜닝을 기준으로 한 실증적 권장 데이터 규모를 제시합니다.

#### 7.6.1 Whisper LoRA 파인튜닝 단계별 권장 데이터 규모

데이터셋의 규모가 클수록 성능은 향상되지만, **데이터와 성능의 관계는 로그 스케일(Log-Scale)**에 가깝습니다. 즉, 처음 1시간에서 10시간으로 늘릴 때 가장 극적인 향상이 나타나고, 100시간 이상부터는 수확 체감(Diminishing Returns)이 발생합니다.

| 단계 | 데이터 규모 | 기대 효과 | 권장 시나리오 |
| :--- | :---: | :--- | :--- |
| **최소 기준** | **1 ~ 5시간** | 베이스 모델 대비 OOV(전문 용어) 오류 감소, 체감 가능한 개선 시작 | 개념 검증(PoC), 사전 가능성 확인 |
| **권장 기준** | **10 ~ 30시간** | WER 기준 **30~50% 오류 감소**, 실무 적용 가능 수준 | 실제 운용 시스템 1차 배포 |
| **고성능 기준** | **50 ~ 100시간** | WER 기준 **50~70% 이상 오류 감소**, 인간 수준에 근접 | 핵심 시스템, 높은 정확도 요구 환경 |
| **포화 구간** | **100시간 이상** | 성능 향상이 점진적으로 둔화됨 | 데이터 추가보다 데이터 **품질** 개선이 더 효과적 |

*   **핵심 원칙: 10~30시간이 실무의 황금 구간입니다.** H100(80GB) 단일 GPU 기준으로 이 규모의 데이터는 **수 시간(약 2~6시간) 이내에 학습이 완료**되며, 도메인 적응 효과가 명확히 나타납니다.

#### 7.6.2 데이터 규모별 성능 향상 시뮬레이션 (한국어 특수 도메인 기준)
아래 수치는 Whisper Large-v3 베이스 기준의 도메인 내 CER 개선 경향을 나타낸 참고 수치입니다.

| 데이터 규모 | 파인튜닝 전 CER(추정) | 파인튜닝 후 CER(추정) | 개선율 |
| :---: | :---: | :---: | :---: |
| 0 (베이스 모델) | ~30~50% | - | - |
| 1시간 | - | ~20~30% | ↓ 약 30~40% |
| 10시간 | - | ~10~15% | ↓ 약 60~70% |
| 30시간 | - | ~5~10% | ↓ 약 75~85% |
| 100시간 | - | ~3~7% | ↓ 약 85~90% |

> **주의:** 위 수치는 도메인 내 Test 데이터 기준의 이상적 참고 범위입니다. 실제 성능은 **녹음 환경의 잡음 수준, 전문 용어 빈도, 데이터 품질**에 따라 크게 달라집니다.

#### 7.6.3 데이터 품질 vs 데이터 양: 무엇이 더 중요한가?
실무 경험상 **"100시간의 낮은 품질 데이터"보다 "10시간의 완벽한 품질 데이터"가 훨씬 더 뛰어난 파인튜닝 결과를 냅니다.** 다음 품질 기준을 반드시 충족해야 합니다.

*   ✅ **라벨 정확도:** 6장에서 구축한 HITL 검수를 통해 전사 CER이 2% 미만인 데이터만 학습에 사용합니다. 오전사 데이터는 모델에 오답을 학습시켜 성능을 저하시킵니다.
*   ✅ **오디오 다양성:** 동일한 화자, 동일한 환경의 데이터만 반복하면 모델이 특정 목소리에 과적합(Overfitting)됩니다. **최소 10인 이상의 다양한 화자**, 다양한 녹음 환경(실내/야외/통신망 품질 차이)이 골고루 섞여야 합니다.
*   ✅ **도메인 커버리지:** 학습 데이터에 없는 용어는 테스트에서도 실패합니다. 운용 환경에서 실제로 사용되는 전문 용어, 교신 패턴, 숫자/부호 읽기 방식이 모두 포함되어야 합니다.
*   ✅ **노이즈 분포:** 운용 환경의 잡음 패턴이 학습 데이터에도 적절히 포함되어야 합니다. 완벽히 깨끗한 음성만 학습하면 실제 잡음 환경에서 성능이 급락합니다.

#### 7.6.4 데이터 부족 시 보완 전략
운용 데이터가 충분하지 않을 때 사용할 수 있는 보완책입니다.

*   **데이터 증강 (Data Augmentation):** 기존 오디오에 다양한 노이즈(엔진음, 백색잡음 등)를 인위적으로 합성하거나, 속도·피치를 변환(SpecAugment 등)하여 데이터를 수 배로 늘립니다. 단, 과도한 증강은 오히려 품질을 떨어뜨릴 수 있으므로 적절한 범위에서 적용합니다.
*   **지속 학습 (Continual Learning):** 초기 소량 데이터로 1차 파인튜닝을 수행하고, 운용 중에 지속적으로 데이터를 수집·검수하여 주기적으로 재학습(Incremental Training)하는 전략입니다.
*   **유사 도메인 데이터 활용:** 일반 한국어 공개 데이터셋(AI Hub 등)과 도메인 특화 데이터를 혼합하되, 도메인 특화 데이터의 비중을 높게(예: 7:3) 유지하여 학습합니다.
*   **합성 데이터 (Synthetic Data):** TTS(Text-to-Speech) 시스템으로 도메인 특화 텍스트 스크립트를 음성으로 합성하여 학습 데이터를 확보하는 방법입니다. 발음 패턴과 용어 학습에는 효과적이지만, 실제 통신 잡음 패턴을 학습시키기는 어렵습니다.

#### 7.6.5 복합 난이도 시나리오 분석

앞서 제시한 "10~30시간 황금 구간"은 **표준적인 단일 도메인(깨끗하거나 단순한 잡음, 한국어 단일 억양, 균일한 코덱)** 환경을 전제로 한 기준입니다. 하지만 다음 조건들이 **동시에** 복합적으로 존재하는 시나리오는 근본적으로 다른 접근이 필요합니다.

**본 시스템의 복합 난이도 요인 분석:**

| 난이도 요인 | 상세 내용 | 데이터 요구량 가중치 |
| :--- | :--- | :---: |
| **잡음 다양성** | 육상(엔진/폭음), 해상(파도/바람), 항공(로터음/기압) | ×1.5 ~ 2.0 |
| **코덱 특성 차이** | 아날로그 VHF, 디지털 P25, 위성 MELP 등 복수 코덱 혼재 | ×1.5 ~ 2.0 |
| **특수 용어 밀도** | 포네틱 코드, 무기체계명, 전술 약어 등 OOV 빈도 높음 | ×1.5 |
| **방언 및 외국어** | 지역 억양 차이, 영어/영문 혼재 교신 | ×1.5 |
| **심각한 신호 감쇄** | SNR 0dB 이하 구간 빈번, 페이딩, 끊김 | ×2.0 |

각 가중치 요인이 **독립적이라면** 단순 곱이지만, 실제로는 **상호 복합 효과**로 인해 요구 데이터 규모가 지수적으로 증가합니다. **결론적으로 본 시나리오는 표준 권장량(10~30시간)의 약 10~20배인 최소 200~500시간 이상의 실제 도메인 데이터가 요구됩니다.**

#### 7.6.6 최대 성능을 위한 데이터셋 규모 산정

단일 LoRA 모델 하나로 모든 복합 난이도를 커버하려는 접근은 비효율적입니다. 아래의 **계층적 전략(Layered Strategy)**을 권장합니다.

**[권장 전략: 도메인 계층 분리 + 앙상블]**

```
[1단계] 공통 베이스 파인튜닝 (Common Base LoRA)
  ├─ 목표: 코덱 왜곡 및 다양한 환경 잡음 패턴 일반화, 도메인 공통 어휘 학습
  ├─ 데이터: 다양한 환경(도심, 해양, 항공 등) 혼합 정제 데이터
  └─ 권장 규모: 200 ~ 300시간

[2단계] 도메인 특화 파인튜닝 (Domain-Specific LoRA)
  ├─ [지상 환경 LoRA] 차량 주행음 및 지상 환경 특성 반영
  │     └─ 권장 규모: 50 ~ 100시간
  ├─ [해양 환경 LoRA] 선박/파도/바람 등 해양 환경 잡음 반영
  │     └─ 권장 규모: 50 ~ 100시간
  └─ [항공 환경 LoRA] 엔진 및 기체 소음, 고도 환경 특성 반영
        └─ 권장 규모: 50 ~ 100시간
```

**전체 데이터셋 목표 규모 산정:**

| 구분 | 도메인 | 목표 규모 | 비고 |
| :--- | :--- | :---: | :--- |
| 공통 베이스 LoRA | 전 도메인 혼합 | **200~300시간** | 코덱·잡음 일반화, OOV 어휘 주입 |
| 지상 특화 LoRA | 지상 | 50~100시간 | 차량 소음, 생활 환경 음성, 방언 |
| 해양 특화 LoRA | 해양 | 50~100시간 | 파도, 바람, 장비 소음, 코덱 왜곡 |
| 항공 특화 LoRA | 항공 | 50~100시간 | 로터음, 항공 관제 용어, 외국어 혼재 |
| **합계 (최소)** | - | **350 ~ 600시간** | - |
| **합계 (최대 성능)** | - | **500 ~ 1,000시간** | 추가 데이터 투입 시 점진적 성능 향상 |

> **결론:** 단일 통합 모델로 최대 성능을 내기 위해서는 **500시간 이상**이 필요하며, **800~1,000시간**이 수확 체감(포화점)에 도달하는 실질적 상한선입니다. 단, 도메인별 LoRA를 분리한 계층 전략에서는 공통 베이스 200~300시간 + 도메인별 50~100시간의 **총 350~600시간**으로 동등하거나 더 높은 성능을 달성할 수 있습니다.

#### 7.6.7 데이터 수집·구성 우선순위 로드맵

현실적으로 데이터를 단계적으로 수집해야 하는 경우, 다음 순서로 투자 효과를 극대화합니다.

**Phase 1 (PoC, 0~50시간): 핵심 난이도 집중 검증**
*   코덱 왜곡이 심한 샘플(MELP/CVSD) 우선 수집 (투자 대비 WER 감소 효과 최대)
*   핵심 OOV 용어를 포함하는 교신 스크립트 최소 500문장 구축
*   3개 도메인 × 최소 3가지 SNR 레벨 × 3명 화자 이상 확보

**Phase 2 (1차 배포, 50~200시간): 잡음·화자 다양성 확장**
*   각 도메인별 다양한 잡음 환경(SNR 20dB, 10dB, 5dB, 0dB) 균등 분포 확보
*   화자 20인 이상, 지역 억양·성별 다양성 확보
*   외국어 혼재 교신(영어 포네틱 코드) 비율 운용 환경과 동일하게 구성

**Phase 3 (고도화, 200~500시간): 롱테일 시나리오 보완**
*   Phase 1·2 학습 모델의 Test CER 분석 → 오류 다발 구간의 추가 데이터 집중 수집
*   SNR -5dB 이하 극한 환경 샘플 추가 (전체 데이터의 10~15% 비중 권장)
*   지속 학습(Continual Learning) 파이프라인 구축, 운용 데이터 자동 피드백 적용

**Phase 4 (최대 성능, 500~1,000시간): 포화점 도달**
*   능동 학습(Active Learning)으로 모델이 여전히 틀리는 Hard Negative 샘플만 추가 수집
*   데이터 추가 투입 대비 CER 개선이 0.5% 미만으로 수렴하면 데이터 수집보다 **모델 앙상블·후처리 개선**으로 전환

---

### 7.7 지속적인 데이터셋 생성과 재튜닝(Continual Tuning)이 필요한 이유

ASR 파인튜닝은 한 번 완료하면 끝나는 '프로젝트'가 아니라, **살아있는 운용 환경에 지속적으로 적응해야 하는 '운영 프로세스'**입니다. 다음과 같은 이유로 데이터셋의 지속적인 확충과 주기적인 재학습(Re-tuning) 파이프라인 운영이 필수적입니다.

#### 7.7.1 운용 환경의 지속적인 변화 (Domain Shift)
*   **신규 장비 도입:** 새로운 무전기 기종, 위성 통신 장비, 암호화 단말기가 도입되면 코덱(Codec) 특성이 달라져 기존 모델이 새 장비의 음성 특성을 처리하지 못하는 **성능 저하(Performance Degradation)**가 발생합니다. 새 장비로 수집한 데이터로 재튜닝이 필요합니다.
*   **작전 환경 변화:** 도심 전투, 해안 상륙작전, 산악 작전 등 새로운 작전 환경이 추가될 때마다 그 환경 특유의 배경 잡음 패턴이 등장합니다. 기존 모델이 한 번도 경험하지 못한 새 잡음 패턴에는 성능이 급격히 떨어집니다.
*   **기상 및 계절적 요인:** 여름철 고온 다습 환경과 겨울철 건조 환경은 무전기의 특성 및 음성 전파 패턴에 미묘한 차이를 만들어냅니다. 장기 운용 시 계절별 성능 편차가 누적될 수 있습니다.

#### 7.7.2 언어·용어의 지속적인 진화 (Vocabulary Drift)
*   **신규 용어 및 은어 발생:** 군·안전 조직에서는 새로운 작전 계획, 장비명, 전술 약어가 지속적으로 생겨납니다. 최초 파인튜닝 시점에 존재하지 않던 OOV(Out-of-Vocabulary) 용어는 모델이 전혀 인식하지 못하거나, 발음이 비슷한 기존 단어로 잘못 전사합니다.
*   **코드워드·암호 갱신:** 주기적으로 교체되는 작전 코드명이나 암호 약어는 갱신 시마다 신규 용어로 모델에 학습시켜야 합니다.
*   **포네틱 코드 변형:** 교신 주체나 부대에 따라 NATO 표준이 아닌 자체적 변형 포네틱 코드를 사용하는 경우, 이 코드 패턴이 학습 데이터에 없으면 높은 오류율이 발생합니다.

#### 7.7.3 화자 분포의 변화 (Speaker Distribution Shift)
*   **인원 교체 사이클:** 군·안전 조직은 주기적인 인사이동으로 교신 주체가 바뀝니다. 기존 학습 데이터에 포함되지 않은 새로운 화자(다른 목소리 특성, 발음 습관, 억양)가 등장하면 인식률이 저하됩니다.
*   **화자 다양성의 한계:** 최초 학습 시 아무리 많은 화자를 포함했더라도, 운용 기간이 길어질수록 실제 사용자 풀이 변화합니다. 새 화자 데이터를 지속적으로 수집하지 않으면 모델은 오래된 화자 분포에 편향됩니다.

#### 7.7.4 모델 드리프트 및 피드백 루프 부재 (Model Drift)
*   **지식 노화(Knowledge Aging):** 언어 모델과 마찬가지로, ASR 모델도 학습 시점의 데이터 분포를 기반으로 합니다. 운용 환경이 조금씩 달라지면서 모델의 예측이 실제 환경과 점점 멀어지는 **드리프트(Drift)** 현상이 발생합니다.
*   **오류 누적:** 모델이 틀린 전사 결과를 교정 없이 계속 사용하면, 후단 시스템(명령 처리, 상황 판단)의 오류가 누적되어 운용 신뢰도가 점차 하락합니다. 이를 막기 위해 **오류 샘플을 정기적으로 수집하고 교정 후 재학습**하는 피드백 루프 파이프라인이 필수적입니다.

#### 7.7.5 지속적 학습(Continual Learning) 파이프라인 설계 원칙

위의 모든 이유들을 종합하면, ASR 시스템은 **정기적 점검 및 재학습 사이클**을 운영 프로세스로 내재화해야 합니다.

```
[지속적 학습 파이프라인 개요]

운용 중 자동 수집
┌─────────────────────┐
│ 실시간 전사 로그    │──→ 오류 의심 구간 자동 플래깅
│ (신뢰도 낮은 발화)  │    (모델 신뢰도 점수 임계값 이하)
└─────────────────────┘
          │
          ▼
┌─────────────────────┐
│ HITL검수 (6장 참조) │──→ 고품질 라벨 확정
│(주기: 월 1회 이상)  │
└─────────────────────┘
          │
          ▼
┌─────────────────────┐
│ 증분 재학습         │──→ 기존 학습 데이터 유지 + 신규 데이터 추가
│ (Incremental Train) │    재앙적 망각 방지를 위해 EWC 또는 혼합 재학습
└─────────────────────┘
          │
          ▼
┌─────────────────────┐
│ A/B 평가 배포       │──→ 신규 모델 CER이 기존 대비 개선 시 교체
│ (Test세트 자동 평가)│
└─────────────────────┘
```

**권장 재학습 주기:**

| 트리거 조건 | 재학습 주기 | 설명 |
| :--- | :---: | :--- |
| 신규 장비 도입 | 즉시 | 코덱 특성 변화에 즉각 대응 |
| 인원 교체 / 새 화자 5인 이상 | 월 1회 | 화자 분포 편향 방지 |
| 신규 용어·코드워드 갱신 | 갱신 시 즉시 | OOV 오류 차단 |
| 정기 성능 점검 CER 악화 | 분기 1회 | 드리프트 누적 리셋 |
| 신규 작전 환경 추가 | 환경 추가 시 | 새 잡음 패턴 적응 |

> **핵심:** ASR 시스템의 장기 운용 성공은 최초 모델의 완성도만큼이나 **데이터 수집 → 검수 → 재학습 → 평가**로 이어지는 지속적인 MLOps 파이프라인을 얼마나 체계적으로 운영하느냐에 달려 있습니다.

# 제8장. 학습 파이프라인 (Training Pipeline) 구현

데이터셋이 준비되었고 LoRA의 개념을 이해했다면, 이제 모델을 학습시키는 전체 파이프라인을 구축할 차례입니다. 본 장에서는 HuggingFace 생태계를 활용한 환경 세팅부터 H100 GPU에서의 OOM 방지 전략, 과적합 방지, 체크포인트 관리까지 End-to-End 흐름을 다룹니다.

---

### 8.1 HuggingFace `transformers` 및 `peft` 라이브러리 세팅

파인튜닝 코드를 밑바닥부터 짜는 대신, 검증된 오픈소스 라이브러리 생태계를 엮어 파이프라인을 구성합니다. 각 라이브러리의 역할과 상호 관계를 명확히 이해하는 것이 중요합니다.

#### 8.1.1 핵심 라이브러리 역할 및 의존 관계

| 라이브러리 | 역할 | 주요 사용 클래스/함수 |
| :--- | :--- | :--- |
| **transformers** | Whisper 베이스 모델 로드, 특징 추출, 토크나이저 | `WhisperForConditionalGeneration`, `WhisperProcessor` |
| **peft** | LoRA 어댑터 주입, 학습 파라미터 관리 | `LoraConfig`, `get_peft_model` |
| **datasets** | JSONL 학습 데이터 로드, 전처리 파이프라인 | `load_dataset`, `Audio` |
| **accelerate** | 멀티 GPU 학습 자동화, 혼합 정밀도 관리 | `Accelerator` |
| **evaluate** | WER/CER 평가 메트릭 계산 | `load("wer")`, `load("cer")` |

#### 8.1.2 모델 초기화 및 LoRA 적용 처리 흐름

학습 파이프라인 시작 시 모델을 준비하는 단계는 다음 순서로 진행합니다.

**1단계 - GPU 정밀도 확인 및 베이스 모델 로드**
H100 GPU의 bf16 지원 여부를 먼저 확인하고, 지원이 확인되면 `bfloat16`, 아니면 `float16` 정밀도로 모델을 로드합니다. `device_map="auto"` 옵션을 사용하면 멀티 GPU 환경에서 모델 레이어가 GPU들에 자동으로 분산 배치됩니다.

**2단계 - 프로세서 초기화 및 디코더 설정**
`WhisperProcessor`를 로드하면서 언어를 `"Korean"`, 작업을 `"transcribe"`로 고정합니다. 그 다음 모델의 디코더 강제 토큰(`forced_decoder_ids`)을 한국어 전사 토큰으로 설정합니다. 이 설정이 없으면 모델이 언어를 자동 감지하려 시도하면서 잡음이 심한 구간에서 엉뚱한 언어로 전사하는 오류가 발생할 수 있습니다.

**3단계 - LoRA 설정 정의 및 어댑터 주입**
`LoraConfig`에 랭크(`r`), 알파(`lora_alpha`), 드롭아웃(`lora_dropout`), 어댑터를 적용할 대상 모듈(`target_modules`)을 정의합니다. H100 환경에서는 VRAM이 충분하므로 `r=16~32`까지 여유롭게 설정할 수 있습니다. `get_peft_model()` 함수로 베이스 모델에 LoRA 어댑터를 주입하면, 전체 1.5B 파라미터 중 약 **0.5%만 학습 가능한 상태**가 됩니다.

### 왜 0.5%만 학습해도 충분할까?

0.5%라는 수치가 숫자로만 보면 매우 작아 보일 수 있지만, LoRA(Low-Rank Adaptation)의 핵심은 바로 그 '적은 파라미터로 최대의 효율'을 내는 데 있습니다. 결론부터 말씀드리면, 0.5% 내외의 파라미터만으로도 전사 정확도(WER)를 획기적으로 개선하는 데 충분하며, 때로는 전체 학습(Full Fine-tuning)보다 더 나은 결과를 보여주기도 합니다.

1. **'핵심'만 건드리는 전략 (Intrinsic Rank)**

대규모 모델(1.5B 등)은 이미 언어와 소리에 대한 방대한 지식을 가지고 있습니다. 우리가 하려는 '잡음 섞인 음성 전사'는 모델의 지식을 새로 만드는 것이 아니라, 기존 지식을 특정 도메인(잡음, 감쇄 음성)에 맞게 살짝 비트는 과정입니다. 학계 연구에 따르면, 모델을 특정 태스크에 적응시킬 때 실제로 변화가 필요한 차원(Intrinsic Dimension)은 전체의 아주 일부분에 불과합니다. LoRA는 이 '핵심 차원'만 골라 학습하기 때문에 0.5%로도 충분한 효과가 납니다.

2. **가중치 행렬의 마법: $A \times B$**

LoRA는 거대한 원래의 가중치 행렬을 직접 수정하지 않습니다. 대신 두 개의 아주 작은 행렬 $A$와 $B$를 도입합니다. 원본 행렬 ($W$): $1000 \times 1000$ (수정 안 함, Frozen) LoRA 행렬 ($A, B$): $1000 \times r$ 그리고 $r \times 1000$ (여기서 $r$은 랭크) 만약 $r=16$이라면, 업데이트해야 할 파라미터는 원본의 수천 분의 일로 줄어듭니다. 하지만 이 작은 행렬들이 원본 모델의 출력값에 더해지면서 "잡음이 섞였을 때는 이렇게 해석해"라는 보정 필터 역할을 완벽히 수행합니다.

3. **왜 0.5%가 오히려 정확도에 좋을까?**

*   **과적합(Overfitting) 방지:** 학습 가능한 파라미터가 너무 많으면, 모델이 학습 데이터의 노이즈나 특정 패턴을 통째로 '암기'해버립니다. 0.5%만 학습하면 모델이 일반적인 언어 능력을 유지하면서 꼭 필요한 특징만 학습하므로, 처음 보는 잡음 환경에서도 더 잘 버팁니다.

*   **파괴적 망각(Catastrophic Forgetting) 차단:** 전체 파라미터를 건드리면 모델이 원래 잘하던 기본 성능(일반 대화 인식 등)을 잃어버릴 수 있습니다. LoRA는 기존 지식을 얼려두기 때문에 안전합니다.

---

### 8.2 데이터셋 로드 및 전처리 파이프라인

#### 8.2.1 JSONL 데이터셋 로드 및 오디오 리샘플링 처리 흐름

6장에서 생성한 JSONL 파일을 HuggingFace `datasets` 라이브러리로 로드합니다. 이때 학습용(`train`)과 검증용(`validation`) 두 개의 파일을 각각 로드하여 `DatasetDict` 객체로 묶습니다. 이후 오디오 컬럼에 16kHz 리샘플링 설정을 적용합니다. 실제 리샘플링은 전처리 단계에서 오디오가 처음 읽힐 때 `ffmpeg`를 이용해 자동으로 처리됩니다.

#### 8.2.2 전처리 함수 구현 처리 흐름 (Feature Extraction + Tokenization)

학습 루프에 들어가기 전, 각 오디오 샘플을 모델이 이해할 수 있는 형식으로 변환하는 전처리 단계가 필요합니다. 전처리 함수는 배치 단위로 다음 두 가지 변환을 수행합니다.

*   **오디오 → 멜 스펙트로그램 변환:** 원시 오디오 파형(`numpy array`)을 Whisper의 Feature Extractor에 입력하면 80채널 로그 멜 스펙트로그램(`input_features`)이 반환됩니다. Whisper는 이 스펙트로그램을 입력으로 사용합니다.

*   **텍스트 → 토큰 ID 변환:** 정답 전사 텍스트를 토크나이저에 통과시켜 정수 ID 시퀀스(`labels`)로 변환합니다. 최대 길이(448 토큰)를 초과하는 문장은 자르고, 짧은 문장은 패딩 토큰으로 채웁니다. 이때 패딩 토큰 위치는 `-100`으로 대체하는데, 이는 손실(Loss) 계산 시 패딩 구간을 무시하도록 PyTorch에 알려주는 약속된 값입니다.

전처리가 완료된 데이터셋은 CPU 멀티프로세싱(`num_proc=8`)을 활용하여 병렬로 처리하면 H100 서버의 다수 CPU 코어를 활용해 전처리 시간을 크게 단축할 수 있습니다. 처리 완료 후 원본 오디오 컬럼은 삭제하여 디스크 공간을 절약합니다.

#### 8.2.3 데이터 콜레이터(Data Collator) 처리 흐름

전처리된 개별 샘플들은 각기 다른 길이의 시퀀스를 갖고 있습니다. GPU는 배치 내 모든 샘플이 동일한 형태여야 하므로, 학습 루프마다 배치를 구성할 때 자동으로 길이를 맞춰주는 **데이터 콜레이터**가 필요합니다.

콜레이터는 배치 내 샘플들을 받아 다음 두 작업을 수행합니다.

*   **입력 피처 패딩:** 각 샘플의 멜 스펙트로그램(`input_features`)을 배치 내 가장 긴 것에 맞춰 패딩하고 PyTorch 텐서로 변환합니다.
*   **레이블 패딩:** 각 샘플의 토큰 ID 시퀀스(`labels`)를 배치 내 가장 긴 것에 맞춰 패딩하되, 추가된 패딩 위치는 `-100`으로 마스킹합니다.

---

### 8.3 최적의 하이퍼파라미터(Hyper-parameters) 설정 가이드

학습의 성패는 하이퍼파라미터 튜닝에 달려 있습니다. **H100(80GB HBM3) GPU 1장 기준**의 권장 설정값을 제시합니다.

#### 8.3.1 배치 사이즈와 그래디언트 누적

*   **H100 단일 GPU (80GB) 기준:** LoRA r=16, bf16 기준으로 `per_device_train_batch_size=32~64`가 안정적으로 동작합니다.
*   **유효 배치 사이즈(Effective Batch Size) 계산:**
    ```
    유효 배치 사이즈 = per_device_train_batch_size × gradient_accumulation_steps × num_GPUs
    ```
    예: `batch=32 × accum=2 × 1GPU = 64` (업계 권장: 64~128)
*   **그래디언트 누적(Gradient Accumulation):** 물리적으로 메모리에 올릴 수 없는 큰 유효 배치 사이즈를 달성하기 위해, 여러 스텝에 걸쳐 그래디언트를 누적한 뒤 한 번에 가중치를 업데이트합니다. H100에서는 메모리가 충분하므로 배치 사이즈를 직접 키우는 것이 더 효율적입니다.

#### 8.3.2 학습률 (Learning Rate)과 스케줄러

*   **권장 학습률:** LoRA 파인튜닝에서는 Full Fine-Tuning보다 큰 학습률을 사용합니다. 한국어 도메인 특화 기준으로 `1e-3` ~ `5e-4` 범위에서 시작합니다.
*   **학습률 스케줄러:** 처음에 학습률을 천천히 올리고(Warm-up), 이후 점진적으로 낮추는 방식이 학습 안정성을 크게 높입니다.
    *   `lr_scheduler_type="linear"` (기본 권장)
    *   `warmup_steps=100~500` (전체 스텝의 약 5~10%)
*   **과적합 모니터링:** 학습 손실(Train Loss)은 계속 감소하는데 검증 손실(Validation Loss)이 증가하기 시작하면 과적합 신호입니다. 이 시점에 학습을 중단해야 합니다.

> **[심화] 학습률·워밍업·스케줄러 직관적 이해 — '안개 낀 산에서 골짜기 찾기'**
>
> 학습률(Learning Rate)은 모델이 정답을 향해 **한 번에 얼마나 큰 보폭으로 움직일지** 결정하는 수치입니다. 복잡한 수식 대신 산에서 길 찾는 비유로 이해하면 쉽습니다.
>
> **① 학습률 = 보폭의 크기**
>
> | 상황 | 비유 | 결과 |
> | :--- | :--- | :--- |
> | **학습률이 너무 높을 때** | 한 걸음이 너무 커서 골짜기를 지나 반대편 산등성이로 훌쩍 넘어감 | 왔다 갔다 발산(Divergence), 학습 망가짐 |
> | **학습률이 너무 낮을 때** | 조금씩만 움직여 정답까지 너무 오래 걸리고, 작은 웅덩이(지역 최솟값)에 빠져 멈춤 | 훈련 시간 폭증, 차선책에서 멈춤 |
> | **적절한 학습률** | 처음엔 큼직하게 내려오다가 정답 근처에서 조심스럽게 안착 | 빠르고 안정적인 수렴 |
>
> **② LoRA에서 학습률을 더 크게 쓰는 이유**
>
> 전체 파라미터의 0.5%만 학습하기 때문입니다.
> *   **Full Fine-Tuning:** 15억 개의 나사를 동시에 조이므로, 조금만 크게 움직여도 전체 기계가 뒤틀릴 위험이 있어 아주 작은 학습률을 써야 합니다.
> *   **LoRA:** 기계는 그대로 두고 작은 어댑터 부품만 교체하는 수준이므로, 전반적인 지식을 흔들지 않습니다. 그래서 `1e-3(0.001)` 처럼 더 과감한 학습률로 빠르게 적응시키는 것이 더 효과적입니다.
>
> **③ 워밍업(Warm-up)과 스케줄러 = 준비운동 + 정리운동**
>
> *   **워밍업:** 처음부터 전력 질주하면 근육이 놀라듯, 학습 초반에는 아주 작은 학습률로 시작해서 목표값까지 천천히 올립니다. 모델이 데이터에 적응할 시간을 줍니다.
> *   **스케줄러:** 목적지 근처에서는 속도를 줄여야 정확히 멈출 수 있듯, 학습이 진행될수록 학습률을 서서히 낮춰 정교하게 다듬어지도록 돕습니다.
>
> | 개념 | 비유 | 핵심 |
> | :--- | :--- | :--- |
> | 학습률 | 보폭의 크기 | 너무 크면 정답을 지나치고, 너무 작으면 세월아 네월아 |
> | 워밍업 | 준비운동 | 초반에 모델이 놀라지 않게 살살 시작 |
> | 스케줄러 | 속도 조절 | 끝으로 갈수록 세밀하게 조정하기 위해 천천히 걷기 |
>
> **실무 팁:** `1e-3`으로 시작하여 초기 Loss가 요동치지 않고 부드럽게 내려가는지 확인하는 것이 첫 번째 체크포인트입니다. H100 같은 고사양 GPU를 쓰고도 학습률 설정을 잘못하면 모델 성능이 나오지 않으므로, 반드시 TensorBoard에서 Loss 곡선을 확인하며 튜닝하십시오. (→ 8.7절 모니터링 참조)

#### 8.3.3 에포크(Epoch) vs 스텝(Step) 기반 학습 전략

| 전략 | 권장 상황 | 설정 예 |
| :--- | :--- | :--- |
| **에포크 기반** | 데이터셋이 소규모(~30시간)일 때 | `num_train_epochs=5~10` |
| **스텝 기반** | 대규모 데이터(100시간+) 또는 지속 학습 | `max_steps=5000~20000` |

*   소규모 데이터(10~30시간)는 **5~10 에포크** 권장. 단, 10 에포크 이상은 과적합 위험이 높아집니다.
*   복합 난이도 데이터(500시간 이상, 7장 참조)는 **스텝 기반**으로 설정하고 체크포인트마다 CER을 확인하며 조기 종료를 활용합니다.

### 8.4 혼합 정밀도(Mixed Precision) 및 메모리 최적화

#### 8.4.1 혼합 정밀도 연산 (bf16 vs fp16)

H100 환경에서는 `fp16` 대신 반드시 **`bf16`(BFloat16)**을 사용합니다.

| 특성 | fp32 | fp16 | **bf16** |
| :--- | :---: | :---: | :---: |
| 메모리 | 4 bytes | 2 bytes | **2 bytes** |
| 표현 범위 | 매우 넓음 | 좁음 (언더플로우 위험) | **fp32와 동일** |
| 연산 속도 | 기준 | ~2배 빠름 | **~2배 빠름** |
| H100 TensorCore | O | O | **O (최적화)** |

*   **bf16 권장 이유:** fp16은 아주 작은 숫자를 표현할 때 언더플로우(0으로 수렴)가 발생하여 학습이 불안정해지는 버그가 있습니다. bf16은 fp32와 동일한 수 범위를 유지하면서도 메모리는 fp16처럼 절반만 사용합니다.
*   H100의 TensorCore는 bf16 연산을 하드웨어 수준에서 최적화하여 fp32 대비 **약 2~4배** 빠른 연산 속도를 발휘합니다.

#### 8.4.2 그래디언트 체크포인팅(Gradient Checkpointing)

VRAM을 추가로 절약하기 위해 사용하는 기법입니다. 일반적으로는 순전파(Forward Pass) 중 중간 결과물(Activation)을 모두 메모리에 보관하지만, 그래디언트 체크포인팅은 **일부 레이어의 중간 결과를 버리고 역전파 시 재계산**하여 VRAM을 30~40% 절감합니다. H100에서도 더 큰 배치 사이즈 확보를 위해 활성화를 권장합니다.

설정 방법은 모델 객체에 `gradient_checkpointing_enable()`을 호출하는 것으로, PEFT와 함께 사용할 때는 `enable_input_require_grads()`도 함께 호출해야 그래디언트가 어댑터 행렬까지 올바르게 흘러 들어갑니다.

---

### 8.5 학습 실행 설정 (`Seq2SeqTrainingArguments`) 주요 파라미터 설명

`Seq2SeqTrainingArguments`는 학습 루프의 모든 동작을 제어하는 설정 객체입니다. 주요 파라미터 그룹과 각각의 의미는 다음과 같습니다.

**[출력 및 기본 경로]**
*   `output_dir`: 체크포인트와 최종 모델이 저장될 디렉터리 경로입니다.

**[배치 및 학습률]**
*   `per_device_train_batch_size`: GPU 1장당 한 번에 처리하는 샘플 수입니다. H100 80GB 기준 `32~64`를 권장합니다.
*   `gradient_accumulation_steps`: 가중치 업데이트 전에 그래디언트를 누적할 스텝 수입니다. 유효 배치 사이즈 = 배치 × 누적 스텝입니다.
*   `learning_rate`: 파인튜닝 시작 시의 학습률입니다. LoRA에서는 `1e-3` 내외를 권장합니다.
*   `warmup_steps`: 학습 초반에 학습률을 0에서 목표값까지 천천히 올리는 단계의 스텝 수입니다. 급격한 초기 학습을 방지합니다.
*   `num_train_epochs` / `max_steps`: 에포크 기반 또는 스텝 기반 중 하나를 선택합니다.

**[혼합 정밀도 (H100 필수)]**
*   `bf16=True`, `fp16=False`: H100에서는 반드시 bf16을 활성화하고 fp16은 비활성화합니다.

**[평가 및 저장 전략]**
*   `evaluation_strategy="steps"`, `eval_steps=500`: 500 스텝마다 검증셋으로 WER을 평가합니다.
*   `save_strategy="steps"`, `save_steps=500`: 평가와 동일한 주기로 체크포인트를 저장합니다.
*   `save_total_limit=3`: 디스크 공간 절약을 위해 가장 최근 3개의 체크포인트만 유지합니다.
*   `load_best_model_at_end=True`: 학습이 끝나면 중간에 저장된 체크포인트 중 WER이 가장 낮았던 시점의 가중치를 자동으로 복원합니다.

**[성능 최적화]**
*   `gradient_checkpointing=True`: 8.4.2절의 VRAM 절감 기법을 활성화합니다.
*   `dataloader_num_workers=8`: 데이터 로딩을 위한 CPU 병렬 워커 수입니다. GPU 가동 중 CPU에서 미리 다음 배치를 준비합니다.
*   `dataloader_pin_memory=True`: 데이터를 CPU에서 GPU로 전송할 때 고정 메모리(Pinned Memory)를 사용하여 전송 속도를 높입니다.

### 8.6 평가 메트릭 정의 및 Trainer 실행 흐름

#### 8.6.1 WER/CER 평가 함수 처리 흐름

`Seq2SeqTrainer`는 지정된 평가 주기마다 검증셋 전체를 처리하고 `compute_metrics` 함수를 호출합니다. 이 함수의 내부 처리 순서는 다음과 같습니다.

1.  **패딩 토큰 복원:** 레이블 배열에서 손실 계산 무시를 위해 삽입했던 `-100` 값을 원래의 패딩 토큰 ID로 되돌립니다.
2.  **토큰 ID → 텍스트 디코딩:** 모델이 생성한 예측 토큰 ID 배열과 정답 레이블 토큰 ID 배열을 각각 `batch_decode()`로 사람이 읽을 수 있는 텍스트로 변환합니다. 이때 `[BOS]`, `[EOS]` 등의 특수 토큰은 제거합니다.
3.  **WER/CER 계산:** 디코딩된 예측 텍스트 목록과 정답 텍스트 목록을 `evaluate` 라이브러리의 WER·CER 함수에 입력하여 평균 지표를 계산합니다.
4.  **결과 반환:** 계산된 WER·CER 수치를 딕셔너리 형태로 반환합니다. Trainer는 이 값을 TensorBoard에 자동으로 기록하고, 최고 성능 체크포인트 선택 기준으로 활용합니다.

#### 8.6.2 Trainer 구성 및 학습 실행 흐름

`Seq2SeqTrainer` 객체는 모델, 학습 설정, 학습·검증 데이터셋, 데이터 콜레이터, 평가 함수, 콜백(Callback)을 하나로 묶는 최상위 조율자(Orchestrator)입니다.

**EarlyStoppingCallback 동작 방식:** 검증 WER이 `early_stopping_patience=3`으로 설정된 경우, WER이 3회 연속 평가에서 개선되지 않으면 학습을 자동 중단합니다. 데이터 과적합 진입 시점을 자동으로 감지하여 불필요한 학습 시간 낭비를 방지합니다.

**학습 실행 흐름:**
```
trainer.train() 호출
    │
    ├─ 배치 로드 → 순전파(Forward) → 손실(Loss) 계산
    │   → 역전파(Backward) → LoRA 어댑터 파라미터만 업데이트
    │   → 지정 스텝마다 WER 평가 → TensorBoard 기록
    │   → 지정 스텝마다 체크포인트 저장
    │   → EarlyStopping 조건 충족 시 자동 종료
    │
    └─ 완료 후 최고 WER 체크포인트 자동 복원

trainer.model.save_pretrained() 호출
    → LoRA 어댑터 가중치만 지정 경로에 저장
    → 이후 9장에서 베이스 모델과 병합(Merge)
```

### 8.7 학습 모니터링 및 TensorBoard 활용

#### 8.7.1 주요 모니터링 지표

학습 중 다음 지표를 실시간으로 확인하며 이상 신호를 조기에 포착합니다.

| 지표 | 정상 패턴 | 이상 신호 | 대응 방안 |
| :--- | :--- | :--- | :--- |
| **Train Loss** | 단조 감소 | 감소하다 급격히 튀어오름 | 학습률 낮춤, 배치 확인 |
| **Validation Loss** | 초반 감소 후 수렴 | Train Loss는 감소하는데 Validation Loss가 증가 | **과적합** → EarlyStopping |
| **Validation WER** | 점진적 감소 | 전혀 줄지 않거나 증가 | 데이터 품질 점검, 학습률 조정 |
| **GPU 메모리 사용률** | 80~95% 유지 | OOM(Out of Memory) 오류 | 배치 사이즈 줄이기, Gradient Checkpointing 활성화 |

#### 8.7.2 TensorBoard 실행 방법

TensorBoard는 학습 중 또는 완료 후, 별도 터미널에서 `tensorboard --logdir=./logs --port=6006` 명령어로 실행합니다. 이후 브라우저에서 `http://localhost:6006`에 접속하면 Train Loss, Validation Loss, WER 추이를 그래프로 실시간 확인할 수 있습니다.

#### 8.7.3 학습 중 자주 발생하는 문제 해결(Troubleshooting)

*   **OOM (Out of Memory) 오류 발생 시:**
    1. 배치 사이즈(`per_device_train_batch_size`)를 절반으로 줄이고 그래디언트 누적 스텝(`gradient_accumulation_steps`)을 2배로 늘려 유효 배치 사이즈를 유지합니다.
    2. 그래디언트 체크포인팅(`gradient_checkpointing=True`) 설정이 활성화되어 있는지 확인합니다.
    3. 모델 로드 시 bf16 정밀도 혹은 8-bit 양자화 로드 옵션을 적용했는지 확인합니다.

*   **Validation WER이 전혀 줄지 않을 때:**
    1. 학습 데이터의 라벨 정확도를 점검합니다. 오전사 비율이 높으면 모델이 오답을 학습하게 됩니다.
    2. 학습률을 10배 낮추거나 높여 가며 민감도를 탐색합니다.
    3. 디코더 강제 토큰(`forced_decoder_ids`)이 한국어로 올바르게 고정되어 있는지 확인합니다.

*   **Train Loss와 Validation Loss 격차가 매우 클 때 (심각한 과적합):**
    1. LoRA 드롭아웃(`lora_dropout`) 값을 0.05에서 0.1~0.2로 높여 정규화를 강화합니다.
    2. 학습 에포크 수를 줄입니다.
    3. 데이터 증강(SpecAugment 등) 적용을 검토합니다.

# 제9장. 평가 및 추론 최적화 (Evaluation & Inference)

모델 학습(LoRA 파인튜닝)이 완료되었다면, 이 모델이 정말로 잡음 환경과 도메인 용어를 잘 인식하는지 **정량적으로 평가**하고, 실제 서비스에 배포할 수 있는 형태로 **추론을 최적화**하는 단계로 넘어가야 합니다. 본 장은 학습 완료 후 운영 배포 직전까지의 전 과정을 다룹니다.

---

### 9.1 정량적 성능 평가 지표 (WER, CER)

모델의 성능은 주관적 인상이 아니라, 수학적 지표를 통해 베이스 모델 대비 또는 타 시스템과 **객관적으로 비교 가능한 수치**로 표현해야 합니다.

#### 9.1.1 단어 오류율 (WER: Word Error Rate)

*   **개념:** 영어 기반 ASR에서 가장 널리 쓰이는 지표로, 띄어쓰기(Word) 단위로 정답 문장과 예측 문장의 차이를 측정합니다.
*   **계산 공식:**
    ```
    WER = (S + D + I) / N

      S (Substitution): 다른 단어로 잘못 인식한 수
      D (Deletion):     있어야 할 단어를 빠뜨린 수
      I (Insertion):    없는 단어를 끼워넣은 수
      N:                정답 문장의 전체 단어 수
    ```
*   **예시:** 정답: "현재 좌표 알파 브라보" (4 단어), 예측: "현재 좌표 알파 브**레이**" → S=1, WER = 1/4 = **25%**
*   **한계점:** 한국어는 교착어 특성상 "먹었다"와 "먹었습니다"를 완전히 다른 단어로 취급하므로 WER이 실제 성능을 지나치게 낮게 평가하는 경향이 있습니다.

#### 9.1.2 문자 오류율 (CER: Character Error Rate)

*   **개념:** 단어가 아닌 '글자(Character)' 단위로 오류를 측정합니다. 한국어·중국어·일본어 등 형태소 분석이 어려운 언어의 ASR 평가에서 **사실상 표준(De Facto Standard)**으로 쓰입니다.
*   **계산 공식:** WER과 동일하나 단위를 '글자'로 변경.
*   **결과 해석:**
    *   `CER 10%`: 100글자를 말할 때 10글자를 틀리게 전사.
    *   `CER 5% 이하`: 실무 운용 가능 수준.
    *   `CER 3% 이하`: 매우 높은 정확도, 전문가 수준.
*   **왜 한국어에서는 WER보다 CER인가?** 한국어는 교착어의 특성상 띄어쓰기에 따라 의미 전달의 차이가 크지 않으면서도 정작 띄어쓰기 규칙은 매우 복잡합니다.
    *   **WER(어절 단위)의 한계:** "학교에 갔다"를 "학교에갔다"로 전사할 경우, 의미는 통하지만 어절 단위에서는 100% 오답으로 처리됩니다. 
    *   **CER(글자 단위)의 강점:** 띄어쓰기 오류가 전체 점수를 갉아먹는 것을 방지하고, 모델이 '음절'을 얼마나 정확하게 잡아냈는지 실질적인 성능을 측정할 수 있게 해줍니다.
    *   **CER 계산 공식 (수학적 정의):** 모델이 예측한 문장과 실제 정답(Ground Truth) 사이의 편집 거리(Levenshtein Distance)를 기반으로 계산합니다.
        $$CER = \frac{S + D + I}{N} \times 100 (%) $$
        S (Substitution, 대치): 글자를 틀리게 인식 (예: 사과 → 사구)
        D (Deletion, 삭제): 있어야 할 글자가 빠짐 (예: 사과 → 사)
        I (Insertion, 삽입): 없는 글자가 추가됨 (예: 사과 → 사과꽃)
        N: 정답 문장의 전체 글자 수 

#### 9.1.3 실무적 데이터 전처리 (Normalization)

CER을 측정하기 전, '공정한 평가'를 위해 반드시 정규화 과정을 거쳐야 합니다. 이 과정이 없으면 실제 모델 성능보다 CER이 나쁘게 나올 수 있습니다.

*   **구두점 및 특수문자 제거:** 마침표(.), 쉼표(,), 물음표(?) 등은 인식 성능보다는 스타일의 영역이므로 제거 후 비교합니다.
*   **공백 정규화:** 연속된 공백을 하나로 합치거나 전체 공백을 제거하고 비교하기도 합니다.
*   **수치 및 단위 변환:** "10퍼센트"와 "10%"를 동일하게 처리할지 기준을 정해야 합니다.

#### 9.1.4 평가 처리 흐름

파인튜닝된 모델의 성능을 측정하려면, 학습 시 한 번도 노출하지 않은 **별도의 테스트 데이터셋**을 미리 분리해 두어야 합니다. 평가는 다음 순서로 진행합니다.

1.  **테스트 데이터 준비:** 6장에서 구축한 전체 데이터셋의 약 10~20%를 학습에 사용하지 않고 평가 전용으로 보관합니다. 이 테스트셋은 각 도메인(육상·해상·항공)과 잡음 수준(SNR 20dB, 5dB, 0dB)이 균등하게 포함되어야 실제 운용 환경을 대표할 수 있습니다.

2.  **모델 추론 실행:** Faster-Whisper 기반으로 변환된 파인튜닝 모델을 테스트 오디오 파일에 순서대로 적용하여 전사 텍스트(예측값)를 생성합니다. 이때 사용하는 추론 설정(언어 고정, 빔 탐색 크기 등)은 실제 운용 서버와 동일하게 맞춰야 일관된 평가 결과를 얻을 수 있습니다.

3.  **정답 대조 및 지표 계산:** 생성된 예측 텍스트 전체와 정답 텍스트 전체를 `evaluate` 라이브러리의 WER·CER 계산 함수에 입력하면 전체 테스트셋에 대한 평균 WER·CER이 반환됩니다.

4.  **베이스 모델과 비교 보고:** 동일한 테스트셋으로 파인튜닝 전 베이스 모델(Whisper Large-v3)의 WER·CER도 측정하여, 파인튜닝 전후의 수치를 나란히 비교한 표를 작성합니다. 이것이 파인튜닝 효과를 공식적으로 보고하는 핵심 산출물입니다.

---

### 9.2 학습된 LoRA 어댑터 병합(Merge) 및 가중치 추출

학습이 끝난 직후 디렉터리에는 수십 MB의 가벼운 LoRA 어댑터 파일(`adapter_model.bin`)만 저장됩니다. 이 상태로 바로 추론하면 매번 베이스 모델과 어댑터를 합산하는 연산이 발생하여 추론 속도가 저하됩니다.

#### 9.2.1 어댑터 병합(Merge) 처리 흐름

**처리 순서:**
1.  **베이스 모델 로드:** HuggingFace에서 원본 `openai/whisper-large-v3` 모델을 bf16 정밀도로 메모리에 올립니다.
2.  **LoRA 어댑터 적용:** `PeftModel.from_pretrained()`를 통해 학습 완료된 LoRA 어댑터를 베이스 모델 위에 씌웁니다. 이 시점에서 모델 구조는 "베이스 + 어댑터"의 이중 구조입니다.
3.  **수학적 병합 실행:** `merge_and_unload()` 함수를 호출합니다. 내부적으로 7장에서 설명한 `W = W₀ + (α/r) × BA` 수식에 따라 어댑터 가중치가 베이스 가중치 안으로 합산됩니다. 이 연산이 완료되면 어댑터는 사라지고, 파인튜닝된 지식이 내재된 순수한 Whisper 모델 하나만 남습니다.
4.  **디스크 저장:** 병합된 모델을 HuggingFace 포맷으로 지정 경로에 저장합니다. 이 시점부터는 PEFT 라이브러리 없이도 모델을 로드하고 추론할 수 있습니다.

#### 9.2.2 어댑터 유지 vs 병합의 선택 기준

| 전략 | 장점 | 단점 | 적합한 상황 |
| :--- | :--- | :--- | :--- |
| **어댑터 유지** | 도메인별 어댑터 스위칭 가능, 베이스 모델 공유 | 추론 시 약간의 오버헤드 | 멀티 도메인 서비스 (7.4.3 참조) |
| **어댑터 병합** | 추론 오버헤드 0, 배포 단순화 | 베이스 모델 분리 불가 | 단일 도메인 전용 서버 배포 |

---

### 9.3 CTranslate2 변환 및 Faster-Whisper 추론 최적화

HuggingFace의 `transformers` 기본 추론 코드는 학습에 유리하나 추론 속도가 상대적으로 느립니다. 실제 서비스 배포 시에는 반드시 전용 추론 엔진인 **CTranslate2 + Faster-Whisper**를 도입해야 합니다.

#### 9.3.1 CTranslate2 변환 처리 흐름

병합이 완료된 HuggingFace 포맷의 모델을 CTranslate2 전용 바이너리 포맷으로 변환합니다. 변환 도구(`ct2-transformers-converter`)를 사용하며, 이 과정에서 양자화를 동시에 적용합니다.

**변환 시 핵심 설정값:**
*   `--model`: 9.2절에서 저장한 병합 모델의 경로를 지정합니다.
*   `--output_dir`: CT2 포맷으로 변환된 결과물이 저장될 경로를 지정합니다.
*   `--quantization`: 가중치를 어떤 정밀도로 압축할지 결정합니다. H100 환경에서는 `bfloat16`이 정확도 손실 없이 가장 권장됩니다.

**양자화 옵션별 비교:**

| 양자화 | VRAM 사용량 | 추론 속도 | 정확도 손실 | 권장 환경 |
| :--- | :---: | :---: | :---: | :--- |
| `float32` | 기준 | 느림 | 없음 | 평가 기준선 측정 시 |
| `bfloat16` | ×0.5 | 빠름 | **거의 없음** | **H100 (권장)** |
| `int8` | ×0.25 | 매우 빠름 | 미미함 | 추론 속도 최우선 시 |
| `int8_float16` | ×0.25 | 매우 빠름 | 약간 있음 | 균형형 |

#### 9.3.2 Faster-Whisper 추론 설정 및 주요 파라미터 의미

Faster-Whisper를 이용한 추론 시 각 파라미터의 역할을 정확히 이해하고 설정해야 합니다.

**모델 로드 시 주요 설정:**
*   `device`: 추론에 사용할 장치를 지정합니다. GPU 추론 시 `"cuda"`, CPU 추론 시 `"cpu"`로 설정합니다.
*   `compute_type`: 연산 정밀도를 지정합니다. H100에서는 반드시 `"bfloat16"`을 사용합니다.
*   `device_index`: 멀티 GPU 환경에서 어떤 GPU를 사용할지 번호로 지정합니다.

**전사 호출 시 주요 파라미터:**
*   `language`: 전사 언어를 고정합니다. 한국어는 `"ko"`로 지정합니다. 이를 지정하지 않으면 모델이 언어 자동 감지를 수행하여 처리 속도가 느려지고, 잡음이 심한 구간에서 잘못된 언어로 판정하는 오류가 발생할 수 있습니다.
*   `beam_size`: 빔 탐색(Beam Search)의 후보 개수를 설정합니다. 값이 클수록 더 많은 경우의 수를 탐색하여 정확도가 높아지지만 처리 속도는 느려집니다. 실무에서는 `5`가 속도와 정확도의 균형점으로 권장됩니다.
*   `vad_filter`: 내장 VAD(Voice Activity Detection) 필터 사용 여부입니다. `True`로 설정하면 무음 및 잡음 구간을 자동으로 건너뛰어 처리 속도가 향상되고 환각(Hallucination) 발생이 줄어듭니다. 5장의 VAD 전략과 연계되는 설정입니다.
*   `vad_parameters.threshold`: VAD 민감도입니다. 값이 낮을수록 조금이라도 음성처럼 보이는 구간을 모두 발화로 인식합니다. 잡음이 심한 환경에서는 `0.3` 내외의 낮은 값(보수적 설정)을 권장합니다.
*   `no_speech_threshold`: 모델이 이 값 이상의 확률로 "음성 없음"이라고 판단하는 구간은 전사 결과를 출력하지 않습니다. 무음 구간에서 쓸모없는 텍스트가 생성되는 환각 현상을 방지합니다.
*   `condition_on_previous_text`: 이전 세그먼트의 전사 결과를 다음 세그먼트의 디코딩 힌트로 사용할지 결정합니다. 무전 교신처럼 반복 문장 구조가 없는 경우 `False`로 설정하는 것이 환각 방지에 유리합니다.
*   `word_timestamps`: 단어 단위 타임스탬프 생성 여부입니다. 10장의 화자 분할 통합 시 정밀한 시간 매핑을 위해 `True`로 설정합니다.

#### 9.3.3 H100 기준 추론 성능 목표치

| 설정 | 오디오 1시간 처리 시간 | RTF |
| :--- | :---: | :---: |
| HuggingFace 기본 (fp32) | ~60분+ | ~1.0 |
| Faster-Whisper (bfloat16, batch=1) | ~3~5분 | ~0.05~0.08 |
| **Faster-Whisper (bfloat16, batch=16)** | **~1~2분** | **~0.01~0.02** |
| Insane-Fast-Whisper (Flash Attention 2) | <1분 | <0.01 |

*   **실무 목표:** RTF 0.05 이하 (1시간 녹음을 3분 내 전사) → 단일 H100에서 Faster-Whisper 배치 처리로 충분히 달성 가능.

# Part 4. 후처리 및 서빙 (실전 운영 편)

## 제10장. 전사 텍스트 후처리 (Post-processing)

Whisper가 뱉어낸 텍스트가 아무리 정확하더라도, 최종 사용자가 읽기 편한 포맷으로 가공하지 않으면 실무에서 외면받습니다. 본 장에서는 모델 추론 직후 텍스트를 다듬는 필수 후처리 기술들을 소개합니다. 이 단계들은 파이프라인의 **마지막 방어선**으로, 모델이 미처 교정하지 못한 오류를 규칙(Rule) 기반 로직으로 보완합니다.

---

### 10.1 화자 분할 (Speaker Diarization) 기술 통합

무전 교신이나 회의 녹음에는 여러 명의 화자가 등장합니다. **"누가(Who) 언제(When) 무슨 말을 했는가"**를 구조화된 대화록 형태로 분리해 내는 기술이 화자 분할(Diarization)입니다.

#### 10.1.1 Whisper의 화자 분할 한계

Whisper 모델 자체에는 화자를 구분하는 기능이 없습니다. 들려오는 모든 소리를 하나의 연속된 텍스트로 전사할 뿐입니다. 따라서 화자 정보를 얻으려면 별도의 전문 모델을 결합해야 합니다.

#### 10.1.2 Pyannote Audio 연동 아키텍처

화자 분할 파이프라인은 두 개의 AI 모델을 **병렬로** 실행한 뒤, 결과를 하나로 병합하는 방식으로 동작합니다.

**파이프라인 처리 흐름:**
```
원본 오디오 파일
      │
      ├──→ [Pyannote Audio]  ──→ 화자 타임스탬프 생성
      │    (화자 분할 모델)         "Speaker_A: 0.0s ~ 4.1s"
      │                             "Speaker_B: 4.5s ~ 10.3s"
      │
      └──→ [Faster-Whisper] ──→ 전사 텍스트 + 타임스탬프 생성
           (ASR 모델)              "안녕하세요, 오늘 회의 주제는: 0.2s ~ 3.9s"
                                   "네, 말씀해 주세요: 4.6s ~ 6.1s"
                │
                ▼
      [타임스탬프 교집합 병합 로직]
                │
                ▼
      [Speaker_A] 안녕하세요, 오늘 회의 주제는 신제품 출시 일정입니다.
      [Speaker_B] 네, 말씀해 주세요.
```

**처리 단계별 상세 설명:**

*   **1단계 - 화자 분할 모델 실행:** Pyannote Audio 모델이 오디오 파형의 음색(Voice Biometrics) 패턴을 분석합니다. 화자의 실제 발화 내용은 처리하지 않고, "이 구간에서 말하고 있는 사람의 음색이 누구인가"만 판별합니다. 결과물은 `(시작 시각, 종료 시각, 화자 ID)` 형태의 타임스탬프 목록입니다.

*   **2단계 - ASR 전사 실행:** Faster-Whisper가 오디오의 내용을 텍스트로 변환하고, 각 발화 세그먼트의 시작·종료 시각 정보를 함께 생성합니다. 단어 단위 타임스탬프 옵션을 활성화하면 더욱 정밀한 매핑이 가능합니다.

*   **3단계 - 타임스탬프 교집합 병합:** Pyannote의 화자 타임스탬프와 Whisper의 발화 타임스탬프를 시간 축 위에서 비교합니다. 각 Whisper 발화 구간의 중간 시각이 어떤 화자의 구간 안에 속하는지 확인하여, 해당 발화에 화자 레이블을 부여합니다. 두 모델의 타임스탬프가 완벽히 일치하지 않아도 중간값 기준으로 귀속시키기 때문에 안정적으로 동작합니다.

*   **최종 결과:** `[화자 A: 0.2s] 안녕하세요, 오늘 회의 주제는 신제품 출시 일정입니다. / [화자 B: 4.6s] 네, 말씀해 주세요.` 형태의 구조화된 대화록이 완성됩니다.


---

### 10.2 ITN (Inverse Text Normalization) 규칙 기반 변환

음성 인식 모델은 들리는 발음 그대로를 텍스트로 출력합니다. 이를 문서에서 통용되는 형식(숫자, 기호, 약어 등)으로 역변환하는 작업이 **ITN(역정규화)**입니다.

#### 10.2.1 ITN이 필요한 이유

| 전사 원문 (모델 출력) | ITN 적용 후 |
| :--- | :--- |
| 일 월 십오 일 오후 세 시 십 분 | 1월 15일 오후 3시 10분 |
| 백이십오만 원 | 1,250,000원 |
| 알파 브라보 | A·B (NATO 코드 약어 변환) |
| 영 점 팔 킬로미터 | 0.8km |
| 삼 회 반복 | 3회 반복 |

#### 10.2.2 ITN 처리 방식: 정규표현식(Regex) 기반 치환 파이프라인

ITN 모듈은 미리 정의된 **변환 규칙 목록**을 순서대로 텍스트에 적용하는 방식으로 동작합니다.

**처리 흐름:**
1.  전사 텍스트를 입력받습니다.
2.  규칙 목록을 순서대로 순회하며, 각 규칙의 탐색 패턴이 텍스트에 존재하는지 검사합니다.
3.  패턴이 발견되면 해당 부분을 정해진 형식으로 치환합니다.
4.  모든 규칙을 적용한 뒤의 최종 텍스트를 반환합니다.

**주요 규칙 예시:**

*   **포네틱 코드 변환 규칙:** "알파", "브라보", "찰리" 등 NATO 포네틱 알파벳이 단독 단어로 등장할 때, 해당 알파벳(A, B, C)으로 치환하는 규칙을 정의합니다. 단, "알파 팀"처럼 다른 명사와 결합된 경우는 치환하지 않도록 단어 경계(Word Boundary)를 조건으로 설정합니다.

*   **수치 단위 변환 규칙:** "킬로미터", "노트", "마일" 등 발음 형태의 단위 앞에 숫자가 올 때 `km`, `kts`, `mi` 등의 기호로 압축합니다.

*   **시각 표현 변환 규칙:** "오전/오후 X시 Y분" 패턴을 탐지하여 `HH:MM AM/PM` 형식의 시각 표현으로 변환합니다. 오후 시각의 경우 12를 더하는 연산이 필요합니다.

*   **한국어 숫자 표현 변환 규칙:** "일", "십", "백", "천", "만" 등의 한글 수사(數詞)를 결합하여 아라비아 숫자로 변환하는 로직입니다. 복잡도가 높아 `hangul_numeral` 등 전문 라이브러리를 활용하는 것을 권장합니다.

**설계 원칙:** ITN 규칙 목록은 소스 코드 안에 하드코딩하지 않고 별도의 YAML 또는 JSON 파일로 분리하여 관리합니다. 이렇게 하면 새로운 도메인 용어나 단위가 추가될 때 코드를 수정하지 않고 설정 파일만 업데이트하면 즉시 반영됩니다.

#### 10.2.3 ITN 관련 주요 패키지 및 실무 권장 전략

직접 Regex 파이프라인을 구현하는 방식 외에, 검증된 오픈소스 라이브러리를 활용하면 개발 공수를 크게 줄일 수 있습니다.

**[표] 한국어 ITN 관련 주요 패키지 비교**

| 패키지 | 제공사 | 특징 | 에어갭 적합성 |
| :--- | :--- | :--- | :---: |
| **`nemo_text_processing`** | NVIDIA NeMo | 한국어 포함 다국어 TN/ITN 공식 지원. Pynini 기반 FST(유한 상태 변환기) 문법 사용. 가장 완성도 높음 | ✅ (whl 사전 반입) |
| **`hangul_numeral`** | 오픈소스 | 한글 수사(일, 십, 백 …) → 아라비아 숫자 변환에 특화. 경량이고 의존성 단순 | ✅ |
| **`num2words`** | 오픈소스 | TN 방향(숫자→문자)만 지원, ITN 방향 미지원. 데이터 증강 시 보조 도구로 활용 | ✅ |
| **Kiwi** 형태소 분석기 | bab2min | ITN 자체는 아니나 한글 수사 분리·패턴 인식 전처리 보조 도구로 활용 가능 | ✅ |
| **자체 Regex 파이프라인** | 직접 구현 | 10.2.2절에 기술된 방식. 외부 의존성 없이 규칙 추가·수정이 용이 | ✅ |

**실무 권장 조합 전략 (단계별 적용):**

1.  **`hangul_numeral`로 한글 수사 변환:** 날짜·금액·횟수 등 숫자 표현을 아라비아 숫자로 먼저 변환합니다. 신뢰도가 높고 설치가 간단하여 가장 먼저 도입하는 것을 권장합니다.
2.  **자체 YAML 규칙으로 도메인 특화 처리:** 단위(`km`, `kts`), 약어, 포네틱 코드 등 서비스 특화 패턴은 10.2.2절의 Regex 파이프라인으로 직접 정의합니다. 운영팀이 코드 수정 없이 규칙을 추가할 수 있어 유지보수에 유리합니다.
3.  **`nemo_text_processing` (선택):** 날짜·시각·통화 등 범용 패턴의 완성도를 높이고 싶을 때 추가 도입합니다. 다만 `pynini` C++ 빌드 의존성이 있어 설치 난이도가 있습니다.

> ⚠️ **에어갭 환경 주의사항:** `nemo_text_processing`은 `pynini` 라이브러리에 의존하며, 이 라이브러리는 C++ 컴파일이 필요한 경우가 있습니다. 내부망 반입 시 운영 서버의 OS·Python 버전에 맞는 `manylinux` wheel 파일(`.whl`)을 개발 환경에서 미리 `pip download`로 수집하여 반입해야 합니다. `hangul_numeral`과 자체 Regex 파이프라인 조합은 이러한 빌드 문제가 없어 에어갭 환경에서 가장 안전한 선택입니다.

### 10.3 맞춤법 교정 및 도메인 특수 용어 보호

#### 10.3.1 도메인 용어 보호(Whitelist) 맞춤법 교정 처리 흐름

아무리 파인튜닝을 잘해도 조사나 띄어쓰기에서 미세한 오류가 발생할 수 있습니다. 한국어 교정 라이브러리를 연동하되, 도메인 특수 용어가 파괴되지 않도록 반드시 **3단계 보호 로직**을 적용해야 합니다.

**처리 흐름:**
1.  **[1단계] 도메인 용어 임시 마스킹:** 전사 텍스트 안에 화이트리스트에 등록된 도메인 용어가 있으면, 맞춤법 교정기가 인식할 수 없는 임시 플레이스홀더(`__PROTECTED_0__`, `__PROTECTED_1__` 등)로 치환하여 숨겨둡니다. 이 단계에서 보호 용어와 원래 용어의 매핑 정보를 따로 기록해 둡니다.

2.  **[2단계] 맞춤법 교정 실행:** 도메인 용어가 플레이스홀더로 대체된 텍스트를 맞춤법 교정 라이브러리에 통과시킵니다. 교정기는 플레이스홀더를 일반 텍스트로 인식하여 변경하지 않습니다.

    > ⚠️ **폐쇄망 환경 필수 확인:** 부산대 맞춤법 검사기(`py-hanspell`)는 교정 요청마다 외부 서버(`speller.cs.pusan.ac.kr`)로 HTTP 요청을 보내는 구조이므로 **에어갭 환경에서 사용 불가**합니다. 다음 대안을 사용해야 합니다.
    >
    > | 라이브러리 | 에어갭 가능 | 특징 |
    > | :--- | :---: | :--- |
    > | **PyKoSpacing** | ✅ | 띄어쓰기 전문 딥러닝 모델. 모델 파일(`.h5`)을 사전 반입하면 완전 오프라인 동작 |
    > | **Kiwi** 형태소 분석기 | ✅ | 띄어쓰기 교정 기능 내장, 추가 모델 반입 불필요 |
    > | **자체 Regex 규칙** | ✅ | 도메인 특화 교정에 최적, 외부 의존성 없음 |


3.  **[3단계] 도메인 용어 복원:** 교정이 완료된 텍스트 안의 플레이스홀더를 1단계에서 기록해 둔 원래 도메인 용어로 복원합니다. 결과적으로 조사·띄어쓰기는 교정되었으나 도메인 핵심 용어는 원형 그대로 유지된 텍스트가 완성됩니다.

**처리 예시:**

| 단계 | 텍스트 내용 |
| :--- | :--- |
| 원본 입력 | "현재 좌표는 알파 브라보 이고 오스카 마이크 중입니다" |
| 1단계 후 (마스킹) | "현재 좌표는 __PROTECTED_0__ 이고 __PROTECTED_1__ 중입니다" |
| 2단계 후 (교정) | "현재 좌표는 __PROTECTED_0__ 이고 __PROTECTED_1__ 중입니다." |
| 3단계 후 (복원) | "현재 좌표는 알파 브라보이고 오스카 마이크 중입니다." |

#### 10.3.2 화이트리스트/블랙리스트 관리 전략

*   **화이트리스트(보호 목록):** 교정기가 절대 건드리면 안 되는 도메인 핵심 용어 목록입니다. NATO 포네틱 코드("알파", "브라보" 등), 무기체계명, 전술 약어, 부대 고유 호출명 등을 포함합니다.
*   **블랙리스트(강제 치환 목록):** 모델이 자주 틀리는 특정 단어를 올바른 도메인 표현으로 강제 교정합니다. 예: "오스카마이크" → "오스카 마이크", "카피했습니다" → "카피(수신확인)". 블랙리스트는 맞춤법 교정 전에 먼저 적용하는 것이 원칙입니다.
*   **관리 원칙:** 이 두 목록은 별도의 설정 파일(YAML/JSON)로 관리하여, 코드 변경 없이 운영팀이 직접 용어를 추가·삭제할 수 있는 구조를 설계합니다. 신규 용어가 추가될 때마다 서버를 재배포하는 불편함을 없앨 수 있습니다.

제시해주신 **10.3 맞춤법 교정 및 도메인 특수 용어 보호** 섹션에, 음질 저하로 인한 오인식을 기술적으로 극복하기 위한 **'유사도 기반 강제 교정(Fuzzy Matching)'** 내용을 추가하여 보강해 드립니다.

이 기능은 단순히 텍스트를 고치는 수준을 넘어, 모델이 "날파"라고 적었어도 도메인 지식을 바탕으로 "알파"라고 확신하며 교정하는 **'최후의 안전장치'** 역할을 합니다.

---

### 10.3.3 유사도 기반 핵심 용어 강제 보정 (Fuzzy Logic Correction)

음질이 극도로 불량한 경우, 모델은 핵심 용어를 유사한 발음의 일반 단어로 오인식할 수 있습니다. 이를 방지하기 위해 단순 일치(Exact Match)를 넘어 **편집 거리(Edit Distance)** 알고리즘을 활용한 강제 보정 단계를 추가합니다.

**1. 작동 원리: 자모 분리 기반 편집 거리 측정**
*   **자모 분리:** 한국어는 "알파"와 "날파"를 통째로 비교하면 차이가 커 보이지만, 자모를 분리(`ㅇㅏㄹㅍㅏ` vs `ㄴㅏㄹㅍㅏ`)하여 비교하면 거리(Distance)가 매우 가깝습니다.
*   **임계값(Threshold) 설정:** 유사도 점수가 특정 기준(예: 80% 이상)을 넘으면, 모델의 전사 결과보다 화이트리스트의 단어가 정확하다고 판단하여 강제로 치환합니다.

**2. 유사도 보정 처리 흐름**
1.  **[후보군 추출]** 전사된 문장에서 화이트리스트 용어와 글자 수가 비슷하거나 발음 구조가 유사한 구간을 탐색합니다.
2.  **[유사도 계산]** 레벤슈타인 거리(Levenshtein Distance) 또는 자모 결합 유사도를 계산합니다.
3.  **[강제 치환]** 계산된 유사도가 설정된 임계값보다 높을 경우, 도메인 용어로 강제 업데이트합니다.

**3. 도메인 용어 강제 보정 예시**

| 전사 결과 (오인식) | 유사 용어 (화이트리스트) | 자모 유사도 | 최종 보정 결과 |
| :--- | :--- | :---: | :--- |
| "현재 위치 **날파** 브라보" | **알파** (Alpha) | 92% | "현재 위치 **알파** 브라보" |
| "**오스카 마이** 진행" | **오스카 마이크** (OM) | 85% | "**오스카 마이크** 진행" |
| "**카피** 했습니다" | **카피(수신확인)** | 100% | "**카피(수신확인)**" |

**4. 구현 시 고려사항 (에어갭 환경)**
*   **SymSpell/FuzzyWuzzy:** 외부 API 없이 Python 라이브러리만으로 동작 가능하며, 미리 구축된 사전(Dictionary)을 기반으로 매우 빠른 검색 속도를 보장합니다.
*   **과교정(Over-correction) 주의:** 유사도 임계값을 너무 낮게 설정하면 일반적인 대화 내용까지 핵심 용어로 바뀌는 '역효과'가 발생할 수 있습니다. 따라서 핵심용어에만 한정하여 적용하거나, 모델의 확신도가 낮을 때만 이 로직을 가동하는 이중 안전장치를 권장합니다.

### 10.4 전체 후처리 파이프라인 통합 흐름

각 후처리 모듈들은 순차적으로 연결된 하나의 통합 파이프라인으로 묶어 운영합니다. 오디오 파일 하나를 입력받아 최종 완성된 대화록을 출력하기까지의 전체 흐름은 다음과 같습니다.

**텍스트 후처리 단계 순서:**

1.  **[Blacklist]** 자주 틀리는 고정 패턴 강제 치환
2.  **[Fuzzy Matching]** 오인식된 핵심 용어를 유사도 기반으로 강제 보정 (→ 10.3.3절)
3.  **[Masking]** 보정된 핵심 용어를 `__PROTECTED__`로 보호
4.  **[Speller]** 오프라인 맞춤법 교정기 적용 (PyKoSpacing / Kiwi)
5.  **[Unmasking]** 보호된 용어 복원 및 최종 출력

**통합 파이프라인 처리 순서:**

```
[입력] 오디오 파일 경로
        │
        ▼
[Step 1] 화자 분할 여부 판단
  - 화자 분할 활성화 시: Pyannote + Faster-Whisper 병렬 실행 → 화자 태깅 세그먼트 목록 생성
  - 단독 전사 모드 시: Faster-Whisper 단독 실행 → 화자 = "Unknown"으로 세그먼트 생성
        │
        ▼
[Step 2] 세그먼트별 후처리 적용 (반복)
  ┌──────────────────────────────────────────────────────┐
  │ 각 세그먼트 텍스트에 대해:                           │
  │   2-1. 앞뒤 공백 제거 (Strip)                        │
  │   2-2. Blacklist 강제 치환 (고정 오인식 패턴 교정)   │
  │   2-3. Fuzzy Matching 보정 (유사도 기반 핵심 용어)   │
  │   2-4. ITN 변환 적용 (숫자·단위·코드)                │
  │   2-5. 맞춤법 교정 (용어 보호 3단계 포함)            │
  └──────────────────────────────────────────────────────┘
        │
        ▼
[Step 3] 세그먼트 목록 조립
  - 화자 ID, 시작 시각, 종료 시각, 교정된 텍스트를 하나의 레코드로 묶음
        │
        ▼
[출력] 완성된 대화록 (세그먼트 목록)
  예) [Speaker_A] (0.2s) 안녕하세요, 오늘 회의 주제는 신제품 출시 일정입니다.
      [Speaker_B] (4.6s) 네, 말씀해 주세요.
```

**설계 시 주의사항:**
*   모든 후처리 단계는 세그먼트 단위로 독립적으로 적용됩니다. 한 세그먼트에서 오류가 발생해도 나머지 세그먼트에 영향을 주지 않도록 예외 처리(try/except)를 반드시 각 단계에 적용해야 합니다.
*   Fuzzy Matching은 모델의 확신도(Log-probability)가 낮은 구간에만 선택적으로 적용하는 이중 안전장치를 권장합니다. 전체 텍스트에 무조건 적용하면 과교정(Over-correction) 위험이 있습니다. (→ 10.3.3절 참조)
*   화자 분할 모델은 연산이 무거우므로, 단일 화자 교신이 명확한 경우 단독 전사 모드로 전환하여 처리 속도를 높이는 선택지를 API에서 파라미터로 제공하는 것이 권장됩니다.


# 제11장. 내부망 추론 API 서버 구축 및 배포

모든 파인튜닝과 후처리 파이프라인이 완성되었다면, 다른 시스템(클라이언트, 웹 포털, 시스템 등)이 오디오를 전송하고 전사 결과를 받아갈 수 있는 **독립적인 웹 API 서버**를 구축해야 합니다. 내부망(에어갭) 환경에서는 외부 클라우드에 의존 없이 모든 것이 내부에서 자급자족되어야 합니다.

---

### 11.1 FastAPI 기반 비동기(Async) 배치 처리 아키텍처

#### 11.1.1 동기식 서버의 문제점과 비동기 해결책

AI 모델 추론은 연산량이 매우 크기 때문에 **동기식(Synchronous)** 서버로 구성하면 심각한 병목 현상이 발생합니다. 첫 번째 요청의 오디오를 GPU로 처리하는 수 분 동안 서버 전체가 멈추어(Blocking), 뒤이어 들어오는 모든 요청이 응답을 받지 못하고 대기합니다. 동시 사용자가 여러 명이거나 연속적인 교신 기록 파일을 일괄 처리해야 하는 실무 환경에서는 이 구조가 치명적입니다.

이를 해결하기 위해 **비동기 + 작업 큐(Queue)** 구조를 채택합니다. 클라이언트는 파일을 서버에 전송하는 즉시 "접수 완료" 응답(Task ID)을 받고 연결을 끊습니다. 실제 무거운 GPU 연산은 백그라운드 워커(Worker)가 순서대로 처리합니다. 클라이언트는 이후 Task ID를 이용해 작업 완료 여부를 능동적으로 조회합니다.

#### 11.1.2 전체 서버 아키텍처 구성도

```
클라이언트 (오디오 파일 POST)
        │
        ▼
┌───────────────────────────┐
│  FastAPI 웹서버           │  :8000 포트
│  (비동기 HTTP 엔드포인트) │
└──────────┬────────────────┘
           │ 작업 큐 적재
           ▼
┌───────────────────────────┐
│  Redis 메시지 브로커      │  작업 대기열 관리
└──────────┬────────────────┘
           │ 작업 소비
           ▼
┌───────────────────────────┐
│  Celery 워커              │  GPU 추론 실행
│  (ASR + 후처리 파이프라인)│
└──────────┬────────────────┘
           │ 결과 저장
           ▼
┌───────────────────────────┐
│  SQLite / Redis           │  작업 상태 및 결과 저장
└───────────────────────────┘
```

#### 11.1.3 FastAPI 서버 처리 흐름 및 주요 엔드포인트 설계

**[전사 요청: POST /transcribe]**

클라이언트가 오디오 파일을 이 엔드포인트로 전송하면 서버는 다음 순서로 처리합니다.

1.  **파일 수신 및 형식 검증:** 업로드된 파일의 확장자를 확인하여 지원하는 형식(WAV, MP3, OGG, M4A, FLAC)인지 검사합니다. 지원하지 않는 형식이면 오류(HTTP 400)를 즉시 반환합니다.
2.  **임시 저장 및 Task ID 발급:** 파일을 서버 내부의 임시 디렉터리에 저장하고, 이 작업을 식별하는 고유 Task ID(UUID)를 발급합니다. Task ID와 초기 상태("queued")를 작업 저장소에 등록합니다.
3.  **즉시 응답 반환 (Non-blocking):** 클라이언트에게 Task ID와 "queued" 상태를 담은 JSON 응답을 즉시 돌려보냅니다. 이 시점에서 클라이언트와 서버의 HTTP 연결은 종료됩니다. GPU 연산은 아직 시작되지 않았습니다.
4.  **백그라운드 추론 실행:** FastAPI의 `BackgroundTasks` 또는 Celery 워커가 대기열에서 작업을 꺼내어, 10장에서 설계한 통합 후처리 파이프라인(전사 → 화자분할 → ITN → 맞춤법 교정)을 순서대로 실행합니다.
5.  **결과 저장:** 추론이 완료되면 Task ID와 연결하여 전사 결과를 Redis 또는 DB에 저장하고 상태를 "done"으로 업데이트합니다. 오류 발생 시 상태를 "failed"로 기록합니다.

**[결과 조회: GET /result/{task_id}]**

클라이언트는 이전에 받은 Task ID로 이 엔드포인트를 폴링(Polling)하여 작업 진행 상태와 완료된 결과를 확인합니다. 응답에는 현재 상태(`queued` / `processing` / `done` / `failed`)와 전사 결과 텍스트(완료 시), 오류 메시지(실패 시)가 포함됩니다.

**[서버 상태 확인: GET /health]**

서버의 정상 작동 여부를 모니터링하기 위한 헬스체크 엔드포인트입니다. GPU 사용 가능 여부, GPU 모델명, 전체 VRAM 용량, 현재 사용 중인 VRAM 용량을 반환합니다. Docker의 `healthcheck` 설정과 연동하여 서버가 비정상 상태일 때 자동으로 재시작되도록 구성합니다.

**[주요 설계 원칙]**
*   **단일 워커(Worker):** GPU는 동시에 여러 작업을 처리할 때 VRAM 부족(OOM)이 발생할 수 있습니다. GPU 연산 워커는 반드시 단일 프로세스로 실행하여, 한 번에 하나의 추론만 독점적으로 수행하도록 설계해야 합니다.
*   **요청 속도 제한(Rate Limiting):** 대기열이 무한히 쌓이지 않도록 동시 처리 가능 수량의 상한선을 설정하고, 초과 시 적절한 안내 메시지를 반환합니다.
*   **임시 파일 정리:** 추론이 완료된 후 서버 내부에 임시 저장된 오디오 파일은 즉시 삭제하여 디스크 공간을 확보합니다.

#### 11.1.4 서버 실행 방법

FastAPI 서버는 `uvicorn` ASGI 서버를 통해 실행합니다. 개발 환경에서는 코드 변경 시 자동 재시작되는 `--reload` 옵션을 사용하고, 운영 환경에서는 제거합니다. GPU 추론은 단일 프로세스에서만 이루어져야 하므로 `--workers` 값은 반드시 1로 고정하고, 오디오 파일 업로드 대기 시간을 고려하여 `--timeout-keep-alive` 값을 충분히 크게 설정해야 합니다.

### 11.2 NVIDIA Triton Inference Server 구성 (대규모 배포 옵션)

하루 수천 건 이상의 대규모 처리가 필요하거나 엔터프라이즈급 성능이 필요할 때 고려하는 아키텍처입니다.

#### 11.2.1 Triton의 주요 장점

*   **동적 배칭(Dynamic Batching):** 여러 요청을 Triton이 자동으로 하나의 배치로 묶어 GPU에 한꺼번에 던집니다. GPU 가동률이 극대화되어 단순 FastAPI 서버 대비 처리량(Throughput)이 수배 향상됩니다.
*   **멀티 모델 동시 서빙:** ASR 모델, 화자 분할 모델(Pyannote), 후처리 로직을 하나의 Triton 서버에서 동시에 올려둘 수 있습니다. 각 모델은 독립적으로 버전 관리되며, 서비스 중단 없이 특정 모델만 업데이트하는 롤링 배포가 가능합니다.
*   **앙상블 파이프라인:** 전처리 → 인코더 → 디코더 → 후처리로 이어지는 모델 체인을 Triton 서버 내부에서 직접 연결합니다. 각 단계 결과물이 서버 메모리 안에서 직접 다음 단계로 전달되므로 네트워크 왕복에 의한 지연이 발생하지 않습니다.

#### 11.2.2 Triton 모델 레포지토리 구조

Triton은 특정 디렉터리 구조(Model Repository)를 약속으로 사용합니다. 각 모델 디렉터리 안에 `config.pbtxt`라는 설정 파일과 버전 번호 하위 폴더(`1/`) 안에 실제 가중치 파일을 배치합니다. ASR 인코더, 디코더, 전체를 묶는 앙상블 파이프라인이 각각 별도 디렉터리로 관리됩니다. 이 구조를 올바르게 구성해야 Triton이 모델을 인식하고 배포할 수 있습니다.

#### 11.2.3 FastAPI vs Triton 선택 기준

| 항목 | FastAPI 직접 서버 | NVIDIA Triton |
| :--- | :--- | :--- |
| **구축 난이도** | 낮음 (Python 숙련자 가능) | 높음 (DevOps 전문성 필요) |
| **일 처리량** | ~수백 건/일 | ~수만 건/일 |
| **GPU 활용률** | 수동 관리 필요 | 자동 최적화 |
| **모델 업데이트** | 서버 재시작 필요 | 무중단 롤링 업데이트 |
| **권장 상황** | 소규모 내부 시스템 | 대규모 실시간 처리 |

### 11.3 Docker Compose 기반 컨테이너 패키징

폐쇄망 환경 인수인계의 핵심은 **단 한 줄의 명령어로 서비스가 기동되는 컨테이너 패키지**를 완성하는 것입니다.

#### 11.3.1 디렉터리 구조

인수인계용 최종 패키지 디렉터리는 다음과 같이 구성됩니다.

```
asr_service/
├── docker-compose.yml         # 전체 서비스 구성 파일
├── .env                       # 환경 변수 설정 (포트, GPU 번호 등)
├── api_server/
│   ├── Dockerfile             # 컨테이너 이미지 빌드 설정
│   ├── main.py                # FastAPI 서버 소스 코드
│   └── requirements.txt       # Python 패키지 목록
├── models/
│   └── ct2_ko_military_bf16/  # CT2 변환 모델 가중치 (★ 핵심 자산)
└── logs/                      # 전사 로그 저장 경로
```

#### 11.3.2 `docker-compose.yml` 설계 원칙

`docker-compose.yml` 파일은 ASR API 서버 컨테이너와 Redis 브로커 컨테이너, 두 가지 서비스를 하나로 묶어 관리합니다. 이 파일에 다음 항목들을 명확히 정의해야 합니다.

*   **GPU 할당:** `deploy.resources.reservations.devices` 항목에 NVIDIA GPU 드라이버와 사용할 GPU 수를 명시합니다. 이 설정이 없으면 컨테이너가 GPU를 인식하지 못합니다.
*   **볼륨 마운트:** 모델 가중치 폴더(`models/`)를 컨테이너 내부 경로에 읽기 전용으로 마운트합니다. 컨테이너가 재시작되어도 모델 파일은 호스트에 유지됩니다. 로그 폴더 역시 마운트하여 컨테이너 외부에서 로그를 열람할 수 있도록 합니다.
*   **포트 설정:** `.env` 파일에 API 서비스 포트를 변수로 지정하여, 코드 수정 없이 포트 번호를 변경할 수 있도록 유연하게 설계합니다.
*   **헬스체크(Healthcheck):** `/health` 엔드포인트를 주기적으로 호출하여 서버가 정상 응답하는지 자동으로 감시합니다. 일정 횟수 이상 실패하면 Docker가 컨테이너를 자동으로 재시작합니다.
*   **재시작 정책:** `restart: unless-stopped` 설정으로 서버가 OOM 등 예상치 못한 원인으로 종료되더라도 자동으로 재기동됩니다.

#### 11.3.3 Dockerfile 설계 원칙

Dockerfile은 운영 서버에 배포될 컨테이너 이미지를 정의합니다. 핵심 설계 원칙은 다음과 같습니다.

*   **베이스 이미지:** PyTorch 공식 이미지(`pytorch/pytorch:2.3.0-cuda12.1-cudnn8-runtime`)를 베이스로 사용합니다. 이 이미지에는 **PyTorch 2.3.0 + CUDA 12.1 + cuDNN 8.9**가 사전 설치되어 있어, 별도의 PyTorch 설치 없이 바로 GPU 추론 환경이 구성됩니다. bare CUDA 이미지(`nvidia/cuda:...`) 대비 PyTorch 호환성 문제를 원천 차단할 수 있으며, `requirements.txt`에서 `torch` 항목을 제거하여 `.whl` 반입 목록도 간소화됩니다.

    > **에어갭 환경 이미지 반입 방법 (인터넷망 → 내부망):**
    > ```bash
    > # [인터넷망] 이미지 Pull 및 tar 저장
    > docker pull pytorch/pytorch:2.3.0-cuda12.1-cudnn8-runtime
    > docker save pytorch/pytorch:2.3.0-cuda12.1-cudnn8-runtime -o pytorch_2.3_cu121.tar
    >
    > # [내부망] tar 파일 반입 후 로드
    > docker load -i pytorch_2.3_cu121.tar
    > ```
*   **FFmpeg 포함:** 다양한 오디오 포맷(MP3, OGG 등)을 처리하기 위해 `ffmpeg`를 시스템 패키지로 함께 설치해야 합니다.
*   **오프라인 패키지 설치 (에어갭 환경 핵심):** 폐쇄망에서는 외부 인터넷(`pip install`)이 차단됩니다. 인터넷이 되는 개발 PC에서 `pip download` 명령어로 필요한 모든 패키지를 `.whl` 파일로 미리 내려받은 뒤, 이 파일들을 Docker 이미지 빌드 시 함께 복사하여 로컬에서 설치(`pip install --no-index --find-links`)하는 방식을 사용합니다.
*   **실행 명령:** 컨테이너 기동 시 `uvicorn`이 자동으로 FastAPI 서버를 시작하도록 `CMD` 명령어를 설정합니다. 워커 수는 반드시 1로 고정합니다.

#### 11.3.4 폐쇄망 배포 순서 및 최종 인수인계 산출물

**배포 절차 (운영 서버에서 순서대로 실행):**

1.  **도커 이미지 적재:** 개발 환경에서 `docker save`로 내보낸 이미지 tar 파일을 운영 서버로 복사한 뒤 `docker load` 명령어로 로컬 도커에 등록합니다.
2.  **디렉터리 배치:** `docker-compose.yml`, `.env`, `models/` 폴더를 서버의 지정 경로에 배치합니다.
3.  **서비스 기동:** 해당 경로에서 `docker-compose up -d` 명령어 한 줄로 전체 서비스(API 서버 + Redis)가 백그라운드에서 동시에 기동됩니다.
4.  **기동 확인:** `docker-compose ps`로 컨테이너 상태를 확인하고, `curl http://localhost:8000/health`로 API 서버가 정상 응답하는지 검증합니다.

**복구 시나리오:**
*   **OOM으로 API 서버 다운 시:** `docker-compose restart asr_api` → 5초 내 자동 복구.
*   **모델 가중치 업데이트 시:** `models/` 폴더의 파일을 교체한 뒤 컨테이너만 재시작. 소스 코드 변경 불필요.
*   **서버 전체 이전 시:** 위 패키지 폴더 전체를 신규 서버에 복사하고 `docker-compose up -d`만 실행하면 완료.
